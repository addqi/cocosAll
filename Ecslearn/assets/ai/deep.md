# 仔细检查（Deep Review）

> **触发词**：当用户说"仔细检查"，即触发本规则，对指定代码进行全盘深度审计。

---

## 目标

逻辑零缺陷，代码质量 S+。不是"看一眼觉得没问题"，是逐行证明它正确。

---

## ⚠️ 强制前置步骤（禁止跳过）

### 阶段零：确定审查范围

1. **明确目标**：用户指定了文件/函数/模块就审查那个，没指定就问
2. **完整读取**：必须读取所有相关代码，包括被调用的依赖、类型定义、上下游调用链
3. **禁止凭印象**：每一个判断都必须基于实际读到的代码

---

## 审查流程

### 第一遍：逻辑正确性（最高优先级）

逐函数、逐分支验证：

```
对每个函数：
1. 前置条件 - 调用时参数的所有可能取值是什么？
2. 后置条件 - 返回值在所有路径上是否符合预期？
3. 不变量   - 循环/递归的不变量是否成立？终止条件是否正确？
4. 边界     - null/undefined/空数组/零值/负数/溢出，全部过一遍
5. 异步     - await 是否遗漏？竞态条件？错误传播链是否完整？
6. 状态突变 - 谁在改状态？改的时机对不对？有没有意外的副作用？
```

**输出格式**：

```
【逻辑审计】
✅ functionName - 正确，原因：[简述]
❌ functionName - 缺陷：[精确描述问题 + 触发条件 + 修复方案]
⚠️ functionName - 风险：[不确定的隐患 + 建议验证方式]
```

### 第二遍：数据流完整性

```
1. 数据从哪来？类型是否保证？
2. 数据怎么变？每次变换是否有损？
3. 数据到哪去？消费方是否拿到正确的东西？
4. 错误数据进来会怎样？有没有防御？防御是否充分？
```

### 第三遍：品味与质量

用 Linus 标准审视：

```
1. 特殊情况 - 有没有能通过重构数据结构消除的 if/else？
2. 缩进深度 - 超过3层的必须指出
3. 函数职责 - 一个函数是否只做一件事？
4. 命名     - 名字是否准确反映意图？
5. 重复     - 有没有 copy-paste 代码？
6. 死代码   - 有没有永远走不到的分支？
```

### 第四遍：向后兼容与破坏性

```
1. 公开接口是否有变化？
2. 现有调用方会不会崩？
3. 默认值/行为是否被悄悄改了？
```

## 最终输出

```
┌──────────────────────────────────────┐
│          仔细检查报告                 │
├──────────────────────────────────────┤
│ 【逻辑正确性】 ✅ 零缺陷 / ❌ 发现 N 个问题  │
│  - [逐条列出，按严重度排序]              │
│                                      │
│ 【数据流】 ✅ 完整 / ⚠️ 有风险点         │
│  - [列出风险]                          │
│                                      │
│ 【品味评分】 S+ / S / A / B / C / D    │
│  - [关键改进点]                        │
│                                      │
│ 【破坏性】 ✅ 零破坏 / ❌ 有破坏风险      │
│  - [列出风险]                          │
│                                      │
│ 【总结】                               │
│  可以上线 / 需要修复后再上线 / 建议重写    │
└──────────────────────────────────────┘
```

---

## 铁律

- **不放过任何一行**：审查不是抽样，是全量
- **每个判断给证据**：说有 bug 就指出哪一行、什么输入会触发
- **修复方案必须具体**：不是"建议优化"，是"第 47 行改成 xxx"
- **宁可误报不可漏报**：有疑问就标 ⚠️，让用户决定
