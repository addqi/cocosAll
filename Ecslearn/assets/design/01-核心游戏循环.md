# 01 - 核心游戏循环

> 前置阅读：[00-项目总览](./00-项目总览.md)

## 本章目标

实现一个**能跑起来的游戏骨架**：启动 → 主菜单 → 进入战斗 → 战斗结束 → 结算 → 回到主菜单。没有花哨内容，但循环完整。

---

## 1. 游戏状态机

整个游戏只有这几个状态，用一个简单枚举就够了：

```
┌──────────┐
│  Boot    │ ──→ 加载资源/配置
└────┬─────┘
     ▼
┌──────────┐
│  Home    │ ──→ 主菜单界面
└────┬─────┘
     ▼ (点击开始)
┌──────────┐
│ Loading  │ ──→ 加载关卡资源
└────┬─────┘
     ▼
┌──────────┐     ┌──────────┐
│ Playing  │ ──→ │  Reward  │ ──→ 三选一技能
└────┬─────┘     └────┬─────┘
     │                │
     │ (清完敌人)       │ (选完技能)
     │                ▼
     │           ┌──────────┐
     │           │Transition│ ──→ 进入下一层
     │           └────┬─────┘
     │                │
     │                ▼ (回到 Playing)
     │
     ▼ (HP ≤ 0)
┌──────────┐
│ GameOver │ ──→ 结算/复活选择
└────┬─────┘
     ▼ (回到 Home)
```

### 代码实现

```typescript
enum GamePhase {
    Boot,
    Home,
    Loading,
    Playing,
    Reward,
    Transition,
    GameOver,
}

class GameStateMachine {
    private _phase: GamePhase = GamePhase.Boot;

    get phase(): GamePhase { return this._phase; }

    transition(next: GamePhase): void {
        const prev = this._phase;
        if (!this._isValidTransition(prev, next)) {
            console.warn(`Invalid transition: ${GamePhase[prev]} → ${GamePhase[next]}`);
            return;
        }
        this._phase = next;
        EventBus.emit('game:phaseChanged', { prev, next });
    }

    private _isValidTransition(from: GamePhase, to: GamePhase): boolean {
        const valid: Record<GamePhase, GamePhase[]> = {
            [GamePhase.Boot]:       [GamePhase.Home],
            [GamePhase.Home]:       [GamePhase.Loading],
            [GamePhase.Loading]:    [GamePhase.Playing],
            [GamePhase.Playing]:    [GamePhase.Reward, GamePhase.GameOver],
            [GamePhase.Reward]:     [GamePhase.Transition],
            [GamePhase.Transition]: [GamePhase.Playing],
            [GamePhase.GameOver]:   [GamePhase.Home],
        };
        return valid[from]?.includes(to) ?? false;
    }
}
```

**注意**：没有 `Paused` 状态。暂停是 UI 层面的事（`cc.game.pause()`），不属于游戏逻辑状态。不要把 UI 关注点混进游戏状态机。

## 2. 场景结构

只用 **一个场景**，通过节点层级切换内容。原因：Cocos 场景切换有开销且状态管理复杂，单场景 + 分层更可控。

```
Scene (根节点)
├── GameLayer          ← 游戏世界（地图、角色、敌人、弹道）
│   ├── MapRoot        ← 地图瓦片
│   ├── EntityRoot     ← 所有游戏实体的父节点
│   └── ProjectileRoot ← 弹道单独一层，方便管理
├── UILayer            ← HUD、血条、金币
│   ├── HUD            ← 战斗中常驻信息
│   └── PopupRoot      ← 弹窗容器
├── OverlayLayer       ← 全屏遮罩、转场特效
└── Camera             ← 跟随玩家的摄像机
```

### 参考：幽灵射手的做法

幽灵射手用 `GameManager` 的静态属性 `ndPlayer`、`arrMonster` 直接持有节点引用。有效，但耦合紧密。我们用层级节点分组 + ECS 管理实体引用，更清晰。

## 3. Game Loop 核心

每帧做的事情非常简单：

```typescript
// GameLoop.ts - 挂载在场景根节点上
@ccclass('GameLoop')
export class GameLoop extends Component {
    private _stateMachine: GameStateMachine;
    private _systems: GameSystem[] = [];

    onLoad(): void {
        this._stateMachine = new GameStateMachine();
        this._initSystems();
        this._stateMachine.transition(GamePhase.Home);
    }

    update(dt: number): void {
        if (this._stateMachine.phase !== GamePhase.Playing) return;

        for (const system of this._systems) {
            system.update(dt);
        }
    }

    private _initSystems(): void {
        // 按执行顺序注册，顺序很重要
        this._systems = [
            new InputSystem(),
            new MovementSystem(),
            new CombatSystem(),
            new AISystem(),
            new ProjectileSystem(),
            new BuffSystem(),
            new CleanupSystem(),
        ];
    }
}
```

**系统执行顺序**的设计原则：
1. **Input** 先读取输入，写入 InputComponent
2. **Movement** 根据输入/AI 更新位置
3. **Combat** 处理攻击判定与伤害
4. **AI** 更新敌人决策（为下一帧准备）
5. **Projectile** 更新弹道飞行与碰撞
6. **Buff** 处理 Buff tick（持续伤害、回血等）
7. **Cleanup** 回收死亡实体、清理过期弹道

## 4. 关卡生命周期

一层关卡从生到死的完整流程：

```
1. enterFloor(floorNum)
   ├── 加载/生成地图
   ├── 放置敌人（从配置表读取）
   ├── 初始化玩家位置
   └── stateMachine → Playing

2. Playing 中
   ├── 每帧跑 System.update()
   ├── 敌人死亡 → enemiesAlive--
   └── 玩家死亡 → stateMachine → GameOver

3. enemiesAlive === 0
   ├── 生成传送门
   └── 玩家触碰传送门 → stateMachine → Reward

4. Reward 阶段
   ├── 展示三选一技能
   └── 玩家选择后 → stateMachine → Transition

5. Transition
   ├── floorNum++
   ├── 清理当前层（回收实体、地图）
   └── 回到步骤 1
```

### 层管理器


## 5. 对象池

> 参考：幽灵射手用 `PoolManager` 做节点池，我们做同样的事但更简洁。

敌人和弹道是频繁创建/销毁的对象，必须池化：

```typescript
class SimplePool<T> {
    private _pool: T[] = [];
    private _factory: () => T;
    private _reset: (item: T) => void;

    constructor(factory: () => T, reset: (item: T) => void) {
        this._factory = factory;
        this._reset = reset;
    }

    get(): T {
        const item = this._pool.pop() ?? this._factory();
        this._reset(item);
        return item;
    }

    put(item: T): void {
        this._pool.push(item);
    }
}
```

用法：
```typescript
const arrowPool = new SimplePool(
    () => instantiate(arrowPrefab),
    (node) => { node.active = true; node.setPosition(Vec3.ZERO); }
);

// 射箭
const arrow = arrowPool.get();
entityRoot.addChild(arrow);

// 箭命中后回收
arrow.active = false;
arrow.removeFromParent();
arrowPool.put(arrow);
```

## 6. 新手实现步骤

按这个顺序做，每一步都有可见结果：

### Step 1：空循环能跑（预计 30 分钟）
- [ ] 创建 `GameStateMachine` 类
- [ ] 创建 `GameLoop` 组件，挂载到场景根节点
- [ ] 状态切换时在控制台打印日志
- **验收**：运行后控制台输出 `Boot → Home`

### Step 2：场景分层（预计 30 分钟）
- [ ] 在场景中手动创建 GameLayer / UILayer / OverlayLayer 节点
- [ ] GameLayer 下创建 MapRoot / EntityRoot / ProjectileRoot
- [ ] UILayer 下创建 HUD / PopupRoot
- **验收**：场景层级结构清晰可见

### Step 3：系统骨架（预计 1 小时）
- [ ] 定义 `GameSystem` 接口（`init()`, `update(dt)`, `cleanup()`）
- [ ] 创建空的 InputSystem / MovementSystem
- [ ] GameLoop 中注册并每帧调用
- **验收**：Playing 状态下 System.update 被调用（console.log 验证）

### Step 4：状态切换串联（预计 1 小时）
- [ ] Home 界面放一个「开始」按钮 → transition 到 Loading → Playing
- [ ] Playing 中模拟一个 3 秒计时器 → transition 到 GameOver
- [ ] GameOver 放一个「返回」按钮 → 回到 Home
- **验收**：完整循环可走通：Home → Playing → GameOver → Home

## 关键设计决策

| 决策 | 理由 |
|------|------|
| 单场景 | 避免场景切换的状态丢失问题，新手更容易理解 |
| 显式状态机 | 比 boolean 标志位（isGameStart, isPaused, isOver）清晰一百倍 |
| System 执行顺序固定 | 避免帧内竞争条件，调试时可预测 |
| 对象池从第一天就用 | 后面补池化比一开始就设计池化痛苦十倍 |
