# 重构步骤 1 - 职业与基础架构

> 目标：先把玩家系统从“单职业弓箭手控制器”重构为“通用玩家外壳 + 职业行为实现”。
>
> 当前约束：
> - 运行时只读 JSON
> - Excel 不接运行时
> - 本阶段优先保证旧战斗流程不坏，再逐步切换到多职业结构

---

## 一、这一步到底在解决什么问题

当前项目最脏的地方不是 Buff，也不是技能，而是 `PlayerControl`。

它同时负责：

- 生命周期
- 输入
- 状态机
- 射击策略
- 弓箭手配置
- 技能上下文
- 升级系统接线

这就是典型 God object。继续往里面塞战士、召唤师，只会把代码拖进屎坑。

所以第一阶段只做一件事：

**把玩家公共外壳和职业行为彻底分离。**

---

## 二、阶段目标

完成本大步骤后，项目应该满足：

1. `PlayerControl` 只保留玩家公共外壳职责
2. 弓箭手逻辑进入 `ArcherBehavior`
3. 为战士 / 召唤师预留行为骨架
4. 对外接口尽量保持兼容
5. 外部模块不再直接依赖“弓箭手实现细节”
6. `GameLoop` 不再同时承担启动、预加载、对象池和副作用注册
7. 敌人和敌方投射物后续可逐步摆脱对 `PlayerControl` 的直接依赖

---

## 三、总实施顺序

本大步骤拆成 8 个小步骤：

1. 先建立行为基类和工厂
2. 再建立游戏层行为抽象
3. 再把弓箭手逻辑搬进去
4. 再改 `PlayerControl`
5. 再改状态机和配置入口
6. 最后补战士 / 召唤师骨架和回归验证
7. 再拆 `PlayerControl` 周边运行时胶水
8. 再瘦身 `GameLoop` 和敌我交互接口

原则很简单：

**每一步完成后，项目都必须还能跑。**

## 四、Step 1.1 - 建立玩家行为基础设施

### 目标

先把“职业行为”这个概念正式建出来。

### 需要新增的文件

- `baseSystem/player/PlayerBehaviorBase.ts`
- `baseSystem/player/PlayerBehaviorFactory.ts`
- `baseSystem/player/playerBehavior.ts`
- `baseSystem/player/index.ts`

### 改动内容

建立与敌人行为系统对齐的玩家行为基建：

- 抽象基类
- 工厂
- 装饰器注册
- 模块导出

### 设计要求

- `PlayerBehaviorFactory` 只按 `behaviorId` 创建实例
- 行为类通过装饰器注册
- 不允许在 `PlayerControl` 里直接 `new ArcherBehavior()`

### 完成定义

达到以下条件才算完成：

- 可以通过 `PlayerBehaviorFactory.create(id)` 创建行为实例
- 新增职业行为不需要修改工厂主体

### 验证手段

- 手动打印注册表内容，确认 `archer` 能被正确创建
- 临时用假数据调用工厂，确认返回实例类型正确

### 测试用例

#### 用例 1：合法职业 ID 创建成功

- 输入：`archer`
- 预期：返回 `ArcherBehavior` 实例

#### 用例 2：非法职业 ID 创建失败

- 输入：未知职业 ID
- 预期：返回 `null` 或抛出明确错误日志

#### 用例 3：重复注册保护

- 场景：两个行为类注册同一个 ID
- 预期：启动时给出明确报错，不允许静默覆盖

## 五、Step 1.2 - 建立游戏层行为抽象

### 目标

把基础设施提升到游戏层，让职业行为真的能驱动战斗。

### 需要新增的文件

- `player/base/PlayerBehavior.ts`
- `player/base/types.ts`
- `player/base/index.ts`
- `player/behaviors.ts`

### 改动内容

定义游戏层行为能力：

- 如何读取职业配置
- 如何决定“是否攻击”
- 如何创建攻击状态
- 如何构建技能 / 升级 / 攻击所需上下文

### 设计要求

行为抽象至少要覆盖：

- `config`
- `wantAttack(...)`
- `createAttackState(...)`
- `buildSkillContext(...)`
- 预留职业私有命令入口

### 完成定义

- `PlayerBehavior` 成为唯一职业扩展入口
- 后续职业逻辑不再直接挂在 `PlayerControl` 上

### 验证手段

- 用空壳行为实例驱动玩家节点，确保类型结构完整
- 编译期检查 `PlayerControl` 可以只依赖抽象类型

### 测试用例

#### 用例 1：行为接口最小闭包完整

- 检查：`PlayerControl` 所需方法在行为抽象中都能找到
- 预期：不需要回退到具体职业类

#### 用例 2：技能上下文可从行为层提供

- 场景：技能系统不直接读取弓箭手私有字段
- 预期：上下文构建入口在行为层统一暴露

## 六、Step 1.3 - 搬迁弓箭手逻辑到 `ArcherBehavior`

### 目标

把“当前唯一能打的职业”先提炼成第一个标准行为实现。

### 重点涉及文件

- `player/PlayerControl.ts`
- `player/states/PlayerShootState.ts`
- `shoot/ShootResolver.ts`
- `player/archer/ArcherBehavior.ts`
- `player/archer/ArcherAttackState.ts`
- `player/archer/archerConfig.ts`
- `player/archer/index.ts`

### 改动内容

迁移这些弓箭手私有概念：

- `_shootPolicy`
- 射击状态中的发箭逻辑
- 弓箭配置读取
- 弓箭手相关攻击决议

### 设计要求

- 弓箭手逻辑只存在于 `player/archer/`
- `ShootResolver` 不再直接依赖通用 `playerConfig` 里的弓箭字段
- 攻击状态变成行为实现的一部分，而不是公共状态类里写死

### 完成定义

- 当前弓箭手功能和改造前一致
- 但弓箭手实现不再散落在通用玩家模块里

### 验证手段

- 进入游戏，确认原有停下射箭逻辑不变
- 比较重构前后：箭数量、攻速、命中效果都一致

### 测试用例

#### 用例 1：静止射箭回归

- 操作：玩家停下，有目标
- 预期：正常进入攻击状态并发射箭矢

#### 用例 2：移动打断攻击

- 操作：玩家移动中
- 预期：攻击逻辑符合当前策略定义

#### 用例 3：升级后的射击行为不丢失

- 操作：给玩家加额外弹道、穿透、弹射
- 预期：弓箭手行为继续正确使用这些属性

## 七、Step 1.4 - 改造 `PlayerControl` 为通用外壳

### 目标

让 `PlayerControl` 不再知道自己是弓箭手。

### 重点涉及文件

- `player/PlayerControl.ts`

### 改动内容

`PlayerControl` 只保留：

- 节点生命周期
- 公共数据持有
- 状态机调度
- 外部兼容接口
- behavior 实例挂载

### 必须保持兼容的接口

- `PlayerControl.instance`
- `applyDamage`
- `grantXp`
- `revive`
- 其他已经被外部大量引用的公共能力

### 不该再出现在 `PlayerControl` 里的东西

- 弓箭手配置读取
- `IShootPolicy` 私有实现细节
- 具体攻击轨迹生成逻辑
- 具体职业技能拼装逻辑

### 完成定义

- `PlayerControl` 只依赖 `PlayerBehavior`
- 不直接 import `player/archer/*`

### 验证手段

- 全局搜索 `PlayerControl`，确认不存在职业硬编码分支
- 编译运行，外部对 `PlayerControl.instance` 的调用不报错

### 测试用例

#### 用例 1：外部引用兼容

- 场景：`GameSession`、敌人行为、技能系统仍调用 `PlayerControl.instance`
- 预期：功能不坏

#### 用例 2：behaviorId 可切换

- 场景：在编辑器上切换 `behaviorId`
- 预期：能创建不同职业行为实例，至少弓箭手实例正常

## 八、Step 1.5 - 改状态机与配置入口

把“攻击”抽象为公共概念，而不是“射击”。

### 重点涉及文件

- `player/states/PlayerContext.ts`

- `player/states/PlayerAttackState.ts`

- `player/config/player.json`
- `player/archer/archer.json`

### 改动内容

- `Shoot` 状态改成 `Attack`
- `PlayerContext` 增加 `behavior`
- 共有配置和职业专属配置拆分
- 共有属性和职业专属属性拆分

### 设计要求

- 共有层只保留所有职业都会用到的数据
- 弓箭手字段迁到 `archer.json`
- 以后战士 / 召唤师也各自有自己的 JSON

### 完成定义

- 通用状态机不再带弓箭命名
- 配置结构上已经具备多职业扩展能力

### 验证手段

- 全局搜索 `ShootState`、`shoot` 等命名，确认公共层已去弓箭化
- 弓箭手仍然能从专属配置正确读取参数

### 测试用例

#### 用例 1：配置拆分后功能不变

- 操作：正常进入游戏并攻击
- 预期：箭速、射程、扇形角等仍正确

#### 用例 2：共有属性仍然生效

#### 用例 3：职业专属属性只影响对应职业

- 操作：调整 `archer.json` 中弓箭参数
- 预期：只影响弓箭手行为，不污染通用层

## 十、Step 1.7 - 拆 `PlayerControl` 周边运行时胶水

### 目标

继续给 `PlayerControl` 减肥，把“玩家运行时装配”从控制器主体里抽出去。

### 当前脏点

`PlayerControl` 现在同时装配：

- `EntityBuffMgr`
- `HitEffectMgr`
- `UpgradeManager`
- `SkillSystem`
- 鼠标世界坐标同步
- `DamagePopupMgr`
- `FlashWhite`

这说明它不只是控制器，还是一个运行时装配中心。

### 建议拆分

- `PlayerRuntime`：持有玩家局内运行时服务
- `PlayerServices`：暴露给技能 / 敌人 / UI 的薄接口
- `PlayerControl`：只负责节点桥接和最少量公共入口

### 重点涉及文件

- `player/PlayerControl.ts`
- `player/base/*`
- `skill/SkillTypes.ts`
- `upgrade/types.ts`

### 完成定义

- `PlayerControl` 主体不再直接 new 一堆服务对象
- 服务装配可以被测试和替换

### 验证手段

- 阅读 `PlayerControl.ts` import 列表，确认明显缩短
- 检查技能 / 升级构建上下文是否改由行为层或 runtime 服务提供

### 测试用例

#### 用例 1：玩家运行时服务正常挂载

- 场景：进入游戏初始化玩家
- 预期：Buff / Skill / Upgrade / HitEffect 仍全部可用

#### 用例 2：外部访问不穿透内部实现

- 场景：技能或 UI 访问玩家能力
- 预期：通过 façade / service 访问，而不是直接依赖 `PlayerControl` 内部字段

## 十一、Step 1.8 - 瘦身 `GameLoop` 与敌我交互接口

### 目标

处理这一阶段新增发现的两个基础架构问题：

1. `GameLoop` 过胖
2. 敌人直接依赖 `PlayerControl`

### 需要处理的内容

#### A. `GameLoop` 拆分

当前 `GameLoop` 同时负责：

- 创建 `World`
- 创建 ECS/System
- 初始化 `ProjectilePool`
- 资源预加载
- 副作用注册时机

建议拆出：

- `GameBootstrap`
- `ResourcePreloader`
- `RegistryLoader`

#### B. 敌我交互接口抽象

后续敌人和敌方投射物不应继续直接认识 `PlayerControl`。

建议预留：

- `ITargetable`
- `IAttackTarget`
- `IDamageable`

先允许用 façade 适配，不要求这一步全量替换完成。

### 重点涉及文件

- `core/GameLoop.ts`
- `enemy/minion/**/*.ts`
- `enemy/projectile/EnemyArrow.ts`
- `player/PlayerControl.ts`

### 完成定义

- `GameLoop` 只保留最小运行时驱动职责
- 敌人模块后续可以面向接口而不是玩家具体类

### 验证手段

- 全局搜索 `PlayerControl` 在敌人模块中的引用点，列出替换计划
- 启动后资源加载、对象池、输入更新流程不回归

### 测试用例

#### 用例 1：启动流程回归

- 场景：进入游戏
- 预期：`World`、资源、对象池仍正常初始化

#### 用例 2：敌人攻击玩家回归

- 场景：敌人近战或投射物命中玩家
- 预期：仍能正确造成伤害

#### 用例 3：接口替换预留验证

- 场景：引入目标接口后，敌人不直接依赖 `PlayerControl`
- 预期：功能不变，引用关系更干净

## 十二、本大步骤完成后的统一验收

完成 `重构步骤 1` 后，应该达到下面这些标准：

| 验收项 | 预期结果 |
|------|---------|
| 当前弓箭手功能是否保留 | 是 |
| `PlayerControl` 是否仍是 God object | 否 |
| 新增职业是否还需要改 `PlayerControl` | 否 |
| 公共状态机是否还带弓箭手命名 | 否 |
| 配置是否支持共有层 + 职业专属层 | 是 |
| `GameLoop` 是否仍是启动大杂烩 | 否 |
| 敌人模块是否必须直接依赖 `PlayerControl` | 否，至少已建立替换路径 |

## 十三、建议的手工回归清单

每次完成一个小步骤，都建议至少做这 8 个手工回归：

1. 进入场景，玩家正常出生
2. 玩家移动正常
3. 玩家停止后能攻击
4. 攻速正常影响射击间隔
5. 额外弹道仍然生效
6. 穿透 / 弹射 / 命中效果不丢失
7. 升级面板相关入口不报错
8. 技能系统相关入口不报错
9. 启动资源预加载不报错
10. 敌人攻击玩家不报错

## 十四、这一阶段不要做什么

不要在这一步里顺手做：

- Buff 原语归并
- 升级系统 registry 化
- 技能工厂化
- 统一攻击载荷
- `config` / `ui` 双根全量迁移

这些都是真的重要，但都不是这一阶段的主任务。

这一步只做一件事：

**把玩家从单职业控制器改成职业行为架构。**
