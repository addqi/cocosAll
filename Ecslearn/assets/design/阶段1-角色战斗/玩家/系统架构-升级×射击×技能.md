# 系统架构：升级系统 × 射击系统 × 技能系统

> 目标：完成全部步骤后，新增任意肉鸽升级效果 = **写 1 个 TS 类 + 1 个 JSON + index 加 1 行 import**，核心代码 0 改动，0 个 if/else。

---

## 一、三个系统总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                        肉鸽升级选择界面                               │
│                      （玩家三选一获取升级）                            │
└───────────────────────────┬─────────────────────────────────────────┘
                            │
                    ┌───────▼────────┐
                    │ UpgradeManager  │ ← 系统1：升级系统 + 被动效果
                    │ （分发 + 被动）  │
                    └──┬─────┬─────┬─┘
                       │     │     │
            ┌──────────┘     │     └──────────┐
            ▼                ▼                ▼
     ┌────────────┐  ┌────────────┐   ┌────────────┐
     │ Buff 系统   │  │ 射击系统    │   │ 技能系统    │
     │ (已有)      │  │ (新建)     │   │ (新建)     │
     │ 改数字      │  │ 5阶段管线  │   │ 主动技能    │
     └──────┬─────┘  └─────┬──────┘   └─────┬──────┘
            │              │                 │
            └──────────────┼─────────────────┘
                           │
                 ┌─────────▼─────────┐
                 │     属性图          │
                 │ EntityPropertyMgr  │
                 │  （唯一数据源）      │
                 └───────────────────┘
```

### 系统 1：升级系统（UpgradeManager）+ 被动效果

| 职责 | 说明 |
|------|------|
| 升级池管理 | 根据流派/进阶方向过滤可用升级 |
| 效果分发 | 读升级 JSON，按 `type` 字段分发到 Buff 系统 / 射击系统 / 技能系统 |
| **被动效果** | 被动技能（如自动速射）= 一种升级。选中即生效，不需要按键触发 |
| 升级移除 | 进化消耗时反向撤销 |
| 进化检测 | 每次获取新升级后检查进化配方 |

**被动技能归属于升级系统**：被动技能和普通升级的触发方式完全一样 — 选中后自动生效。
区别仅在于：被动可能有等级（Lv1→Lv2→Lv3），每级是一个独立升级条目。

### 系统 2：射击系统（ShootingPipeline）

| 职责 | 说明 |
|------|------|
| 5 阶段管线 | 触发 → 决议 → 配置 → 飞行 → 命中 |
| 读取属性 | 从属性图获取 AttackSpeed、ExtraProjectiles、PierceCount 等 |
| 命中效果管线 | 维护 `IHitEffect[]` 注册表，命中时顺序执行 |
| 策略切换 | 接受外部设置射击触发策略（IShootPolicy） |

**射击系统不知道"升级"是什么**，只知道属性值和已注册的命中效果。

### 系统 3：技能系统（SkillSystem）—— 仅主动技能

| 职责 | 说明 |
|------|------|
| 主动技能管理 | 持有玩家当前可用的主动技能列表 |
| 冷却管理 | 每个技能独立 CD，每帧 tick 递减 |
| 按键触发 | 监听 1/2/3 按键，触发对应技能 |
| 技能执行 | 调用射击系统或独立逻辑（箭雨 = 批量生成弹丸，闪身 = 位移 + 射击） |

**主动技能和被动技能完全分离**：
- 被动：选中 → 自动生效 → 归升级系统
- 主动：选中 → 装备到技能栏 → 按键手动触发 → 归技能系统

## 二、升级效果分类与分发规则

### 所有升级效果的 type 字段

| type 值 | 分发目标 | 适用场景 | 注册/工厂模式 |
|---------|----------|----------|---------------|
| `buff` | EntityBuffMgr | 改数字：攻速、攻击、穿透、额外弹道... | 已有 @buffEffect + BuffFactory |
| `hit_effect` | HitEffectMgr | 命中触发：灼烧、闪电链、击退、分裂... | 新建 @hitEffect + HitEffectFactory |
| `shoot_policy` | 射击系统阶段 1 | 改触发方式：按住连射、自动射击 | 策略对象直接引用（无需工厂） |
| `active_skill` | SkillSystem | 装备主动技能：箭雨、闪身 | 技能系统内部管理 |

## 三、射击系统 5 阶段管线

```
  阶段1 ──→ 阶段2 ──→ 阶段3 ──→ 阶段4 ──→ 阶段5
  触发判定    射击决议    弹丸配置    弹丸飞行    命中结算
  ─────────────────────────────────────────────────
  "该不该射"  "射几支"    "每支箭     "飞行中     "命中后
              "什么方向"   什么属性"    怎么动"     做什么"
```

### 阶段 1：触发判定

| 项 | 说明 |
|----|------|
| 输入 | ActionComp（输入状态）、是否有目标、是否移动中 |
| 输出 | boolean：是否进入/停留在射击状态 |
| 扩展方式 | `IShootPolicy` 策略对象，最高优先级覆盖 |
| 决定方式 | **互斥选择**：同时只有一个策略生效 |

```typescript
interface IShootPolicy {
    readonly priority: number;
    wantShoot(input: ActionComp, hasTarget: boolean, isMoving: boolean): boolean;
}
```

策略实现（总共 3-4 个，不需要工厂/JSON）：

| 策略 | priority | wantShoot 逻辑 |
|------|----------|----------------|
| ClickToShoot | 0 | justPressed Attack |
| HoldToShoot | 1 | active Attack |
| AutoShootLv1 | 2 | hasTarget && !isMoving |
| AutoShootLv2 | 2 | hasTarget（移动中射速 -30% 由属性控制） |
| AutoShootLv3 | 2 | hasTarget |

PlayerControl 持有 `IShootPolicy`，升级切换时直接替换引用。

### 阶段 2：射击决议

| 项 | 说明 |
|----|------|
| 输入 | 属性图、目标列表 |
| 输出 | `ShotDescriptor[]`（每支箭的起点、终点、曲线） |
| 扩展方式 | **读属性值**，代码内计算 |

核心逻辑（ShootResolver）：

```
arrowCount    = 1 + prop.getValue(ExtraProjectiles)
spreadAngle   = prop.getValue(SpreadAngle)    // 默认 15°
attackSpeed   = prop.getValue(AttackSpeed)     // 决定冷却间隔
arrowSpeed    = prop.getValue(ArrowSpeed)      // 决定飞行时间
```

新增升级"多重射击 +1 箭"只需要一个 JSON，ExtraProjectiles 属性 +1，ShootResolver 自然多射一支。

### 阶段 3：弹丸配置

| 项 | 说明 |
|----|------|
| 输入 | 属性图 |
| 输出 | `ProjectileConfig`（每支箭的行为参数） |
| 扩展方式 | **从属性图拷贝到箭** |

```typescript
interface ProjectileConfig {
    pierceCount: number;       // 穿透次数
    bounceCount: number;       // 弹射次数
    splitCount: number;        // 分裂数量
    splitDamageRatio: number;  // 分裂伤害系数
    homingStrength: number;    // 追踪强度
    damageRatio: number;       // 当前伤害系数（穿透衰减用）
    hitEffects: IHitEffect[];  // 从 HitEffectMgr 获取当前激活的命中效果列表
}
```

箭出生时把属性值快照到 config 上。后续飞行和命中只读 config，不再查属性图。

### 阶段 4：弹丸飞行

| 项 | 说明 |
|----|------|
| 输入 | 箭自身 ProjectileConfig + 当前位置 |
| 输出 | 位置更新、碰撞检测 |
| 扩展方式 | **参数驱动**，ArrowProjectile 扩展 |

飞行行为由 config 参数决定：
- `homingStrength > 0` → 飞行中持续修正朝向目标
- 到达终点或碰撞目标 → 进入阶段 5

### 阶段 5：命中结算

| 项 | 说明 |
|----|------|
| 输入 | `HitContext`（攻击者、目标、命中位置、伤害结果） |
| 输出 | 各种副作用（伤害、施加 buff、生成子弹丸...） |
| 扩展方式 | **`IHitEffect[]` 注册管线（装饰器 + 工厂 + JSON）** |

```typescript
interface HitContext {
    attacker: {
        combat: PlayerCombat;
        prop: PlayerProperty;
        buffMgr: EntityBuffMgr;
    };
    target: {
        combat: EnemyCombat;
        control: EnemyControl;
        buffMgr: EntityBuffMgr;
    };
    hitPos: Vec3;
    projectileConfig: ProjectileConfig;
    damageResult: DamageResult | null;  // 第一个 DamageEffect 填充
    isKill: boolean;                    // takeDamage 后判断
}
```

```typescript
abstract class HitEffectBase {
    data: HitEffectData;
    constructor(data: HitEffectData) { this.data = data; }
    abstract onHit(ctx: HitContext): void;
}
```

命中效果注册模式（完全复制 Buff 系统）：

```
@hitEffect 装饰器  →  HitEffectFactory.register(类名)
HitEffectFactory   →  根据 JSON effectClass 字符串创建实例
hitEffects/index.ts →  副作用 import 触发注册
```

**命中后弹道后续行为**（在所有 hitEffect 执行完之后）：

```
if (config.pierceCount > 0)   → 继续飞行，pierceCount--, damageRatio 衰减
else if (config.bounceCount > 0) → 寻找下一个目标，重新构建曲线飞过去
else if (config.splitCount > 0)  → 生成 N 支子弹丸，各自走阶段 3-5
else                              → 回池
```

穿透/弹射/分裂不是命中效果，是**弹道后续行为**，由 ProjectileConfig 参数驱动。
它们的值来自属性图，通过 Buff 系统修改 — 所以"穿透 +2"只需要一个 JSON buff。

## 四、HitEffect 注册系统规格

### 基础设施（baseSystem/hitEffect/）

| 文件 | 职责 |
|------|------|
| `HitEffectBase.ts` | 抽象基类：`data` + `abstract onHit(ctx)` |
| `hitEffect.ts` | `@hitEffect` 装饰器：`HitEffectFactory.register(ctor.name, ctor)` |
| `HitEffectFactory.ts` | 工厂：`register(name, ctor)` + `create(data): HitEffectBase` |
| `types.ts` | `HitEffectData`、`HitContext` 类型定义 |
| `index.ts` | 对外导出 |

### 游戏层效果（game/hitEffects/）

| 文件 | 效果 | onHit 逻辑 |
|------|------|-----------|
| `DamageHitEffect.ts` | 基础伤害 | 读攻击属性 → takeDamage → 填充 ctx.damageResult |
| `LifestealHitEffect.ts` | 吸血 | 读 ctx.damageResult → 按 lifestealRate 回血 |
| `BurnOnHitEffect.ts` | 灼烧 | 给目标 addBuff(灼烧DOT) |
| `FrostOnHitEffect.ts` | 冻伤 | 给目标 addBuff(减速/冰冻) |
| `ChainLightningEffect.ts` | 闪电链 | 概率触发 → 周围敌人 AOE |
| `KnockbackEffect.ts` | 击退 | 给目标施加位移力 |
| `SplitOnHitEffect.ts` | 分裂产卵 | 生成 N 支子弹丸飞向周围敌人 |
| `index.ts` | 注册入口 | 副作用 import 所有效果类 |

**新增一个命中效果的完整步骤**：

1. 在 `game/hitEffects/` 写一个 TS 类，继承 `HitEffectBase`，加 `@hitEffect` 装饰器
2. 在 `game/config/hitEffectConfig/` 写一个 JSON，`effectClass` 对应类名
3. 在 `game/hitEffects/index.ts` 加一行 `import './XxxEffect'`
4. 在升级 JSON 的 `effects` 数组里引用该 hitEffectData
5. **核心代码改动 = 0**

## 五、UpgradeManager 分发逻辑

```typescript
class UpgradeManager {
    apply(config: UpgradeConfig): void {
        for (const eff of config.effects) {
            switch (eff.type) {
                case 'buff':
                    this._buffMgr.addBuff(eff.buffData, this._buffOwner);
                    break;
                case 'hit_effect':
                    const he = HitEffectFactory.create(eff.hitEffectData);
                    if (he) this._hitEffectMgr.add(config.id, he);
                    break;
                case 'shoot_policy':
                    this._shootCtrl.setPolicy(ShootPolicyMap[eff.policy]);
                    break;
                case 'active_skill':
                    this._skillSystem.equip(eff.skillId);
                    break;
            }
        }
        this._checkEvolution();
    }

    remove(upgradeId: string): void {
        // 反向撤销：removeBuff、hitEffectMgr.remove、恢复策略等
    }
}
```

**这是整个项目唯一的 switch。** 4 个 case，对应 4 种 effect type。新增 effect type（几乎不会发生）才需要加 case。

## 六、被动效果 vs 主动技能 对比

| 维度 | 被动效果（归升级系统） | 主动技能（归技能系统） |
|------|----------------------|----------------------|
| 触发方式 | 选中后**立即自动生效** | 选中后装备到技能栏，**按键手动触发** |
| 输入 | 无需玩家操作 | 需要按 1/2/3 键 |
| 冷却 | 无（永久生效或条件触发） | 有独立 CD |
| 例子 | 急速连射、多重射击、烈焰箭、自动速射 | 箭雨倾泻、闪身射击 |
| 归属 | UpgradeManager.apply() 分发 | SkillSystem.equip() 装备 |
| 升级获取 | 肉鸽三选一选中 | 肉鸽三选一选中（type: active_skill） |
| 运行时管理 | BuffMgr / HitEffectMgr / ShootPolicy | SkillSystem.tick() + 按键监听 |

**被动技能就是一种升级**，只不过 JSON 里的 effect 可能包含 `shoot_policy` 或复合效果。
**主动技能是独立实体**，有自己的执行逻辑（箭雨 = 在区域内批量生成弹丸）。

## 七、属性图扩展

### 当前已有属性

| EPropertyId | 用途 |
|-------------|------|
| Hp | 生命值 |
| Attack | 攻击力 |
| Defense | 防御力 |
| CritRate | 暴击率 |
| CritDmg | 暴击伤害 |

| MoveSpeed | 移动速度 |
| LifestealRate | 吸血率 |

### 需要新增的属性

| EPropertyId | 用途 | 哪些升级写它 |
|-------------|------|-------------|
| ExtraProjectiles | 额外弹道数 | 多重射击、弹幕射手 |
| PierceCount | 穿透次数 | 穿透射击、无限穿透 |
| BounceCount | 弹射次数 | 连锁弹射 |
| SpreadAngle | 扇形角度 | 扫射模式 |
| HomingStrength | 追踪强度 | 追猎本能 |
| ArrowSpeed | 箭速 | 箭矢轻量化（从硬编码 config 迁移） |
| AttackRange | 攻击范围 | 远程瞄准（从硬编码 config 迁移） |

新增属性 = 1 个 JSON 文件 + attributeConfigs.ts 引入 + propertyEnum.ts 加枚举。
之后所有改这些值的升级都可以用 `SimpleAttrBuffEffect` + JSON 搞定。

## 九、验收标准

当 Step 0 ~ Step 7 全部完成后，以下操作**不需要修改任何核心代码**：

| 操作 | 需要做的事 |
|------|-----------|
| 新增一个纯属性升级 | 写 1 个 upgrade JSON（type: buff） |
| 新增一个命中效果 | 写 1 个 TS 类 + 1 个 hitEffect JSON + index 加 1 行 + upgrade JSON 引用 |
| 新增一个弹道属性升级 | 写 1 个 upgrade JSON（type: buff，targetAttr 指向弹道属性） |
| 改变射击触发方式 | 写 1 个 upgrade JSON（type: shoot_policy） |
| 新增一个进化组合 | 在进化配方 JSON 里加一条 |

**项目中不会出现**：
- `if (hasUpgrade('flame_arrow'))` 这样的硬编码检查
- `switch (currentBuildType)` 这样的流派分支
- 在 ArrowProjectile 里 `if (isPoisoned)` 这样的特殊情况

所有效果通过**注册 + 数据驱动**自动组合，核心代码只提供管线和属性图。
