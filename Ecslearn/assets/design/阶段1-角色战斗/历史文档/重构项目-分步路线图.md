## 一、核心判断

这不是“顺手整理一下代码”，这是一次必须分阶段做的结构重构。

如果你试图一步到位同时改：

- 玩家职业
- 升级系统
- Buff 系统
- 技能系统
- 命中效果
- 投射物

那大概率会把项目搞死。

正确做法是：

1. 先固定数据边界
2. 再切掉硬编码分支
3. 再引入注册表和 JSON 配置
4. 最后再把复杂技能和职业扩展迁过去

这轮梳理后，额外确认了 5 个必须纳入重构路线的耦合点：

1. `PlayerControl` 是局内运行时总胶水，必须继续减肥
2. `GameLoop` 同时承担组合根、资源预加载、对象池初始化和副作用注册，必须拆
3. `game/entity/` 目录名和职责不匹配，本质上是战斗运行时服务层
4. `config/` 与 `game/config/` 双根并存，且纯数据层反向依赖 `game` 实现，必须收口
5. `ui/` 与 `game/ui/` 双根并存，后续会继续制造重复模式，必须统一

## 二、最终目标结构

重构完成后，项目应该稳定在下面这套职责划分上：

| 模块 | 负责什么 | 不负责什么 |
|------|----------|------------|
| `core/bootstrap` | 组合根、启动顺序、运行时装配 | 不承载玩家/技能具体业务 |
| `actors/player` | 玩家外壳、职业行为、玩家运行时服务 | 不负责升级总分发 |
| `actors/enemy` | 敌人行为、敌人表现、敌我目标抽象 | 不直接依赖玩家具体实现 |
| `combat/` | 攻击执行、投射物、命中效果、战斗运行时服务 | 不关心升级来源 |
| `progression/upgrade` | 读取升级 JSON、分发效果、检查进化 | 不负责具体技能执行 |
| `progression/skill` | 主动技能、冷却、施法入口 | 不直接操作职业私有实现 |
| `status/buff` | 持续状态、属性修饰、周期行为 | 不负责肉鸽升级选择 |
| `config/*.json` | 运行时配置唯一来源 | 不包含程序逻辑 |
| `presentation/ui` | 局内 HUD、弹字、表现 UI | 不承担玩法状态机逻辑 |

一句话总结：

**程序只写“行为原语脚本”，策划/程序通过 JSON 组合具体效果。**

### 目录收口原则

当前 `assets/script` 不是文件夹不够，而是切分维度混用。后续目录整理必须遵守下面 4 条：

1. 不再同时维护两套配置根：`config/` 和 `game/config/` 必须收口为单根
2. 不再同时维护两套 UI 根：`ui/` 和 `game/ui/` 必须收口为单根
3. 战斗流水线相关目录要逐步收拢：`shoot/`、`projectile/`、`hitEffects/`、`entity/` 逻辑上归入 `combat/`
4. 先做语义收口和依赖收口，再做物理文件迁移；不要为了改目录而改目录

## 三（附）、分步重构顺序

## Phase 0：先立规矩，不改行为

目标：先把“什么归谁管”写死，避免后面一边改一边跑偏。

### Step 0.1：明确配置源

- 运行时只读 JSON
- Excel 不接运行时
- Excel 未来只做导出源

### Step 0.2：明确四类效果边界

- `Buff`：持续状态
- `HitEffect`：命中附加效果
- `Skill`：主动施法
- `Upgrade`：局内长期获取的效果分发

### Step 0.3：冻结旧接口

以下接口尽量保持外部不变，避免大面积破坏：

- `PlayerControl.instance`
- `EntityBuffMgr`
- `HitEffectMgr`
- `SkillSystem`

### Step 0.4：冻结目录收口策略

- `game/entity/` 后续按“战斗运行时服务层”处理，不再继续往里塞业务对象
- `config/` 只允许放纯数据与纯类型，不允许再反向依赖 `game` 实现
- `ui/` 和 `game/ui/` 明确后续只保留一个入口
- `game/roguelike/` 不再长期保持占位目录，后续要么接入真实内容，要么不暴露为独立模块

验收标准：

- 设计文档达成统一口径
- 后续改造都围绕同一套职责边界展开

---

## Phase 1：先拆玩家职业，别碰太多别的

目标：把 `PlayerControl` 这个 God object 先切开。

### Step 1.1：建立 `baseSystem/player`

- `PlayerBehaviorBase`
- `PlayerBehaviorFactory`
- `@playerBehavior`

### Step 1.2：建立 `player/base`

- 通用 `PlayerBehavior`
- 通用攻击状态接口
- 通用职业上下文类型

### Step 1.3：把弓箭手逻辑搬进 `ArcherBehavior`

- 从 `PlayerControl` / `PlayerShootState` 提取弓箭实现
- `PlayerControl` 只保留通用生命周期和外部兼容接口

### Step 1.4：预留战士 / 召唤师骨架

- 先建空壳，不急着写完整逻辑

### Step 1.5：拆 `PlayerControl` 周边胶水

- 引入 `PlayerRuntime` / `PlayerServices` 概念
- 把 Buff / Skill / Upgrade / HitEffect 装配从 `PlayerControl` 主体里抽出去
- 为敌人和技能访问玩家提供更薄的 façade

### Step 1.6：瘦身 `GameLoop`

- 抽离 `Bootstrap`
- 抽离资源预加载器
- 抽离副作用注册入口
- `GameLoop` 只保留帧驱动和最小运行时协调

验收标准：

- 玩家职业切换不需要改核心控制器
- 新增职业不再修改 `PlayerControl`

---

## Phase 2：重构升级系统，消灭硬编码 switch

目标：升级系统只做“效果分发”，不理解职业私货。

### Step 2.1：建立 `UpgradeEffectRegistry`

- 注册 `buff`
- 注册 `hit_effect`
- 注册职业专属 effect handler

### Step 2.2：`UpgradeManager` 去掉 `switch`

- `apply` 走 registry
- `remove` 走 registry

### Step 2.3：升级配置加职业标签

- `classTag`
- `poolTag`
- `evolvesFrom`

### Step 2.4：把 `shoot_policy` 从通用层移出去

- 不再让 `UpgradeManager` 直接创建 `AutoShoot`
- 改由职业行为模块注册自己的处理器

验收标准：

- `upgrade/` 不再依赖 `IShootPolicy`
- 新增升级效果类型只需要注册 handler

---

## Phase 3：重构 Buff 与命中效果，抽出行为原语

目标：把重复的效果类压缩成少量基础脚本。

### Step 3.1：Buff 效果归并

基础原语建议保留：

- `AttrModifierEffect`
- `PeriodicDamageEffect`
- `PeriodicHealEffect`
- 后续再加 `ShieldEffect`
- 后续再加 `StateControlEffect`

### Step 3.2：命中效果归并

基础原语建议保留：

- `DamageHitEffect`
- `ApplyBuffOnHitEffect`
- `BonusDamageOnConditionEffect`
- `HealOnHitEffect`
- `ImpulseOnHitEffect`
- `SpawnAreaOnHitEffect`

### Step 3.3：统一 JSON 配置格式

- Buff 用 `buffs.json`
- 命中效果用 `hitEffects.json`
- 升级效果引用这些定义，而不是重复散落参数

### Step 3.4：收口战斗运行时目录

- `EntityBuffMgr`
- `HitEffectMgr`
- `AttributeChangeResolver`
- 其他战斗胶水服务

逻辑上统一归到 `combat/runtime`

验收标准：

- 新增一个常规 debuff 不再新建 TS 文件
- 新增一个属性 buff 只写 JSON

---

## Phase 4：重构技能系统，建立主动技能原语层

目标：技能系统不再直接绑定弓箭手投射物实现。

### Step 4.1：SkillContext 去职业私货

- 删除 `currentShootPolicy`
- 删除 `setShootPolicy`
- 改为通用 `behavior` / `services`

### Step 4.2：建立 `SkillFactory`

- `skills.json`
- `skillClass`
- `params`

### Step 4.3：抽主动技能原语

优先增加：

- `SpawnProjectileSkill`
- `AreaDamageSkill`
- `ApplySelfBuffSkill`
- `DashAttackSkill`
- `SummonSkill`

### Step 4.4：把现有技能迁移

- `ArrowStormSkill`
- `DashShotSkill`

### Step 4.5：建立中立战斗目标接口

- 敌人 AI 不再直接依赖 `PlayerControl`
- 敌方投射物不再直接依赖玩家具体类
- 逐步引入 `ITargetable` / `IAttackTarget` 这类中立接口

验收标准：

- 技能只关心“放什么攻击”或“施加什么状态”
- 不再直接引用弓箭手专属配置

按价值排序，当前最值得先做的是：

1. `PlayerBehavior` 架构
2. `UpgradeEffectRegistry`

## 五、每阶段的硬验收规则

每完成一个阶段，都必须满足：

1. 项目可编译
2. 核心战斗可运行
3. 旧接口不大面积破坏
4. 不引入新的职业硬编码分支
5. 不引入新的配置重复

如果某一步需要同时改 8 个系统，那说明拆分方式错了，应该重新切步骤。

## 六、结论

这次重构的本质不是“把文件夹变漂亮”。

本质是把项目从：

**具体内容驱动代码**

改成：

**基础行为原语 + JSON 组合内容**

只有做到这一步，后面战士、召唤师、火球、毒云、流血、护盾、召唤狼这些内容才不会把项目拖进屎坑。
