### ECS 边界

| 实体 | 是否进 ECS | 原因 |
|------|-----------|------|
| 小怪 | ✅ | 数量多，需要 System 批量驱动 AI/移动/攻击 |
| 精英怪 | ✅ | 共享小怪 System + 额外 EliteBrainComp |
| 飞行物 | ✅ | 数量多，玩家/怪物弹幕统一管线 |
| 玩家 | ❌ | 已有 Proxy Entity 模式，保持不变 |

| 档位 | 行为复杂度 | AI 驱动方式 | 攻击方式 |
|------|-----------|------------|---------|

| Boss | 高 | StateMachine（多阶段 + 技能系统） | 独立技能，多阶段 |

## 二、战斗抽象层（Phase 0 前置）

### 现状问题

1. `PlayerCombat` 和 `EnemyCombat` 的 `takeDamage`/`heal` 是同一公式的两份拷贝
2. `GameHitContext` 单向硬编码：attacker = PlayerCombat，target = EnemyCombat
3. `PlayerCombat.attack()` 与 HitEffect 管线职责重叠

### ICombat 接口

```typescript
interface ICombat {
    takeDamage(rawDmg: number): number;
    heal(amount: number): number;
    reset(): void;
    readonly currentHp: number;
    readonly maxHp: number;
    readonly isDead: boolean;
}
```

统一为 `EntityCombat` 类，替代 `PlayerCombat` / `EnemyCombat`。

### ICombatEntity 接口

```typescript
interface ICombatEntity {
    readonly prop: EntityPropertyMgr;
    readonly combat: ICombat;
    readonly buffOwner: IBuffOwner;
    readonly buffMgr: EntityBuffMgr;
    readonly node: Node;
}
```

`PlayerControl`、`MobControl`、`BossControl` 均实现此接口。

### GameHitContext 双向化

```typescript
interface GameHitContext {
    attacker: ICombatEntity;
    target: ICombatEntity;
    baseDamage: number;
    rawDamage: number;
    finalDamage: number;
    isCrit: boolean;
    critMultiplier: number;
    totalHealed: number;
}
```

所有 HitEffect 通过 `attacker`/`target` 操作，不区分玩家/怪物方向。

### 关键组件定义

```typescript
// 移动模式
enum EMovePattern {
    Chase,              // 无脑追踪
    CircleStrafe,       // 绕圈接近
    ChargeAndRetreat,   // 冲撞后后退
    Stationary,         // 站桩远程
    Flee,               // 远离目标（回血型用）
}

// 攻击模式
enum EAttackPattern {
    Melee,              // 近战
    RangedSingle,       // 远程单发
    RangedBurst,        // 远程连射
    AreaSlam,           // 范围砸地
    SelfDestruct,       // 自爆
}

class AIComp implements IComponent {
    movePattern: EMovePattern = EMovePattern.Chase;
    detectionRange: number = 200;
    detected: boolean = false;
}

class AttackComp implements IComponent {
    pattern: EAttackPattern = EAttackPattern.Melee;
    range: number = 50;
    windUpDuration: number = 0.5;
    cooldown: number = 1.0;
    damage: number = 10;
    // 运行时
    state: 'idle' | 'windup' | 'execute' | 'recovery' = 'idle';
    timer: number = 0;
}

class CombatRefComp implements IComponent {
    constructor(public ref: ICombatEntity) {}
}
```

## 四、ECS System 管线

### 更新顺序

```
GameLoop.update(dt):
  // ── 玩家链（已有）──
  RawInputSystem
  ActionMapSystem
  PlayerControlSystem

  // ── 怪物 AI 链（新增）──
  MobDetectionSystem         ← 检测玩家距离 → AIComp.detected
  MobAISystem                ← movePattern + IPathProvider → VelocityComp
  EliteBrainSystem           ← EliteBrainComp → policy.evaluate + behavior.update
  MobAttackDecisionSystem    ← 进入攻击范围 → AttackComp.state = windup
  MobAttackExecuteSystem     ← windUp 到期 → 执行攻击 → recovery

  // ── 飞行物链（新增）──
  ProjectileMoveSystem       ← 曲线/直线飞行 → 位置更新
  ProjectileCollisionSystem  ← 碰撞层检测 → HitEffect 管线

  // ── 通用链 ──
  MoveSyncSystem             ← 所有 VelocityComp + NodeRefComp → 节点位置同步
  AttackIndicatorSystem      ← 红圈生命周期（创建 / 缩放 / 销毁）
```

### 各 System 职责

| System | 过滤条件 | 输入 | 输出 |
|--------|---------|------|------|
| MobDetectionSystem | MobTag + AIComp + NodeRefComp | 玩家位置 | AIComp.detected |
| MobAISystem | MobTag + AIComp + VelocityComp（无 EliteBrainComp） | AIComp.movePattern, IPathProvider | VelocityComp |
| EliteBrainSystem | EliteBrainComp | EliteCtx | VelocityComp + AttackComp（由行为修改） |
| MobAttackDecisionSystem | MobTag + AttackComp + AIComp | 玩家距离, AttackComp.state | AttackComp.state 转移 |
| MobAttackExecuteSystem | AttackComp (state=execute) | CombatRefComp, 玩家 ICombatEntity | HitEffect 管线 |
| ProjectileMoveSystem | ProjectileComp | curve, elapsed | NodeRefComp 位置 |
| ProjectileCollisionSystem | ProjectileComp + CollisionLayerComp | 碰撞层目标列表 | HitEffect 管线 |
| AttackIndicatorSystem | AttackComp (state=windup) | timer, range | 红圈节点缩放/销毁 |

## 七、精英行为系统

### 行为单元 IEliteBehavior

```typescript
interface IEliteBehavior {
    readonly id: string;
    enter(ctx: EliteCtx): void;
    update(ctx: EliteCtx, dt: number): void;
    exit(ctx: EliteCtx): void;
    readonly isComplete: boolean;
}
```

### EliteCtx

```typescript
interface EliteCtx {
    entity: ICombatEntity;
    node: Node;
    body: Node;
    velocity: VelocityComp;
    attack: AttackComp;
    anim: MobAnimation;
    playerRef: ICombatEntity;
    pathProvider: IPathProvider;
}
```

### 行为库

| 行为 | id | 逻辑 |
|------|-----|------|
| 冲撞 | `charge` | 锁定玩家位置 → 高速冲刺 → 到达后短暂硬直 |
| 近战连击 | `melee_combo` | 停下 → 攻击动画 × N → 每次红圈判定 |
| 远程弹幕 | `ranged_burst` | 停下 → 向玩家发射 N 发弹幕（走飞行物管线） |
| 撤退回血 | `retreat_heal` | 远离玩家 → 安全距离 → 回血动画 → heal() |
| 分裂 | `split` | 分裂动画 → 生成 N 个小怪 → 自身缩小或死亡 |
| 范围砸地 | `area_slam` | 蓄力红圈 → 范围伤害 |
| 召唤 | `summon` | 召唤动画 → 生成 N 个小怪 |
| 自我强化 | `buff_self` | 给自己 addBuff |

注册模式：`@eliteBehavior` 装饰器 + `EliteBehaviorFactory`，复用 Buff/HitEffect 注册模式。

**新增一种精英行为 = 写 1 个 TS 类 + 装饰器注册。核心代码 0 改动。**

### 切换策略 IPhasePolicy

```typescript
interface IPhasePolicy {
    evaluate(ctx: EliteCtx, currentBehaviorId: string): string | null;
}
```

| 策略 | 逻辑 |
|------|------|
| SequentialPolicy | 当前行为 isComplete → 下一个，循环 |
| HpThresholdPolicy | HP% 区间 → 映射行为 |
| RandomPolicy | 当前行为 isComplete → 随机选 |
| ConditionalPolicy | 通用条件（hp < 0.3 → retreat_heal） |
| CompositePolicy | 组合多策略，按优先级首个非 null 生效 |

## 九、攻击指示器（红圈）系统

### 攻击生命周期

```
idle → [进入攻击范围] → windup → execute → recovery → idle
                          │                    │
                      创建红圈              销毁红圈
                    红圈逐渐缩小          (或命中特效)
                    玩家可闪避
```

### 实现要点

- 红圈节点池化（类似 ProjectilePool）
- windup 阶段：红圈从大缩到攻击范围大小，进度 = 1 - timer/windUpDuration
- execute 瞬间：检测玩家是否在范围内 → 造成伤害
- 不同 AttackPattern 的红圈形状不同：
  - Melee → 圆形（以怪为中心）
  - RangedSingle → 不需要红圈（飞行物碰撞）
  - AreaSlam → 大圆形（目标位置）

## 十、难度缩放架构

### 三幕结构（类杀戮尖塔）

| 幕 | 普通房 | 精英房 | Boss 房 |
|----|-------|--------|---------|
| Act 1 | 小怪 ×N 房 | — | 1 精英怪 |
| Act 2 | 小怪+精英 ×N 房 | 精英房 | 1 Boss |
| Act 3 | 混合 ×N 房 | 精英+小Boss | 1 大 Boss |

### 难度叠加机制

```
怪出生时的 Buff 挂载顺序：
1. 基础属性     ← mob JSON 的 baseStats
2. 幕全局 Buff  ← ActManager.actBuffs（每通幕累加）
3. 场地 Buff    ← RoomManager.terrainBuffs（如火山燃火之环）
4. 精英天赋 Buff ← innateBuffs（精英自带强化）
```

全部走 `EntityBuffMgr.addBuff()` — 复用已有 Buff 基础设施。

### 场地 Buff 示例

| 场地 | 效果 | 实现 |
|------|------|------|
| 火山 | 燃火之环：玩家靠近怪物时受持续伤害 | 怪物出生时挂 "FireAura" Buff（tick 类） |
| 冰原 | 全怪移速 +20%，玩家移速 -10% | 怪物挂 MoveSpeed Buff + 玩家挂 MoveSpeed 负 Buff |
| 墓地 | 怪物死亡后 3s 复活一次 | 怪物挂 "Undying" Buff（onRemove 触发复活逻辑） |

### 后期行为强化

不通过代码实现，通过 Buff 叠加：
- "Act2 全怪攻速 +30%" → AttackSpeed Mul Buff
- "Act3 全怪血量 +50%" → Hp Mul Buff
- 后期小怪种类更多 → RoomConfig 的 mob 列表配更多种类

## 十二、验收标准

当 Phase 0 ~ Phase 7 全部完成后：

| 操作 | 需要做的事 |
|------|-----------|
| 新增一种小怪 | 写 1 个 mob JSON + 1 套动画资源 |
| 新增一种精英行为 | 写 1 个 TS 类（IEliteBehavior）+ @eliteBehavior 装饰器 + index 加 1 行 |
| 新增一种精英怪 | 写 1 个 elite JSON（组合已有行为 + 选择切换策略） |
| 调整难度曲线 | 修改 ActConfig JSON 的 actBuffs 数值 |
| 新增场地效果 | 写 Buff JSON + RoomConfig 引用 terrainBuffs |
| 新增 Boss | 写 1 个 BossControl + 状态定义（Boss 复杂度高，不追求纯 JSON） |

**项目中不会出现**：
- `if (mobType === 'slime')` 这样的怪物类型硬编码
- `switch (elitePattern)` 这样的精英模式分支
- 在 MobAISystem 里 `if (isBoss)` 这样的特殊情况
- 玩家攻击怪物和怪物攻击玩家走两套不同的伤害管线

所有效果通过 **注册 + 数据驱动 + 碰撞层** 自动组合，核心代码只提供 System 管线和 ICombatEntity 接口。
