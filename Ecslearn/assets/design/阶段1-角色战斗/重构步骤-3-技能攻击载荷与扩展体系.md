# 重构步骤 3 - 技能、攻击载荷与扩展体系

> 目标：把技能系统从“当前两个技能的特化实现”重构为“主动技能原语 + 攻击载荷 + 可扩展内容框架”。
>
> 当前约束：
> - 运行时只读 JSON
> - Excel 不接运行时
> - 本阶段建立的是长期扩展地基，不追求一步把所有技能都数据化

---

## 一、这一步到底在解决什么问题

当前技能层的核心问题不是“技能少”，而是**技能直接操心了太多底层实现**。

比如一个技能类可能同时负责：

- 读职业专属配置
- 找敌人
- 算轨迹
- 创建投射物
- 组装命中回调
- 调用命中效果管线
- 修改射击策略

这种结构只能支撑少量 demo 技能。  
一旦你要加：

- 火球
- 冰环
- 冲锋斩
- 召唤狼
- 环绕飞剑
- 爆炸箭

项目会迅速烂掉。

所以第三阶段要解决的核心问题是：

**把技能从“具体实现拼装器”改成“施法入口 + 攻击请求生产者”。**

---

## 二、阶段目标

完成本大步骤后，项目应该满足：

1. `SkillContext` 不再泄漏职业私货
2. 技能可以通过工厂 + JSON 创建
3. 常见主动技能被收敛为少量技能原语
4. 普攻、主动技能、未来召唤物都能逐步走统一攻击载荷
5. 新增技能不再默认复制旧技能脚本
6. `shoot/`、`projectile/`、`hitEffects/`、战斗运行时服务开始逻辑收口到 `combat/`
7. 敌人、召唤物、玩家后续都可以通过中立战斗接口接入统一攻击链

---

## 三、总实施顺序

本大步骤拆成 10 个小步骤：

1. 先清理技能上下文
2. 再建立技能定义和工厂
3. 再抽第一批技能原语
4. 再迁移现有技能
5. 再定义统一攻击载荷
6. 再建立攻击执行器
7. 再收口 combat 目录
8. 再建立中立战斗接口
9. 再补未来扩展原语
10. 最后做系统级回归

顺序原因很简单：

没有统一上下文，就没法做工厂。  
没有工厂和原语，就没法做数据驱动。  
没有载荷层，就没法让技能和普攻共用攻击逻辑。

## 四、Step 3.1 - 清理 `SkillContext`

### 目标

去掉技能上下文中的职业私货。

### 当前脏点

当前 `SkillContext` 直接暴露：

- `currentShootPolicy`
- `setShootPolicy()`

这明显是弓箭手私货，不该出现在通用技能接口里。

### 正确做法

技能上下文只保留通用内容：

- `playerProp`
- `playerCombat`
- `playerNode`
- `buffMgr`
- `buffOwner`
- `hitEffectMgr`
- `mouseWorldPos`
- `behavior`
- `services`

### 设计要求

- 职业私有能力通过 `behavior` 或专用 service 暴露
- 技能层不直接依赖 `shoot/*`

### 完成定义

- 通用技能不再被弓箭手概念污染
- 战士 / 召唤师技能上下文可直接复用

### 验证手段

- 全局搜索 `SkillTypes.ts`，确认移除 `IShootPolicy`
- 编译检查技能实现是否仍然可用

### 测试用例

#### 用例 1：通用技能上下文构建

- 场景：构建一个不依赖射击策略的技能
- 预期：可直接运行

#### 用例 2：弓箭手私有能力通过 behavior 暴露

- 场景：需要切换射击模式的技能
- 预期：通过 `behavior` 调用，不污染通用接口

## 五、Step 3.2 - 建立 `SkillFactory` 与 `skills.json`

### 目标

让技能定义和技能实例化分离。

### 建议新增内容

- `skill/SkillFactory.ts`
- `config/skillConfig/skills.json`

### 技能定义建议字段

- `id`
- `name`
- `classTag`
- `skillClass`
- `params`
- `cooldown`
- `icon`
- `levelScaling`
- `tags`

### 设计要求

- 技能类只定义行为原语
- 技能具体参数来自 JSON
- 同一个 `skillClass` 可支持多个不同技能实例

### 完成定义

- 新增一个技能不再默认改 `SkillSystem`
- 技能实例可通过配置创建

### 验证手段

- 用现有技能做工厂化试点
- 从 JSON 创建 `ArrowStormSkill` / `DashShotSkill`

### 测试用例

#### 用例 1：合法技能配置创建成功

- 输入：有效 `skillClass`
- 预期：返回对应技能实例

#### 用例 2：非法技能配置处理

- 输入：不存在的 `skillClass`
- 预期：有明确报错，不静默失败

#### 用例 3：参数注入有效

- 场景：同一 `skillClass` 不同参数生成两种技能
- 预期：行为差异来自 JSON，而不是复制脚本

## 六、Step 3.3 - 抽第一批技能原语

### 目标

用少量技能原语覆盖 80% 的主动技能需求。

### 第一批建议实现的原语

#### 1. `SpawnProjectileSkill`

适合：

- 箭雨
- 火球
- 冰锥
- 飞刀
- 回旋镖
- 多重火球

#### 2. `AreaDamageSkill`

适合：

- 冰环
- 雷暴
- 毒云
- 爆炸
- 地刺

#### 3. `ApplySelfBuffSkill`

适合：

- 狂暴
- 护盾
- 临时加速
- 自动射击模式

#### 4. `DashAttackSkill`

适合：

- 冲锋斩
- 闪现斩
- 骑射

#### 5. `SummonSkill`

适合：

- 召唤狼
- 召唤炮台
- 召唤骷髅
- 召唤分身

### 完成定义

- 新技能优先基于原语参数化
- 不再默认“每个技能复制一个大类”

### 验证手段

- 用 2-3 个不同技能验证同一原语的复用能力

### 测试用例

#### 用例 1：`SpawnProjectileSkill` 复用

- 技能 A：箭雨
- 技能 B：火球
- 预期：共享一个 skillClass，只是参数不同

#### 用例 2：`ApplySelfBuffSkill` 复用

- 技能 A：狂暴
- 技能 B：护盾
- 预期：通过不同参数 / payload 生效

#### 用例 3：`SummonSkill` 复用

- 技能 A：召唤狼
- 技能 B：召唤炮台
- 预期：调用统一召唤入口

## 七、Step 3.4 - 迁移现有技能

### 目标

先把已有技能迁到新结构，避免新旧两套长期并存。

### 第一批迁移对象

- `ArrowStormSkill`
- `DashShotSkill`

### 迁移策略

#### `ArrowStormSkill`

目标迁入：

- `SpawnProjectileSkill`
- 或保留为复杂技能，但底层投射物生成改走统一攻击请求

#### `DashShotSkill`

目标迁入：

- `ApplySelfBuffSkill + behavior command`
- 或 `DashAttackSkill`

### 完成定义

- 现有两个技能不再直接长期依赖脏上下文
- 至少一部分逻辑迁移到原语或统一 service

### 验证手段

- 对比迁移前后技能效果
- CD、数量、持续时间都必须一致

### 测试用例

#### 用例 1：箭雨回归

- 操作：释放箭雨
- 预期：箭数量、落点、命中行为一致

#### 用例 2：闪身射击回归

- 操作：释放闪身射击
- 预期：临时增益、自动攻击、恢复逻辑一致

## 八、Step 3.5 - 定义统一攻击载荷 `AttackPayloadDef`

### 目标

把“命中后发生什么”从技能和普攻里抽出来。

### 为什么必须有这一层

如果没有统一 payload：

- 普攻自己一套命中逻辑
- 火球自己一套命中逻辑
- 爆炸自己一套命中逻辑
- 召唤物又自己一套命中逻辑

那迟早烂。

### `AttackPayloadDef` 建议至少包含

- `id`
- `damagePolicy`
- `damageRatio`
- `hitEffects`
- `targetBuffs`
- `attackerBuffs`
- `tags`

### 设计目标

- 普攻可引用 payload
- 技能可引用 payload
- 召唤物未来也可引用 payload

### 完成定义

- “技能做什么”和“命中后发生什么”彻底拆开

### 验证手段

- 让两个不同攻击入口复用同一个 payload

### 测试用例

#### 用例 1：普攻与技能共用 payload

- 场景：火焰箭普通攻击和火球技能都引用灼烧 payload
- 预期：命中附加行为一致

#### 用例 2：仅替换 payload 不改技能类

- 场景：同一 projectile 技能换成冰冻 payload
- 预期：技能入口不变，命中效果变了

## 九、Step 3.6 - 建立统一攻击执行器

### 目标

让技能不再直接 new 投射物、自己算一整套攻击落地逻辑。

### 建议新增的执行层

- `AttackExecutor`
- `ProjectileSpawner`
- `AreaExecutor`
- `MeleeExecutor`
- `SummonExecutor`

### 技能层只负责什么

- 决定释放时机
- 决定释放目标
- 生成 `AttackSpec`

### 执行层负责什么

- 根据 `attackType` 执行
- 生成 projectile / area / melee / summon
- 绑定 payload

### 完成定义

- 技能脚本不再直接依赖底层投射物实现

### 验证手段

- 选两个技能接入执行层
- 检查它们是否仍能正常命中和触发效果

### 测试用例

#### 用例 1：投射物技能走执行器

- 技能：箭雨或火球
- 预期：攻击可通过 executor 生成并命中

#### 用例 2：范围技能走执行器

- 技能：冰环 / 爆炸
- 预期：无需投射物也可共用统一执行框架

#### 用例 3：召唤技能走执行器

- 技能：召唤狼
- 预期：执行器负责创建实体，技能只发请求

## 十、Step 3.7 - 收口 `combat/` 目录

### 目标

把当前被切碎的战斗流水线在目录语义上收回来。

### 当前脏点

同一条攻击链目前散落在：

- `shoot/`
- `projectile/`
- `hitEffects/`
- `entity/`
- `skill/`

这会导致后续做火球、召唤、近战时，不知道该往哪个目录继续塞。

### 正确做法

后续逻辑上统一按 `combat/` 组织：

- `combat/attack`
- `combat/projectile`
- `combat/hit`
- `combat/runtime`

短期不要求一次性全量迁移，但至少要完成：

1. 文档口径统一
2. 新增战斗代码按 `combat` 语义归类
3. 旧目录后续只减不增

### 完成定义

- 后续新增火球 / 爆炸 / 召唤物相关代码，不再继续横向散落

### 验证手段

- 审核新增文件放置位置
- 检查攻击链相关依赖是否逐步向 `combat` 收拢

### 测试用例

#### 用例 1：投射物攻击目录归类正确

- 场景：新增一个火球类主动技能
- 预期：投射物与命中逻辑归在 `combat` 语义下，而不是散落多个横向目录

#### 用例 2：范围攻击目录归类正确

- 场景：新增一个爆炸 / 冰环技能
- 预期：攻击执行仍归在 `combat` 语义层

## 十一、Step 3.8 - 建立中立战斗接口

### 目标

让战斗链路不再只服务“玩家打敌人”这一种关系。

### 当前脏点

现在很多攻击逻辑天然假设：

- 攻击方是玩家
- 目标是敌人
- 投射物命中回调里拿到的是 `EnemyControl`

这会卡死：

- 召唤物
- 敌人投射物
- 友军单位
- 未来多人或镜像单位

### 正确做法

逐步引入中立接口：

- `IAttackSource`
- `IAttackTarget`
- `ITargetable`
- `IDamageable`

### 设计要求

- `ShootResolver`、攻击执行器、payload 不再直接绑定 `PlayerProperty + EnemyControl`
- 技能和弹道逻辑面向接口，不面向具体 actor

### 完成定义

- 玩家、敌人、召唤物可以逐步接到同一战斗接口上

### 验证手段

- 列出 `EnemyControl` / `PlayerControl` 在攻击链中的直依赖点
- 为后续替换建立适配层

### 测试用例

#### 用例 1：敌方投射物接口预留

- 场景：敌方箭矢命中目标
- 预期：攻击链逻辑可以逐步面向中立目标接口

#### 用例 2：召唤物攻击接口预留

- 场景：召唤狼攻击敌人
- 预期：不需要复制一套完全独立的命中体系

## 十三、Step 3.10 - 全链路系统验收

### 目标

### 重点回归内容

- 技能 CD
- 技能释放

- 投射物攻击

- 未来近战 / 召唤攻击接口

#### 载荷层

- 命中伤害
- 命中附加 Buff
- 命中效果组合

### 统一手工测试用例

- 从 JSON 装备技能

#### 用例 2：同原语不同参数回归

- 两个不同 projectile 技能

- 技能入口不变，只替换 payload

- 通过升级增强技能参数或 payload
- 预期：技能表现正确变化

## 十四、本大步骤完成后的统一验收

完成 `重构步骤 3` 后，应该达到下面这些标准：

| 验收项 | 预期结果 |
|------|---------|
| `SkillContext` 是否还带弓箭手私货 | 否 |
| 新增技能是否还默认复制旧技能类 | 否 |
| 技能是否可通过 JSON + 工厂创建 | 是 |
| 普攻 / 技能 / 召唤物是否具备统一攻击载荷方向 | 是 |
| 新增未来技能是否能优先归到原语层 | 是 |
| 战斗目录是否仍横向继续发散 | 否 |
| 攻击链是否仍硬绑定“玩家打敌人” | 否 |

## 十五、建议的手工回归清单

每次完成一个小步骤，建议固定跑这 10 条：

1. 技能装备正常
2. 技能卸下正常
3. 技能 CD 正常
4. 技能释放正常
5. 投射物技能正常
6. 范围技能正常
7. 自 Buff 技能正常
8. payload 命中效果正常
9. 升级增强技能正常
10. 普攻未被重构误伤
11. 战斗目录不继续新增横向散落代码
12. 敌方 / 召唤物接口预留不报设计冲突

## 十六、这一阶段不要做什么

不要在这一步里顺手做：

- 所有技能一次性完全 JSON 化
- 所有命中效果完全 DSL 化
- 立刻重写所有投射物类
- 把复杂 Boss 技能一起塞进来

这些事情都不是当前重点。

这一步只做一件事：

**把技能从零散脚本集合，变成“原语 + 工厂 + 攻击载荷”的长期扩展架构。**
