# 升级系统教程

> 基于 Ecslearn 项目 `game/upgrade` 模块，适用于 Cocos Creator 3.x + TypeScript。

---

## 一、概述

### 1.1 升级系统定位

肉鸽游戏的核心循环：**击杀 → 升级 → 三选一 → 永久增强**。升级系统管理的是这些**被动选择**——选一次就永久生效，直到本局结束。

```
升级系统（被动永久）→ UpgradeManager → Buff / HitEffect / ShootPolicy
技能系统（主动释放）→ SkillSystem    → IActiveSkill.execute()
```

### 1.2 设计理念

| 原则 | 说明 |
|------|------|
| **配置驱动** | 新升级 = 1 个 `UpgradeConfig` 对象，核心代码零改动 |
| **三种效果类型** | `buff` / `hit_effect` / `shoot_policy` 覆盖所有被动场景 |
| **完整回退** | `remove()` 时自动回滚所有 Buff、命中效果、射击策略 |
| **进化组合** | `evolvesFrom` 定义前置条件，自动检测可触发的进化 |

### 1.3 整体架构

```
UpgradeConfig（配置）
    ↓
UpgradeManager.apply(config)
    ↓ effects[]
    ├── type: 'buff'         → buffMgr.addBuff(data)
    ├── type: 'hit_effect'   → hitEffectMgr.add(data)
    └── type: 'shoot_policy' → setShootPolicy(create(data))
    ↓
AppliedRecord（记录 buffIds / hitEffectIds / wasPolicy）
    ↓
UpgradeManager.remove(id)
    ├── buffMgr.removeBuff(buffId) × N
    ├── hitEffectMgr.remove(hid) × N
    └── setShootPolicy(defaultPolicy)
```

## 二、核心类型

### 2.1 UpgradeEffect

```typescript
export interface UpgradeEffect {
    type: 'buff' | 'hit_effect' | 'shoot_policy';
    data: any;
}
```

一个升级可以包含**多个** effect（如「灼热穿刺」同时加穿透 Buff + 灼烧命中效果）。

### 2.2 UpgradeConfig

```typescript
export interface UpgradeConfig {
    id: string;                                     // 唯一标识
    name: string;                                   // 显示名
    desc: string;                                   // 描述文本
    tier: number;                                   // 阶级（1/2/3）
    rarity: 'common' | 'rare' | 'epic';            // 稀有度
    category: 'attr' | 'proj' | 'onhit' | 'policy';// 分类
    effects: UpgradeEffect[];                       // 效果列表
    evolvesFrom?: string[];                         // 进化前置条件
}
```

### 2.3 UpgradeTarget

```typescript
export interface UpgradeTarget {
    buffMgr: EntityBuffMgr;
    buffOwner: IBuffOwner;
    hitEffectMgr: HitEffectMgr;
    setShootPolicy(policy: IShootPolicy): void;
}
```

`PlayerControl` 构造 `UpgradeManager` 时传入此接口，升级系统通过它与玩家各子系统交互。

## 三、UpgradeManager —— 升级管理器

### 3.1 核心方法

```typescript
export class UpgradeManager {
    private _applied = new Map<string, AppliedRecord>();
    private _target: UpgradeTarget;

    constructor(target: UpgradeTarget);

    apply(config: UpgradeConfig): boolean;    // 应用升级
    remove(configId: string): boolean;        // 移除升级
    has(configId: string): boolean;           // 是否已应用
    get appliedIds(): string[];               // 已应用列表
    checkEvolution(allConfigs): UpgradeConfig[]; // 检查可触发进化
}
```

### 3.2 apply 流程

```typescript
apply(config: UpgradeConfig): boolean {
    if (this._applied.has(config.id)) return false;  // 不可重复

    const record: AppliedRecord = {
        config, buffIds: [], hitEffectIds: [], wasPolicy: false,
    };

    for (const eff of config.effects) {
        this._applyEffect(eff, record);
    }

    this._applied.set(config.id, record);
    return true;
}
```

### 3.3 效果分发（唯一 switch）

```typescript
private _applyEffect(eff: UpgradeEffect, record: AppliedRecord) {
    switch (eff.type) {
        case 'buff':
            this._target.buffMgr.addBuff(eff.data, this._target.buffOwner);
            record.buffIds.push(eff.data.id);
            break;
        case 'hit_effect':
            this._target.hitEffectMgr.add(eff.data);
            record.hitEffectIds.push(eff.data.id);
            break;
        case 'shoot_policy':
            const policy = this._createPolicy(eff.data);
            this._target.setShootPolicy(policy);
            record.wasPolicy = true;
            break;
    }
}
```

这是**整个项目里唯一一处** `switch(eff.type)`——只有 3 个 case，可控且清晰。

### 3.4 remove 流程

```typescript
remove(configId: string): boolean {
    const record = this._applied.get(configId);
    if (!record) return false;

    for (const buffId of record.buffIds) {
        this._target.buffMgr.removeBuff(buffId);
    }
    for (const hid of record.hitEffectIds) {
        this._target.hitEffectMgr.remove(hid);
    }
    if (record.wasPolicy) {
        this._target.setShootPolicy(this._defaultPolicy);
    }
    this._applied.delete(configId);
    return true;
}
```

**完整回退**：记录了每个升级产生的所有 buffId / hitEffectId / 是否改了策略，移除时逐一清理。

## 四、升级配置实战

### 4.1 Category A：属性强化（buff 类型）

```typescript
const rapidFire: UpgradeConfig = {
    id: 'rapid-fire',
    name: '急速弓弦',
    desc: '攻速+30%',
    tier: 1, rarity: 'common', category: 'attr',
    effects: [{
        type: 'buff',
        data: {
            id: 7001, name: '急速弓弦', duration: 0, maxStack: 1,
            effectClass: 'SimpleAttrBuffEffect',
            targetAttr: 'AttackSpeed-Mul-Buff',
```

## 五、进化系统

### 5.1 什么是进化

当玩家同时拥有多个特定升级时，可以触发更强的**进化升级**。

### 5.2 配置方式

```typescript
const scorchingPierce: UpgradeConfig = {
    id: 'scorching-pierce',
    name: '灼热穿刺',
    desc: '穿透+灼烧 → 火线穿刺',
    tier: 2, rarity: 'epic', category: 'onhit',
    evolvesFrom: ['fire-arrow', 'pierce-shot'],  // 前置条件
    effects: [
        {
            type: 'buff',
            data: { id: 7201, name: '灼热穿刺·穿透', duration: 0,
                    effectClass: 'SimpleAttrBuffEffect',
                    targetAttr: 'PierceCount-Value-Buff', valuePerStack: 2 },
        },
        {
            type: 'hit_effect',
            data: { id: 'upg-scorching-burn', effectClass: 'BurnOnHitEffect',
                    priority: 45, burnRatio: 0.4, burnBuffId: 8003, burnDuration: 5 },
        },
    ],
};
```

### 5.3 检测逻辑

```typescript
checkEvolution(allConfigs: readonly UpgradeConfig[]): UpgradeConfig[] {
    const result: UpgradeConfig[] = [];
    for (const cfg of allConfigs) {
        if (!cfg.evolvesFrom?.length) continue;     // 非进化类跳过
        if (this._applied.has(cfg.id)) continue;     // 已触发跳过
        if (cfg.evolvesFrom.every(id => this._applied.has(id))) {
            result.push(cfg);                        // 前置全满足 → 可进化
        }
    }
    return result;
}
```

**使用时机**：每次 `apply` 后调用 `checkEvolution(ALL_UPGRADES)`，将结果加入三选一候选池。

## 七、新建升级完整指南

### 7.1 步骤清单

1. 在 `game/upgrade/upgradeConfigs.ts` 中新增 `UpgradeConfig` 对象
2. 将其加入 `ALL_UPGRADES` 数组
3. 如果效果需要新的 Buff 效果类 → 在 `skill/effects/` 下新建并注册
4. 如果效果需要新的命中效果类 → 在 `hitEffects/` 下新建并注册

### 7.2 各类型配置模板

**属性 Buff 类**：
```typescript

    id: 'my-upgrade', name: '升级名', desc: '描述',
    tier: 1, rarity: 'common', category: 'attr',

        type: 'buff',
        data: {
            id: 7xxx, name: '升级名', duration: 0,
            effectClass: 'SimpleAttrBuffEffect',
            targetAttr: 'XxxAttr-Value-Buff',     // 或 Xxx-Mul-Buff
```

**命中效果类**：

## 八、与其他系统的关系

```
                    ┌── Buff 系统（EntityBuffMgr）
                    │   └── 属性系统（PlayerProperty → Modifier）
升级系统 ──────────→├── 命中效果系统（HitEffectMgr）
（UpgradeManager）  │   └── 弹道系统（ArrowProjectile → onHit）
                    └── 射击系统（IShootPolicy）
                        └── 输入系统（ActionComp）
```

升级系统是**分发中心**——它本身不实现任何战斗逻辑，只是把配置数据分发到 Buff / 命中效果 / 射击策略三个子系统。

## 十、常见问题

### Q1：升级能叠加吗？

不能。`apply()` 检查 `_applied.has(config.id)` → 同 id 不可重复应用。如果要叠加效果，应在 Buff 层面设置 `maxStack > 1`。

### Q2：移除升级后属性没恢复？

检查 `AppliedRecord.buffIds` 是否正确记录了所有 Buff ID。如果一个升级的 effect 中 `data.id` 缺失或重复，`removeBuff` 会失败。

### Q3：进化升级需要消耗前置吗？

当前实现**不消耗**——前置升级保留，进化额外叠加。如需消耗，在 `apply` 进化后对 `evolvesFrom` 中的 id 调用 `remove()`。

### Q4：如何实现三选一 UI？

1. 从 `ALL_UPGRADES` 中过滤出未 apply 的、满足 tier 条件的候选
2. 调 `checkEvolution(ALL_UPGRADES)` 加入进化候选
3. 按稀有度加权随机抽取 3 个
4. 玩家选择后调 `upgradeManager.apply(chosen)`
