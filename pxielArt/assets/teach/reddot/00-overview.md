# 红点系统（Red Point System）— 总览

> 目标读者：**完全没做过红点系统的新手**。
> 目标结果：**做完六个阶段，你会拥有一个工业级红点系统**，可以直接搬去任意 Cocos / Unity 项目。

---

## 为什么要做这个系统？

你在游戏里一定见过：

- 关卡选择页面 → 没点过的关卡角上有个**小红点**
- 首页「关卡」按钮 → 只要下面任何一个关卡有红点，按钮就亮
- 邮件图标 → **红色数字 "5"** 表示 5 封未读
- 活动中心 → 有奖可领就亮，领完就灭

这些看起来是**四个功能**，其实是**同一个系统**。它们都在回答一个问题：

> "这里有新东西，你还没看过 / 还没处理。"

如果每个按钮都各写一套，项目跑一年就会烂。所以需要一个**统一的红点系统**。

---

## 好的红点系统应该具备什么能力？

| # | 能力 | 解决什么问题 |
|---|------|-------------|
| 1 | 挂在任意 UI 节点上，显示/隐藏 | 最基础的 UI |
| 2 | **树形结构**，子红点向上冒泡 | 父按钮汇总子页面 |
| 3 | **数字显示**（1、5、99+） | 未读数、待领数 |
| 4 | **声明式注册**：在一处集中定义所有红点 | 不让红点逻辑散落各处 |
| 5 | **事件驱动**：业务发事件，红点自己刷新 | 业务代码不关心红点存在 |
| 6 | **动态节点**：运行时注册/注销（如"邮件 #42"） | 未知数量的红点 |
| 7 | **性能优化**：一帧内合并多次刷新 | 上千节点也不卡 |

## 进阶篇：另一种设计哲学（可选）

做完 Stage 1~6，你已经拥有一套工业级红点系统——**路径 id + 全局树 + Registry + EventBus + 脏标记**（下文简称**方案 A**）。

**但这不是唯一的好答案。**

还有一种设计（简称**方案 B**）走完全不同的路：

- 没有全局树，没有路径 id，没有中央 Manager
- 每个红点 = 一个**TypeScript 类**（`implements IRed`）
- 类之间用 `RedGroup` 组合，关系**显式、局部、可复用**
- 用 `Signal<T>` 对象代替字符串事件名（**信号跟着数据走**）
- UI 接入：**只挂一个组件 + 填一个类名**，其他全自动

方案 B 在"开发者新增一个红点要做多少事"上比方案 A 更激进地简化。
但它也有代价（学习曲线更抽象、装饰器依赖等）。两种方案**各有适用场景**，不是替代关系。

| 阶段 | 标题 | 你会得到什么 | 对应文档 |

## 设计原则（贯穿全系列）

### 1. 数据结构优先于代码

> "Bad programmers worry about the code. Good programmers worry about data structures." — Linus

每个阶段开头，我都会先画**数据结构图**。数据对了，代码只是把它翻译成 TS。

### 2. 存事实，不存派生状态

**"有没有红点"**不是事实，**"玩家有没有点过 X 关卡"**才是事实。
存事实，红点数现算。这样业务改需求，红点不用改。

```
❌ 差：存 hasRedDot: boolean
✅ 好：存 seenLevels: Set<string>，红点 = !seenLevels.has(id)
```

### 3. 渲染与数据解耦

红点系统**只产出数字**（`count: number`）。
UI 怎么显示（圆点/方块/动画/颜色）是 UI 的事，红点系统不管。

### 4. 向后兼容，不破坏现有功能

每个阶段的代码**不会删除前面的代码**，只会**增加**或**升级 API**。
你在任何阶段停下来，都能得到一个能跑的红点系统。

### 5. 先跑通，再优化

前 4 个阶段追求**正确**，第 5 阶段才追求**性能**。
不要一上来就脏标记+批量刷新，你连树还没走顺。

## 每个阶段的统一结构

为了让你读起来不累，每篇文档都是这个骨架：

1. **这一阶段要解决什么问题** —— 场景驱动，不空谈
2. **Linus 式思考** —— 数据结构 / 特殊情况 / 复杂度三连问
3. **设计方案** —— 图 + API 形态
4. **完整代码** —— 可复制、可运行
5. **怎么用（示例）** —— 最少 1 个 demo
6. **验证** —— 怎么确认这一阶段做对了
7. **遗留问题 → 下一阶段解决什么**
