## 本节目标（延后实现）

**让棋盘永远居中、在任何分辨率下都不会画到屏幕外**。

预期：

- 默认窗口下，3×3 点阵 + 3 根箭头仍然在屏幕中央。
- 把浏览器窗口缩小到 400×600，棋盘整体缩小，**全部内容仍可见**。
- 放大到 1600×1200，棋盘也适度放大，不会只占屏幕一小块。
- 浏览器改窗口大小时，棋盘动态重排（可选，本章实现）。

一句话：**让画面尺寸根据屏幕自动变化**。

---

## 需求分析

为什么要做窗口适配？

- **同一份关卡数据要在不同设备上玩**。iPhone SE 和 iPad 屏幕大小差 4 倍，代码里硬编 `gap=100` 就死了。
- **关卡大小不一样**：首关 5×5，第 20 关可能 20×30。不缩放，后面的关就溢出屏幕。
- **对照参考项目**：G3_FBase 的 `WindowSizeLogic.onExecute()` 每次窗口变化都重算一套 `scale.min/max/def`，算得很复杂是因为它支持拖拽缩放。我们**只做最小版本**：根据关卡大小和屏幕大小算一个"让棋盘刚好装进屏幕 + 留边距"的 scale。

---

## 实现思路

### 计算棋盘尺寸

棋盘本身占多少像素？

```
boardWidth  = cols * gap
boardHeight = rows * gap
```

注意这是"格子中心的跨度"，不包括两边的半格留白。G3_FBase 里是 `(cols + 1) * gap`——多留一个 gap 作为外围。**我们简化为 `cols * gap`**，足够了。

### 计算 scale

希望棋盘**加 20% 边距后刚好装进屏幕**：

```
scaleX = screenWidth  * 0.8 / boardWidth
scaleY = screenHeight * 0.8 / boardHeight
scale  = min(scaleX, scaleY)
```

取 `min` 是为了两个方向都装得下。

还要**限制最大 scale**，避免小棋盘在大屏幕上放得离谱大：

```
scale = min(scale, 1.5)   // 最多放大到 1.5 倍
```

### 应用 scale

所有 `BoardView` 子节点（点 + 箭头）都挂在 BoardView 节点下，**给 BoardView 节点设 `scale` 就好**。子节点跟着一起缩放。

```typescript
boardView.node.setScale(scale, scale, 1);
```

> **Linus 的品味**：缩放这种事情**设在根节点上一次搞定**，不要去每个点、每个箭头单独算。一条路径，错不了。

## 代码实现

### 文件 1：扩展 `core/Coord.ts`（加一个布局计算函数）

在 `Coord.ts` 末尾追加：

```typescript
import { Config } from '../common/Config';

export interface BoardLayout {
    /** 棋盘原始宽（像素，未 scale） */
    boardWidth: number;
    /** 棋盘原始高 */
    boardHeight: number;
    /** 建议的缩放比例 */
    scale: number;
}

/**
 * 根据关卡尺寸和屏幕尺寸，算出棋盘的原始宽高和建议缩放
 */
export function computeBoardLayout(
    rows: number, cols: number, screenW: number, screenH: number,
): BoardLayout {
    const boardWidth = cols * Config.gap;
    const boardHeight = rows * Config.gap;
    const scaleX = (screenW * 0.8) / boardWidth;
    const scaleY = (screenH * 0.8) / boardHeight;
    let scale = Math.min(scaleX, scaleY);
    if (scale > 1.5) scale = 1.5;
    if (scale < 0.2) scale = 0.2;
    return { boardWidth, boardHeight, scale };
}
```

**关键点**：

- **纯函数**，不依赖 Cocos，可单测。对应 G3_FBase 的 `WindowSizeLogic` 中那一堆计算，只不过我们只保留"算一个 scale"这一件事。
- **scale 上下限**：`0.2 ~ 1.5`。下限防止小屏幕上棋盘变得看不清，上限防止大屏幕上画面太空。数值可以调。

### 文件 2：改造 `GameController.ts`（应用布局 + 监听窗口变化）

```typescript
import {
    _decorator, Component, resources, JsonAsset, Node,
    view, screen,
} from 'cc';
import { LevelData, validateLevelData } from '../core/LevelData';
import { computeBoardLayout } from '../core/Coord';
import { BoardView } from './BoardView';
const { ccclass } = _decorator;

@ccclass('GameController')
export class GameController extends Component {
    private levelData: LevelData | null = null;
    private boardView: BoardView | null = null;

    onLoad() {
        console.log('[Arrow] Game scene loaded. I am alive.');
        this.boardView = this.createBoardView();
        this.loadLevel(1);

        // 监听窗口变化
        view.on('canvas-resize', this.onCanvasResize, this);
    }

    onDestroy() {
        view.off('canvas-resize', this.onCanvasResize, this);
    }

    private createBoardView(): BoardView {
        const node = new Node('BoardView');
        this.node.addChild(node);
        return node.addComponent(BoardView);
    }

    private loadLevel(levelNo: number) {
        const no = levelNo < 10 ? `0${levelNo}` : `${levelNo}`;
        const path = `levels/level_${no}`;
        resources.load(path, JsonAsset, (err, asset) => {
            if (err) {
                console.error(`[Arrow] Load level failed: ${path}`, err);
                return;
            }
            try {
                this.levelData = validateLevelData(asset.json);
                this.onLevelLoaded(this.levelData);
            } catch (e) {
                console.error(`[Arrow] Level data invalid:`, e);
            }
        });
    }

    private onLevelLoaded(data: LevelData) {
        console.log(`[Arrow] Level loaded: ${data.rows} x ${data.cols}, arrows = ${data.arrows.length}`);
        this.boardView?.render(data);
        this.applyLayout();
    }

    private applyLayout() {
        if (!this.levelData || !this.boardView) return;
        const size = screen.windowSize;
        const layout = computeBoardLayout(
            this.levelData.rows, this.levelData.cols, size.width, size.height
        );
        this.boardView.node.setScale(layout.scale, layout.scale, 1);
        console.log(
            `[Arrow] Layout applied: screen(${size.width.toFixed(0)}x${size.height.toFixed(0)}), ` +
            `board(${layout.boardWidth}x${layout.boardHeight}), scale=${layout.scale.toFixed(3)}`
        );
    }

    private onCanvasResize() {
        this.applyLayout();
    }
}
```

**关键点**：

- `**screen.windowSize`**：Cocos 3.8 拿窗口像素尺寸的标准 API。不要用已经废弃的 `cc.view.getVisibleSize()`。
- `**view.on('canvas-resize', ...)**`：Cocos 3.8 的窗口变化事件。浏览器窗口拖拽改大小时会触发。
- `**onDestroy` 里 `off**`：**监听必有解绑**。不解绑 = 场景切换后旧回调仍在 = 内存泄漏 + 可能访问已销毁的节点崩溃。这是新手极其容易忘的事。

## 易错点

### 易错 1：用废弃 API

```typescript
cc.view.getVisibleSize()     // ❌ Cocos 3.x 没有 cc 命名空间了
```

Cocos 3.x 改成模块化：`import { view, screen } from 'cc';`。遇到网上旧教程用 `cc.xxx`，**全部要替换**，不能想当然。

### 易错 2：scale 设在 GameController 节点上

```typescript
this.node.setScale(layout.scale, ...);   // ❌ this.node 是 Canvas
```

**Canvas 自带 Widget 铺满屏幕的逻辑**，给它设 scale 会造成视觉错乱（UI 元素的布局依赖 Canvas 的变换）。

**规则**：要缩放的是"棋盘容器"（BoardView 节点），不是 Canvas。

### 易错 3：没在 `onDestroy` 里解绑

```typescript
onLoad() {
    view.on('canvas-resize', this.onCanvasResize, this);
}
// 没写 onDestroy
```

组件销毁后回调还在 → 下次触发时 `this.boardView` 可能已经是 null → 崩。

**规则**：**所有 `on` 必须配对 `off`**。模板化写在组件里，不要偷懒。

### 易错 4：`computeBoardLayout` 的边距算错

```typescript
const scaleX = screenW * 1.0 / boardWidth;   // ❌ 没留边距
```

棋盘刚好撑满屏幕 → 边缘箭头头（那个三角形尖端）会被裁掉一半。`*** 0.8` 是留 20% 边距的习惯做法**。

## 扩展练习

1. **加一个大棋盘**：写 `level_big.json` 使用 20×20。把 `loadLevel(1)` 改成 `loadLevel('big')`（要你自己调整文件名逻辑），验证棋盘能正常显示。
2. **监听分辨率变化的正确手势**：尝试把 `canvas-resize` 改成 `resize`（错写），观察会不会报错或不生效。学会查 Cocos 3.8 `view` 类的合法事件名。
3. **思考题**：`screen.windowSize` 和 `view.getDesignResolutionSize()` 的区别是什么？你为什么选前者？提示：设计分辨率是"逻辑坐标"，窗口尺寸是"物理像素"。
