# 21 · 扩展：给新人的 5 个练习方向

## 本节目标

到这一章，你已经从 0 做出一个能玩 5 关的 Arrow Escape 游戏，包含：

- 状态机驱动的箭头行为（5 态）
- 触摸交互（点击 + 长按 Ray）
- 胜负判定 + 重试
- 首页 + 存档 + 关卡切换

**这一章不再给新代码**。给你 5 个练习方向，每个方向都可以从参考项目 `G3_FBase/G3_FHome` 里找到对应的工业级实现。

**建议**：**挑一个感兴趣的做**，不要五个都做（会累垮）。做完一个再挑下一个。每个练习预计工作量 2~4 小时。

---

## 练习方向一：皮肤系统

### 目标

让玩家能在 Home 页切换皮肤，不同皮肤改变箭头/点阵/背景的配色。

### 参考来源

G3_FBase / G3_FHome 的：

- `G3_FBase_SkinConfig.ts`
- `G3_FBase_GetSchemeValueFunction.ts`
- `G3_FHome_SkinInitLogic.ts`
- `G3_FHome_SkinSwitchLogic.ts`

它们用 "Scheme" 概念：每个皮肤是一张颜色表（`arrowIdleColor / arrowMoveColor / rayIdleColor / ...`）。

### 实现思路

1. 在 `common/` 下新建 `Skin.ts`：
   ```typescript
   export interface Skin {
       arrowIdle: number;
       arrowMove: number;
       arrowStop: number;
       pointColor: number;
       background: number;
   }
   export const SKINS: Record<string, Skin> = {
       default: { arrowIdle: 0x111633, ... },
       dark:    { arrowIdle: 0xffffff, ... },
       neon:    { arrowIdle: 0x00ffcc, ... },
   };
   ```

2. `GameData` 加字段 `currentSkin: string`，持久化。

3. `ArrowView.pickColor()` 读 `GameData.currentSkin` 选颜色。

4. Home 场景加 3 个小色块按钮，点击切换皮肤。

### 难点

- 实时刷新：切换皮肤后 Game 场景里所有 ArrowView/RayView/BoardView 都要 refresh 一遍。
- 皮肤配置要支持**缺字段回落**（老存档的新皮肤字段不存在）。

## 练习方向二：Hint 提示系统

### 目标

玩家卡关时点击"Hint"按钮，屏幕上闪烁提示"下一步应该发射的箭头"。

### 参考来源

- `G3_FBase_RewardAdResultHintLogic.ts`
- `G3_FBase_RewardAdResultHintArrowLogic.ts`

G3_FBase 里 Hint 和看广告绑定（看完广告给 Hint）。**我们简化为"每关 3 次免费 Hint"**。

### 实现思路

1. 写一个算法找"下一步该发射哪一根"：
   - 遍历所有 Idle 的箭头
   - 对每一根模拟发射（不改真实 runtime，clone 一份）
   - 如果发射后 findCollision 对剩余 Idle 箭头都有解（递归/DFS）→ 这根是安全选择
   - 返回第一个找到的安全索引

2. Hint 按钮挂 HUD 或 Overlay。点击触发算法 → 对应 ArrowView 闪烁 1 秒。

3. 计数：`GameData.hintCount` 持久化。用完提示"广告解锁更多"（教程不实现广告）。

### 难点

- 算法有回溯，可能很慢（5 根箭头有 5! = 120 种顺序）。但 5 根范围内没问题。
- 不要提供 100% 正确答案，要避免算法把游戏变成走迷宫。设计 Hint 时给"多个可行选择中的一个"就够。

## 练习方向三：音效

### 目标

关键事件触发音效：

- 箭头发射 → "whoosh" 声
- 碰撞 → "clang" 声
- 胜利 → "win" 声
- 失败 → "lose" 声

### 参考来源

- `G3_FBase_AudioClickLogic.ts`
- `G3_FBase_AudioDamagedLogic.ts`
- `G3_FBase_AudioEscapeLogic.ts`
- `G3_FBase_AudioWinLogic.ts`

G3_FBase 里每种音效一个 Logic 响应对应信号。**我们合并成一个 AudioManager**。

### 实现思路

1. 在 `assets/resources/audios/` 放 4 个 `.ogg` 音效文件（参考项目有现成的 `a_click.ogg / a_damaged.ogg / a_dead.ogg / a_escape.ogg / a_win.ogg`，可以直接 copy）。

2. `common/AudioManager.ts`：
   ```typescript
   import { AudioClip, AudioSource, resources } from 'cc';
   
   class _AudioManager {
       private source: AudioSource | null = null;
       private clips = new Map<string, AudioClip>();
   
       play(name: string) {
           const clip = this.clips.get(name);
           if (!clip) { this.loadAndPlay(name); return; }
           this.source?.playOneShot(clip);
       }
       
       private loadAndPlay(name: string) {
           resources.load(`audios/${name}`, AudioClip, (err, clip) => {
               if (err) return;
               this.clips.set(name, clip);
               this.source?.playOneShot(clip);
           });
       }
   }
   export const AudioManager = new _AudioManager();
   ```

3. GameController 在关键事件处 `AudioManager.play('click')` 等。

### 难点

- AudioSource 需要挂一个节点上，初始化顺序要对。
- 某些浏览器（Safari）需要用户首次交互后才能播声音（autoplay policy）。

## 练习方向四：新形状关卡

### 目标

扩展关卡配置支持"非方形棋盘"，比如 L 形、T 形。

### 参考来源

- `G3_FBase_Namespace.LevelConfig.shape`
- `src/functions/` 里处理 shape 的函数（G3_FBase 里有 `MaskDanger` 等相关资源）

G3_FBase 用 `shape: string` 指定形状名字，加载一张遮罩图决定哪些格子可用。

### 实现思路

1. LevelData 加字段：
   ```typescript
   interface LevelData {
       ...
       /** 可选：每一格是否可通行的二维矩阵 */
       walkable?: number[][];
   }
   ```

2. `isInsideBoard` 扩展考虑 walkable：
   ```typescript
   export function isWalkable(r, c, data): boolean {
       if (!isInsideBoard(r, c, data.rows, data.cols)) return false;
       if (data.walkable && data.walkable[r-1][c-1] === 0) return false;
       return true;
   }
   ```

3. `BoardView` 渲染时跳过不可用格子不画点。

4. 箭头飞行时把"不可走格子"视为出界，切 End。

### 难点

- 关卡设计需要画图工具或手写 0/1 矩阵，容易出错。写一个关卡编辑器是个更大的副项目。
- 和"箭头占格判断"的语义冲突：unwalkable 格子不能被 coords 包含（关卡不合法）。validateLevelData 要加这层检查。

## 练习方向五：存档重置与统计

### 目标

给玩家三个数据展示：

- 总通关次数
- 本关失败次数
- 全游戏累计游玩时长

加一个"清档"入口（隐藏菜单，比如 Home 左上角按 3 次）。

### 参考来源

- `G3_FHome_FirstUiStartLogic.defineStorageData()`：`startPlayedCount / setupCount`
- `G3_FBase_CombatInitLogic.defineStorageData()`：`restartCount / reviveCount / hintCount`

### 实现思路

1. GameData 扩展：
   ```typescript
   interface GameDataSchema {
       level: number;
       totalPassed: number;       // 累计通关数
       totalFailed: number;       // 累计失败数
       totalPlayTimeSec: number;  // 累计时长（秒）
   }
   ```

2. GameController 里加累计逻辑：
   - 每帧 `GameData.totalPlayTimeSec += dt`（节流每秒存一次避免 IO 爆炸）。
   - triggerWin 时 `totalPassed += 1`；checkFail 时 `totalFailed += 1`。

3. Home 页加一块小字展示统计。

4. "清档"触发：
   ```typescript
   let tapCount = 0;
   cornerNode.on(TOUCH_END, () => {
       tapCount++;
       if (tapCount >= 3) { GameData.reset(); tapCount = 0; }
   });
   ```

### 难点

- 存档频率：每帧都写 localStorage 会卡顿。要做节流（每 5 秒一次，或切场景时一次）。
- 跨场景的时长累计：切到 Home 场景也算时长？本教程认为算，这是总游戏时长。

## 给未来的你的三条建议

1. **读完 G3_FBase 剩下的 100 多个 Logic**。现在你有了基础，能看懂那些 `@ae.atomClass` 做的事背后的实际意图。一个 Logic 一个 Logic 地读，你会发现其中 70% 都是"如果我用 Cocos 3.8 会怎么写"的问题——而你已经写过其中一版了。

2. **不要拿着锤子找钉子**。AtomEngine 4.0 是为它自己生态服务的，不要因为"参考项目这么写"就在 Cocos 3.8 里造轮子。每次想写 "我也做一个 bindAtom / signal / chain" 时先问自己："Cocos 3.8 原生已经有同等效果的功能了吗？"（大概率有。）

3. **Linus 的三个问题永远适用**：
   - 这是个真问题还是臆想出来的？
   - 有更简单的方法吗？
   - 会破坏什么吗？

   写每一个新特性之前、打开 PR 之前、接到需求之前，都问一遍。
