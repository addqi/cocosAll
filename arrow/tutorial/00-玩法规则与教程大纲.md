## 1. 我们在做什么

我们要做的是一款叫 **"Arrow Escape"（箭头逃脱）** 的休闲解谜游戏的 **Cocos Creator 3.8 原生版本**。

参考项目（不是我们的源码，只是玩法参考）：

- `core/features/G3/G3_FBase/` — 战斗核心（箭头 / 点阵 / 射线 / 关卡 / 胜负）
- `core/features/G3/G3_FHome/` — 首页与外围 UI（Level 显示 / Play 按钮 / 底栏）

参考项目使用的是公司自研的 **AtomEngine 4.0 原子化引擎**（运行在 Cocos Creator 2.x 之上）。本教程**不涉及 AtomEngine**，我们直接用 **Cocos Creator 3.8 + TypeScript** 的经典"节点 + 组件 + 事件"模型把同样的玩法复刻一遍。

> 这非常重要：看到参考项目里 `@ae.atomClass`、`ae.LogicAtom`、`defineRender()` 这些东西，**不要抄**，那不是我们要写的东西。
> 教程每一章都会告诉你："在 G3_FBase 里对应的是 XXX 原子，我们这里用 Cocos 3.8 的 YYY 组件/脚本来实现"。

## 2. 核心玩法规则（逆源码梳理）

一句话：**棋盘上有若干根带方向的箭头，玩家逐一点击它们让它们沿方向射出去；如果某根箭头前方被另一根箭头挡住，它会被弹回来；全部成功射出 = 通关，生命值耗尽 = 失败。**

### 2.1 场景构成

| 元素 | 数据来源 | 说明 |
|------|----------|------|
| 网格 | `LevelConfig.rows × cols` | 比如首关 5×5 |
| 点阵（Points） | `arrow.arrows[].coords` | 每根箭头占据的一串格子，每个格子画一个小点 |
| 箭头（Arrows） | `arrow.arrows[]` 每一项 | 一根箭头 = 方向 `direction` + 占据的若干 coords |
| 射线预览（Ray） | 长按某根箭头时出现 | 从箭头末端向方向射出的一条半透明直线 |
| HP（生命值） | `combat.hp`，由 `Config.combatHpMax` 初始化 | 每次发生碰撞 -1（本教程第一版简化为"撞一次就扣 1"） |

以首关 `Level_00001.json` 为例：

```json
{
  "rows": 5, "cols": 5,
  "arrows": [
    { "direction": [0, 1], "origin": [2, 4], "coords": [[2,2], [2,3], [2,4]] },
    { "direction": [0, 1], "origin": [3, 4], "coords": [[3,2], [3,3], [3,4]] },
    { "direction": [0, 1], "origin": [4, 4], "coords": [[4,2], [4,3], [4,4]] }
  ]
}
```

画出来就是：

```
col:    1   2   3   4   5
row 1:
row 2:      ●   ●   ●→
row 3:      ●   ●   ●→
row 4:      ●   ●   ●→
row 5:
```

每根箭头长度是 3 个格子，方向都是 `[0,1]`（正方向的 col 增加，向右）。`origin` 是箭头"头"的位置，`coords` 是从"尾到头"的完整占位。3 根箭头整体占据 `(2..4, 2..4)` —— 棋盘 5×5 的正中 3×3 区域，这样屏幕上看到的点阵会绕中心对称。

### 2.2 核心状态机（G3_FBase 的 `ArrowMoveMode` 枚举）

每根箭头都有一个状态，状态是有限个：

```ts
enum ArrowMoveMode {
    Idle = 0,      // 空闲（还没发射）
    Collide = 1,   // 正在撞击前方的箭头
    Back = 2,      // 被弹回来
    Start = 3,     // 正在向前飞
    End = 4,       // 已飞出边界 = 成功逃脱
}
```

**这是整个游戏最重要的数据结构**。之后几乎所有逻辑都在"读这个状态 + 改这个状态"。

状态转换图（从 G3_FBase 的若干 `MovingXxxLogic` + `TouchEndXxxChain` 总结）：

```
                      点击箭头
                         │
        ┌────────────────▼────────────────┐
        │        Idle (空闲)              │
        │  - 等待玩家点击                  │
        └────┬───────────────────────┬────┘
             │前方无挡                │前方有另一根箭头
             ▼                       ▼
  ┌──────────────────┐     ┌───────────────────┐
  │   Start (飞出)    │     │  Collide (撞击)    │
  │  沿方向匀速移动    │     │  移动到碰撞点      │
  └──────┬───────────┘     └────────┬──────────┘
         │飞出边界                   │撞到被撞箭头头部
         ▼                          ▼
  ┌──────────────┐          ┌────────────────┐
  │  End (胜利)   │          │  Back (回弹)    │
  └──────────────┘          │  往回走到原位    │
                            └───────┬────────┘
                                    │回到起点
                                    ▼
                             ┌──────────────┐
                             │  Idle (空闲)  │← HP - 1
                             └──────────────┘
```

**胜负判定**（源自 `G3_FBase_TouchEndToCombatDoPassDispatchLogic` 和 `G3_FBase_TouchEndToCombatDoFailDispatchLogic`）：

- ✅ **胜利**：所有箭头的 `moveMode >= Start`（即全都已经开始飞出，没有 Idle 的了）
- ❌ **失败**：`combat.hp <= 0`（撞过太多次，生命耗尽）

### 2.3 玩家交互（只讲核心三动作）

| 动作 | 触发 | 结果 |
|------|------|------|
| **点击某根箭头** | TouchStart → TouchEnd 发生在同一根箭头上 | 如果前方无挡：该箭头 `Idle → Start`，开始飞 |
| **长按某根箭头** | 触摸 > 0.3 秒 | 该箭头进入"高亮 + 显示 Ray 射线预览"状态 |
| **松手** | TouchEnd | 高亮消失；如果前方有挡则取消动作，否则执行发射 |

**触摸识别流程**（参考 `G3_FBase_TouchEndToArrowTransformLogic`）：

1. 拿到手指屏幕坐标 → 转本地坐标 → 通过 `findArrowIndex(x, y)` 查哪根箭头被点中
2. 只有 `moveMode === Idle` 的箭头才能被选中
3. 通过 `findCollideTarget(arrowIndex)` 检查前方（沿 direction 方向）是否有其他箭头挡着
4. 无挡 → 派发"开始移动"；有挡 → 取消

这 8 个子系统对应后面教程的 8 大模块：

### 3.3 原子化 → Cocos 3.8 的概念映射表（全教程最重要的一张表）

这张表读者每次看到不懂的参考代码都该回来翻一下：

| AtomEngine 概念 | 作用 | Cocos Creator 3.8 里对应什么 |
|-----------------|------|------------------------------|
| `LogicAtom` + `onChain` | 响应某个信号触发的一段逻辑 | 事件监听回调（`node.on()` / `EventTarget`）或 controller 里的一个方法 |
| `ComponentAtom` | ECS 的 Component，存数据 | 一个 TypeScript 接口 / 一个 cc.Component 里的字段 |
| `RenderAtom` + `defineRender()` | 声明式描述 UI 结构 + 数据绑定 | Cocos 的 **Prefab**（结构）+ 一个 View 组件脚本（绑数据） |
| `EntityDomainAtom` | 管理某一类实体的集合（"所有箭头"） | 一个数组 / Map，或者通过 `node.getComponentsInChildren(ArrowView)` |
| `ConfigAtom` + `defineConfig()` | 静态配置（资源 URL、数值） | JSON 文件（关卡）/ const 模块（数值）/ `@property` 暴露（可调） |
| `FunctionAtom` + `defineFunction()` | 纯函数 | 普通 TypeScript 函数（`export function xxx()`） |
| `SignalAtom` + `ChainAtom` | 信号总线 + 信号间的链接 | `EventTarget` 或 `node.emit()`；大多数时候**直接函数调用就够了** |
| `routeDispatch` / `routeFilter` / `routeTransform` | 信号派发的三段：过滤/转换/派发 | if-else + 函数调用，本教程拆不了也没必要 |
| `@ae.bindAtom('Xxx')` | 声明依赖另一个原子（注入） | `import` + 属性赋值，或 `@property` 引用 |
| `worldMeta.worldTime` | 全局时间（秒） | `director.getTotalTime() / 1000`，或组件里 `this.totalTime += dt` |
| `defineStorageData()` | 持久化存档 | `sys.localStorage` 读写 JSON |

> **一条铁律**：AtomEngine 是用"原子 + 信号"把一段代码切成几十块小肉末，追求极致的"单一职责 + 可替换 + 可观测"。
> Cocos 3.8 原生写法不需要这种切分，一个 `GameController` 组件配 5~8 个方法就能干完原子引擎里 130 多个 Logic 的事。
> **不要试图 1:1 翻译。翻译玩法，不翻译原子。**

### 4.2 为什么这样分

**Linus 式的 3 条理由**：

1. **核心逻辑和 Cocos 解耦**（`core/` 目录）。
   - 关卡数据、状态机、碰撞检测是纯函数 + 纯数据，不依赖 `cc.*`。
   - 好处：可以写单元测试（jest/vitest）、可以换引擎、可以在浏览器 console 里手动调。
   - G3_FBase 里 `FunctionAtom` 就是这个思路的工业化版本。

2. **View 层只管显示，不管游戏规则**（`game/xxxView.ts`）。
   - ArrowView 收到"你现在是 Start 状态、坐标是 (x, y)" → 把箭头画在对应位置。
   - ArrowView 绝对不判断"前面有没有挡住"。
   - 这和 G3_FBase 里 `RenderAtom` 只有 `defineRender()`、不写命令式代码是一个道理。

3. **数据层单例，不到处传参**（`data/GameData.ts`）。
   - 当前关卡、HP、本关撞了几次 → 放 GameData。
   - 对应 G3_FBase 的 `CombatInitLogic.defineStorageData()`。

### 为什么是这个顺序

- **1~3 章是地基**：没有关卡数据就没有一切。
- **4~6 章只画静态**：先确保能看见，再聊动。这是排错最容易的方式。

## 附录 B：我（作者）对参考项目的诚实评价

> Linus 审稿模式，一码归一码。

**🟢 好品味的地方**

- `ArrowMoveMode` 枚举把状态全部收敛到 5 个值，没有 `isMoving`/`hasStarted`/`hasFailed` 这种一串布尔的垃圾写法。教程会 100% 沿用这个设计。
- 关卡配置用"坐标数组"而不是"二维矩阵数字标记"，扩展性和简洁度都好，直接沿用。
- `CombatInitLogic` 的 `storage.combat` 把一局内的所有可变状态聚在一起，断电/后台切换都能恢复，这是经验之谈。

**🟡 凑合的地方**

- 136 个 Logic 里相当一部分是"Dispatch / Filter / Transform"三段式的产物，对于新手这种切分过于工业化。**本教程会合并掉**，一个方法搞定。
- `G3_FBase_CombatFireLogic` 里的 `arrowsInfo[k].collides` 是预计算的，首关没用上（运行时动态算的）。**本教程不做预计算**，代码少一半。

**🔴 新手容易学偏的地方**

- `RenderAtom` 里 `defineRender()` 返回的是一棵声明式树，看着像 React，但这是 AtomEngine 特有的东西，Cocos 3.8 里**没有这个**，对应概念是 Prefab + 响应式数据绑定。教程会反复强调：**不要试图在 Cocos 3.8 里自己造一个 defineRender()**。
- 某些地方用信号链做了很细粒度的"先 A 完再 B 再 C"，对应 Cocos 3.8 里就是一段顺序代码。新手看了会以为"不用信号就写不对"，**完全错误**。
