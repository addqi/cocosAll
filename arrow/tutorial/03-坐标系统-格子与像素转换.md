# 03 · 坐标系统：格子坐标 ↔ 像素坐标

## 本节目标

**在屏幕中央画一个红色小方块**，通过纯代码放到"格子 `[row=1, col=1]`"的位置，验证格子坐标到像素坐标的转换公式正确。

启动游戏后：

- 屏幕正中偏左上，出现一个 20×20 的红色方块
- Console 输出：`[Arrow] Dot at grid [1,1] -> pixel (-100, 100)`

一句话：**把"第几行第几列"翻译成屏幕上的"x, y"**。

---

## 需求分析

### 为什么要单独一章做"坐标"

看起来就是加减乘除几行代码，为什么单开一章？

**因为坐标系是整个游戏的语言**。

- 关卡数据说的是 `[row=1, col=2]`（格子坐标）
- Cocos 节点 `node.setPosition(x, y)` 要的是像素坐标
- 这两种坐标**全程来回转换**：画点、画箭头、点击检测、碰撞判断、射线绘制……

**如果这一章写错 1 个符号，后面 15 章每一章都看到画面错位**。现在多花 20 分钟写清楚，省后面几小时。

### 两个坐标系的长相

**格子坐标系**（来自 JSON 数据）：

```
col=1  col=2  col=3  col=4  col=5
 ●     ●     ●     ●     ●      row=1
 ●     ●     ●     ●     ●      row=2
 ●     ●     ●     ●     ●      row=3
 ●     ●     ●     ●     ●      row=4
 ●     ●     ●     ●     ●      row=5
```

- `row` 从 1 开始，**从上往下递增**（第 1 行在最上）
- `col` 从 1 开始，**从左往右递增**（第 1 列在最左）

**Cocos 3.8 屏幕坐标系**：

```
                +y 向上
                 ▲
                 │
                 │
    ─────────────┼─────────────►  +x 向右
                 │         (0,0) 在屏幕中央
                 │
                 │
```

- 原点 `(0, 0)` 在 Canvas 中央（不是左上）
- x 向右为正，y **向上为正**（和屏幕像素习惯相反，和数学直觉一致）

### 冲突点 + 翻译规则

| | 格子 | Cocos 像素 |
|---|------|-----------|
| 横向 | col 从 1 开始向右 | x 向右 |
| 纵向 | row 从 1 开始**向下** | y **向上** |
| 原点 | 左上 | 中心 |

所以**翻译要做三件事**：

1. 把 col 换算成 x：`x = (col - offsetCol) * gap`
2. 把 row 换算成 y，**并翻转方向**：`y = (offsetRow - row) * gap`
3. 让整个棋盘围绕 (0,0) 居中

## 实现思路

### 参考 G3_FBase 的方案

G3_FBase 里对应这一层的代码主要在：

- `src/configs/G3_FBase_PointConfig.ts` 定义了 `gap = 50`（格子间距）和 `size = 9`（点大小）
- `src/logics/G3_FBase_WindowSizeLogic.ts` 计算棋盘面板的宽高、原点
- `src/functions/G3_FBase_PointCoordFunction.ts`（虽然我们没读过，但从 `CombatPointsInitLogic` 的调用能看出）负责把格子坐标转像素

G3_FBase 算的是这个：

```typescript
panel.width  = (cols + 1) * gap;  // cols=5, gap=50 → 300
panel.height = (rows + 1) * gap;  // rows=5, gap=50 → 300
origin.x = -panel.width / 2;       // -150
origin.y = -panel.height / 2;      // -150
```

这是**把 panel 当成"左下角在 origin、右上角在 (panel.width/2, panel.height/2)"的矩形**，然后格子 `[row, col]` 的位置就是相对 origin 的偏移。

我们完全沿用这个公式。

### 居中对齐的推导

以 5×5 棋盘、`gap=50` 为例：

```
希望画成这样（X 是中心点 (0,0)）:

col=1  col=2  col=3  col=4  col=5
 ●     ●     ●     ●     ●       row=1  ← y = +100
 ●     ●     ●     ●     ●       row=2  ← y = +50
 ●     ●     ● X   ●     ●       row=3  ← y = 0 (中心)
 ●     ●     ●     ●     ●       row=4  ← y = -50
 ●     ●     ●     ●     ●       row=5  ← y = -100
 ↑     ↑     ↑     ↑     ↑
x=-100 x=-50 x=0  x=50 x=100
```

推导公式：

- 中心行 `row = (rows + 1) / 2 = 3`，对应 `y = 0`
- `row = 1` 对应 `y = +100`，`row = 5` 对应 `y = -100`
- 所以 `y = (centerRow - row) * gap`，其中 `centerRow = (rows + 1) / 2`

同理：

- `x = (col - centerCol) * gap`，其中 `centerCol = (cols + 1) / 2`

**不偶数/奇数分情况**。公式对 `rows=4`、`rows=6` 一样成立（中心行是 2.5 / 3.5，反正不画到那里）。这是"**消除特殊情况**"的典型——公式统一，代码只有一条路径。

### 反向转换（像素 → 格子）

后面点击检测要用：玩家点了屏幕 `(x, y)`，算出这是哪个格子。

由正向公式反解：

```
col = x / gap + centerCol
row = centerRow - y / gap
```

再四舍五入到整数：

```typescript
col = Math.round(x / gap + centerCol);
row = Math.round(centerRow - y / gap);
```

### 文件 1：`assets/scripts/common/Config.ts`（全局数值）

在 `scripts/common/` 下新建 `Config.ts`：

```typescript
/**
 * 全局数值配置。所有"魔数"都进这里，绝对不在业务代码里写 50 / 100 这种。
 * 对应 G3_FBase 里的多个 ConfigAtom（PointConfig / ArrowConfig / ...）。
 */
export const Config = {
    /** 格子之间的距离（像素）。对应 G3_FBase_PointConfig.gap */
    gap: 100,

    /** 点的大小（像素）。对应 G3_FBase_PointConfig.size */
    pointSize: 12,

    /** 箭头线宽（像素）。对应 G3_FBase_ArrowConfig.size */
    arrowLineWidth: 10,

    /** 箭头头部大小（像素）。对应 G3_FBase_ArrowConfig.headSize */
    arrowHeadSize: 24,

```

**说明**：

- 用 `as const` 让类型系统把 `Config.gap` 推为字面量 `100` 而不是 `number`，防止误改。
- `gap` 选 100 不是 50：新手教程里大一点看得清。后面想改了随时回来调，所有用到 gap 的地方都会跟着变。**这就是为什么要有 Config 文件**。

### 文件 2：`assets/scripts/core/Coord.ts`（坐标转换，纯函数）

在 `scripts/core/` 下新建 `Coord.ts`：

```typescript
import { Config } from '../common/Config';

/** 像素坐标 */
export interface Pixel {
    x: number;
    y: number;
}

/**
 * 格子坐标 → 像素坐标（棋盘居中于 (0, 0)）
 * @param row  1-based 行号，从上往下
 * @param col  1-based 列号，从左往右
 * @param rows 棋盘总行数
 * @param cols 棋盘总列数
 */
export function gridToPixel(
    row: number, col: number, rows: number, cols: number,
): Pixel {
    const centerRow = (rows + 1) / 2;
    const centerCol = (cols + 1) / 2;
    return {
        x: (col - centerCol) * Config.gap,
        y: (centerRow - row) * Config.gap,
    };
}

/**
 * 像素坐标 → 格子坐标（最近的格子，可能越界）
 */
export function pixelToGrid(
    x: number, y: number, rows: number, cols: number,
): { row: number; col: number } {
    const centerRow = (rows + 1) / 2;
    const centerCol = (cols + 1) / 2;
    return {
        row: Math.round(centerRow - y / Config.gap),
        col: Math.round(x / Config.gap + centerCol),
    };
}

/** 判断格子坐标是否在棋盘范围内 */
export function isInsideBoard(row: number, col: number, rows: number, cols: number): boolean {
    return row >= 1 && row <= rows && col >= 1 && col <= cols;
}
```

**关键点**：

- **三个函数全是纯函数**：输入决定输出，没有副作用。这是 `core/` 的铁律。
- `gridToPixel` 和 `pixelToGrid` 互为逆运算，代码结构对称。对应 Linus 说的"**好代码没有特殊情况**"——正反两个方向用同一套参数。
- `isInsideBoard` 单独拎出来，因为这个判断后面几乎每一章都用（越界 = 飞出了 = 胜利）。

### 文件 3：改造 `GameController.ts`（画一个测试点）

把 `scripts/game/GameController.ts` 替换为：

     * 造一个 1x1 的纯色 SpriteFrame。

     * 自己 new 一个最稳。

        const frame = new SpriteFrame();

> ⚠️ 上面的 `makeSolidColorFrame` 其实返回了一个空 SpriteFrame，画面会出不来。**这是故意的"反例"**，看下面的正确写法。

**正确做法：用 Graphics 组件画**（推荐）

Graphics 是 Cocos 3.8 画几何图形的原生组件，不需要任何 SpriteFrame 资源。把 `spawnTestDot` 和 `makeSolidColorFrame` 整体替换成：

```typescript
import { Graphics } from 'cc';  // 顶部 import 里补上

// ... 类体内：
private spawnTestDot(x: number, y: number) {
    const dot = new Node('TestDot');
    dot.setPosition(new Vec3(x, y, 0));
    // Graphics 需要 UITransform
    dot.addComponent(UITransform).setContentSize(20, 20);
    const g = dot.addComponent(Graphics);
    g.fillColor = Color.RED;
    g.rect(-10, -10, 20, 20);  // 相对自身中心画 20x20
    g.fill();
    this.node.addChild(dot);
}
```

**关键概念**：

- `UITransform`：Cocos 3.8 里所有**2D UI 节点**都需要这个组件，管节点的宽高和锚点。没有它，Graphics 画不出来。
- `Graphics`：2D 绘图组件，有 `fillColor` / `rect()` / `circle()` / `moveTo()` / `lineTo()` / `fill()` / `stroke()` 一堆 API。**本教程 90% 的画面都是它画的**（参考项目 G3_FBase 的 `CombatArrowRender`、`CombatRayRender` 也是 Graphics）。
- `this.node.addChild(dot)`：`this.node` 是 Canvas（`GameController` 挂在 Canvas 上）。把红点作为 Canvas 的子节点，它就出现在 UI 层。

## 易错点

### 易错 1：忘了 `UITransform`，Graphics 什么都画不出

```typescript
const g = dot.addComponent(Graphics);  // ❌ 没加 UITransform
```

Graphics 需要节点有 UITransform 组件（提供 contentSize、anchor）才能正确计算坐标。**规则：创建 2D UI 节点 = 先 `addComponent(UITransform)`，再加 Graphics/Sprite/Label**。

### 易错 2：y 轴方向搞反

新手写过这样的公式：

```typescript
y: (row - centerRow) * Config.gap  // ❌ 忘了翻转
```

结果：`row=1` 画在屏幕下方，`row=5` 画在屏幕上方。**整个画面上下颠倒**。

**正确方向**：`(centerRow - row)`。row 小 = y 大 = 屏幕上方。

**调试技巧**：`console.log` 打出来目测一下，`row=1` 的 y 应该是最大正数。

### 易错 3：节点父子关系错了，节点画出去了但看不见

```typescript
director.getScene().addChild(dot);  // ❌ 加到场景根节点，不在 Canvas 下
```

Cocos 3.8 的 2D UI 必须在 **Canvas 节点的子树**下才会渲染。因为 Canvas 带着 `RenderRoot2D` 组件，负责 2D 渲染批次。

**正确写法**：挂在 `this.node`（也就是 Canvas）下，或者 Canvas 的某个子节点。

### 易错 4：Config.gap 改了之后没生效

改 `Config.gap = 100 → 120`，保存代码，浏览器**强制刷新**（Cmd+Shift+R / Ctrl+F5）。

Cocos 预览有时候 HMR（热更新）失败，尤其是改了常量文件。**发现数值没变，第一反应先强刷一次**。

### 易错 5：公式推成偶数 rows 就错了

偶数情况没有"真正的中心行"。比如 `rows=4`，`centerRow = 2.5`。

- `row=1` 的 y = `(2.5 - 1) * 100 = 150` ✅
- `row=4` 的 y = `(2.5 - 4) * 100 = -150` ✅

公式一样对。**新手常见误区**：觉得要给偶数单独写 if-else，其实完全不用。
