# 08 · 点击箭头 → 触发状态转移

## 本节目标

**让玩家点击任意一根箭头，它的颜色从灰色变成蓝色**（Idle → Start），但**暂时不动**。

预期效果：

- 点击任何一根箭头 → 它变蓝。
- 再次点击同一根 → **没反应**（因为已经不是 Idle 了）。
- 点击棋盘空白区域 → **没反应**。
- Console 打印：`[Arrow] Arrow 0 fired. mode = Start`

一句话：**用户操作第一次改变游戏状态**。

---

## 需求分析

这一章要连通三件事：

1. **怎么拿到用户点击**（Cocos 3.8 的触摸事件）。
2. **怎么知道点的是哪一根箭头**（坐标转换 + 箭头范围检测）。
3. **怎么把"点击"翻译成"状态转移"**（调 `fire(rt, false)`）。

对应 G3_FBase 的：

| G3_FBase | 我们 |
|----------|------|
| `TouchEndToArrowTransformLogic.ts`（pixel → arrowIndex） | `InputController.pickArrow()` |
| `TouchEndArrowToMoveDispatchLogic.ts`（判断无碰撞） | 本章**先跳过**，默认 `blocked = false`（第 11 章再加） |
| `TouchStartLogic.ts` + `TouchEndLogic.ts` | `InputController.onTouchEnd()` |
| `CombatArrowRender.ts` 的 color 绑定 | `ArrowView.refresh()` |

参考项目用了 3 个文件（Transform + Dispatch + Logic），我们合成 **1 个 InputController + 1 个 ArrowView.refresh()**。

---

## 实现思路

### 点击哪根箭头？

玩家在屏幕点一下 `(screenX, screenY)` → 要算出点到了哪根箭头。

**方法 1（简单）：以箭头头格子为圆心，在一定半径内算命中**。

```typescript
for (const [i, rt] of runtimes) {
    const head = rt.coords[rt.coords.length - 1];
    const p = gridToPixel(head[0], head[1], rows, cols);
    const distance = Math.hypot(localX - p.x, localY - p.y);
    if (distance < threshold) return i;
}
```

**方法 2（精确）：检测点是否在箭头的 coords 任意一格内**。

```typescript
// 把 localX, localY 反算成 [row, col]，看等不等于 coords 里任一格
const { row, col } = pixelToGrid(localX, localY, rows, cols);
for (const [i, rt] of runtimes) {
    if (rt.coords.some(c => c[0] === row && c[1] === col)) return i;
}
```

**本教程选方法 2**。理由：

- 直接复用了 `pixelToGrid`，已经有了。
- 玩家点击箭头身体任何位置都算命中，更友好。
- G3_FBase 的 `findArrowIndex` 实际上也是这个思路（虽然细节更精细）。

### 屏幕坐标 → 本地坐标

Cocos 3.8 的触摸事件返回的是**UI 坐标**（屏幕空间，原点左下角）。我们要的是**BoardView 节点本地坐标**（原点 BoardView 中心）。

```typescript
const uiLocation = event.getUILocation();  // {x, y}，相对 Canvas 左下
const worldPos = new Vec3(uiLocation.x, uiLocation.y, 0);
const localPos = boardView.node.getComponent(UITransform).convertToNodeSpaceAR(worldPos);
```

**最后还要除以 scale**，因为 BoardView 节点在第 06 章被 `setScale(layout.scale)` 缩放过，`convertToNodeSpaceAR` 会自动处理这个 scale。**所以 localPos 拿到的就是"未缩放的本地坐标"**，跟 `gridToPixel` 产出的坐标系一致。

> **Linus 的品味**：`convertToNodeSpaceAR` 自动处理 scale 这件事情，是 Cocos 帮你消除了特殊情况。不要自己 `/= scale` 重新发明轮子。

### 谁持有 `ArrowRuntime[]`

数据的家。候选：

- A. 放在 `GameController` 里（作为场景总控，合理）。
- B. 放在 `BoardView` 里（它已经有 `ArrowView[]` 了）。
- C. 放在 `InputController` 里。

**选 A**。理由：

- `ArrowRuntime[]` 是游戏逻辑数据，跟"画面视图"（BoardView/ArrowView）是两回事。
- `InputController` 只应该有"怎么处理输入"的行为，不应该持有数据。
- 以后 HP、level 这些也要放 GameController（第 14、18 章），统一起来。

**对应 G3_FBase**：`CombatInitLogic.defineStorageData()` 里把 `combat.hp`、`combat.arrows[index].moveMode` 等都挂在 combat 这个结构上。我们的 GameController 就是 combat。

## 代码实现

### 文件 1：扩展 `core/Coord.ts`（加一个 `findArrowIndex`）

在 `Coord.ts` 末尾追加：

```typescript
import { ArrowRuntime } from './ArrowState';

/**
 * 根据本地像素坐标查找被点中的箭头索引。
 * 找不到返回 -1。
 */
export function findArrowIndex(
    localX: number, localY: number,
    runtimes: ArrowRuntime[],
    rows: number, cols: number,
): number {
    const { row, col } = pixelToGrid(localX, localY, rows, cols);
    for (let i = 0; i < runtimes.length; i++) {
        const rt = runtimes[i];
        if (rt.coords.some(c => c[0] === row && c[1] === col)) {
            return i;
        }
    }
    return -1;
}
```

### 文件 2：扩展 `ArrowView.ts`（加 `refresh` 方法）

原本 `setData` 只画一次。现在状态会变，需要 `refresh` 根据当前状态更新颜色。

```typescript
import {
    _decorator, Component, UITransform, Graphics, Color, Vec3,
} from 'cc';
import { ArrowData } from '../core/LevelData';
import { ArrowRuntime, ArrowMoveMode } from '../core/ArrowState';
import { gridToPixel } from '../core/Coord';
import { Config } from '../common/Config';
const { ccclass } = _decorator;

/** 颜色方案，数值取自 G3_FBase */
const COLOR_IDLE  = new Color(0x11, 0x16, 0x33, 0xff); // arrowIdleColor
const COLOR_MOVE  = new Color(0x5b, 0x72, 0xfe, 0xff); // arrowMoveColor
const COLOR_STOP  = new Color(0xfe, 0x4b, 0x5e, 0xff); // arrowStopColor

@ccclass('ArrowView')
export class ArrowView extends Component {
    private graphics: Graphics | null = null;
    private data: ArrowData | null = null;
    private rows = 0;
    private cols = 0;

    setData(data: ArrowData, rows: number, cols: number) {
        this.data = data;
        this.rows = rows;
        this.cols = cols;
        this.ensureGraphics();
    }

    /** 由外部在每次状态变化时调用 */
    refresh(rt: ArrowRuntime) {
        if (!this.data || !this.graphics) return;
        this.drawArrow(rt);
    }

    private ensureGraphics() {
        if (this.graphics) return;
        if (!this.node.getComponent(UITransform)) {
            this.node.addComponent(UITransform);
        }
        this.graphics = this.node.addComponent(Graphics);
    }

    private drawArrow(rt: ArrowRuntime) {
        const g = this.graphics!;
        g.clear();

        const color = this.pickColor(rt);
        const { direction } = this.data!;
        const coords = rt.coords;
        if (coords.length === 0) return;

        const tail = coords[0];
        const head = coords[coords.length - 1];
        const tailPx = gridToPixel(tail[0], tail[1], this.rows, this.cols);
        const headPx = gridToPixel(head[0], head[1], this.rows, this.cols);

        const pdx = direction[1];
        const pdy = -direction[0];

        g.strokeColor = color;
        g.lineWidth = Config.arrowLineWidth;
        g.moveTo(tailPx.x, tailPx.y);
        g.lineTo(headPx.x, headPx.y);
        g.stroke();

        const s = Config.arrowHeadSize;
        const nx = -pdy, ny = pdx;
        const tipX  = headPx.x + pdx * s;
        const tipY  = headPx.y + pdy * s;
        const leftX = headPx.x + nx * s / 2;
        const leftY = headPx.y + ny * s / 2;
        const rightX = headPx.x - nx * s / 2;
        const rightY = headPx.y - ny * s / 2;

        g.fillColor = color;
        g.moveTo(tipX, tipY);
        g.lineTo(leftX, leftY);
        g.lineTo(rightX, rightY);
        g.close();
        g.fill();
    }

    private pickColor(rt: ArrowRuntime): Color {
        if (rt.mode === ArrowMoveMode.Start || rt.mode === ArrowMoveMode.End) {
            return COLOR_MOVE;
        }
        if (rt.mode === ArrowMoveMode.Collide || rt.mode === ArrowMoveMode.Back) {
            return COLOR_STOP;
        }
        if (rt.hasFailed) return COLOR_STOP;
        return COLOR_IDLE;
    }
}
```

**关键改动**：

- 原来的 `setData(data, rows, cols)` 只存配置，**不画**。
- **新增 `refresh(rt)`**：这是画面和状态的接口。每次状态变化都要调一次。
- **`pickColor` 独立纯函数**：把"状态→颜色"的映射集中到一处。G3_FBase 的 `CombatArrowRender` 里这段逻辑混在 `draw` 里一大堆 if-else，不如独立函数清爽。

### 文件 3：`game/InputController.ts`（新增）

```typescript
import {
    _decorator, Component, EventTouch, Input, UITransform, Vec3, Node,
} from 'cc';
import { findArrowIndex } from '../core/Coord';
import { ArrowRuntime } from '../core/ArrowState';
import { LevelData } from '../core/LevelData';
const { ccclass } = _decorator;

export type ArrowClickHandler = (arrowIndex: number) => void;

/**
 * 输入控制器。挂在 BoardView 节点上，监听触摸事件。
 * 把"触摸坐标"翻译成"箭头索引"，通过回调交给 GameController 处理。
 * 对应 G3_FBase 的 TouchStartLogic + TouchEndLogic + TouchEndToArrowTransformLogic。
 */
@ccclass('InputController')
export class InputController extends Component {
    private data: LevelData | null = null;
    private runtimes: ArrowRuntime[] = [];
    private onArrowClick: ArrowClickHandler | null = null;

    setup(data: LevelData, runtimes: ArrowRuntime[], handler: ArrowClickHandler) {
        this.data = data;
        this.runtimes = runtimes;
        this.onArrowClick = handler;

        // 确保节点大小足够接收触摸（填满父节点）
        this.ensureTouchableSize();

        this.node.on(Input.EventType.TOUCH_END, this.onTouchEnd, this);
    }

    onDestroy() {
        this.node.off(Input.EventType.TOUCH_END, this.onTouchEnd, this);
    }

    private ensureTouchableSize() {
        let ui = this.node.getComponent(UITransform);
        if (!ui) ui = this.node.addComponent(UITransform);
        // 给个极大的 contentSize 保证任何屏幕点击都能命中这个节点
        ui.setContentSize(100000, 100000);
    }

    private onTouchEnd(event: EventTouch) {
        if (!this.data || !this.onArrowClick) return;

        const uiLoc = event.getUILocation();
        const world = new Vec3(uiLoc.x, uiLoc.y, 0);
        const ui = this.node.getComponent(UITransform)!;
        const local = ui.convertToNodeSpaceAR(world);

        const idx = findArrowIndex(local.x, local.y, this.runtimes, this.data.rows, this.data.cols);
        if (idx < 0) return;

        this.onArrowClick(idx);
    }
}
```

**关键点**：

- **`setup()` 是依赖注入**。GameController 把自己持有的数据和回调注进来，InputController 不自己存数据。
- **`ensureTouchableSize`** 很重要。Cocos 3.8 的 `TOUCH_*` 事件默认只在**节点的 contentSize 范围内**触发。BoardView 节点没设 size → 默认 `(0,0)` → 什么都收不到。设个 10 万 × 10 万确保能接到全屏触摸。
- **`TOUCH_END` 而不是 `TOUCH_START`**。参考 G3_FBase 用的也是 TouchEnd（见 `TouchEndLogic`）。TouchStart 会和"长按"混淆，等到第 20 章长按 Ray 才用 TouchStart。
- **`this.onArrowClick(idx)` 回调**：InputController 不知道 GameController 是谁，也不调用 fire()。这就是"职责单一"：输入组件只做"解析输入"，不做"决定怎么响应"。

### 文件 4：改造 `GameController.ts`（接入 InputController）

```typescript
import {
    _decorator, Component, resources, JsonAsset, Node,
} from 'cc';
import { LevelData, validateLevelData } from '../core/LevelData';
import { BoardView } from './BoardView';
import {
    createRuntime, fire, canFire, ArrowRuntime, ArrowMoveMode,
} from '../core/ArrowState';
import { InputController } from './InputController';
const { ccclass } = _decorator;

@ccclass('GameController')
export class GameController extends Component {
    private levelData: LevelData | null = null;
    private boardView: BoardView | null = null;
    private input: InputController | null = null;

    /** 箭头运行时状态，按索引和 levelData.arrows 一一对应 */
    private runtimes: ArrowRuntime[] = [];

    onLoad() {
        console.log('[Arrow] Game scene loaded.');
        this.boardView = this.createBoardView();
        this.input = this.boardView.node.addComponent(InputController);
        this.loadLevel(1);
    }

    private createBoardView(): BoardView {
        const node = new Node('BoardView');
        this.node.addChild(node);
        return node.addComponent(BoardView);
    }

    private loadLevel(levelNo: number) {
        const no = levelNo < 10 ? `0${levelNo}` : `${levelNo}`;
        const path = `levels/level_${no}`;
        resources.load(path, JsonAsset, (err, asset) => {
            if (err) { console.error(`[Arrow] Load failed`, err); return; }
            try {
                this.levelData = validateLevelData(asset.json);
                this.onLevelLoaded(this.levelData);
            } catch (e) { console.error(`[Arrow] Invalid level:`, e); }
        });
    }

    private onLevelLoaded(data: LevelData) {
        console.log(`[Arrow] Level loaded: ${data.rows}x${data.cols}, arrows=${data.arrows.length}`);
        this.runtimes = data.arrows.map(a => createRuntime(a));
        this.boardView?.render(data);
        this.refreshAllArrows();
        this.input?.setup(data, this.runtimes, (idx) => this.onArrowClick(idx));
    }

    private onArrowClick(idx: number) {
        const rt = this.runtimes[idx];
        if (!canFire(rt)) return;
        fire(rt, false);  // blocked=false，第 11 章再加碰撞预检
        console.log(`[Arrow] Arrow ${idx} fired. mode = ${ArrowMoveMode[rt.mode]}`);
        this.refreshArrow(idx);
    }

    private refreshAllArrows() {
        const views = this.boardView?.getArrowViews() || [];
        for (let i = 0; i < views.length; i++) {
            views[i].refresh(this.runtimes[i]);
        }
    }

    private refreshArrow(idx: number) {
        const views = this.boardView?.getArrowViews() || [];
        views[idx]?.refresh(this.runtimes[idx]);
    }
}
```

> 🛈 **屏幕适配不在本章处理**。本章之后的 `GameController` 一直保持这个"不管屏幕尺寸"的样子，直到第 14 章之后（屏幕适配章节）再统一补上 `computeBoardLayout` + `canvas-resize`。默认窗口下画面能看见，开发时先别被这个问题拖住。

**关键数据流**：

```
用户点击
  ↓
InputController.onTouchEnd
  ↓ findArrowIndex
onArrowClick(idx)  ← 回调到 GameController
  ↓ fire(rt, false)
runtimes[idx].mode = Start
  ↓ refreshArrow(idx)
ArrowView.refresh(rt)
  ↓ pickColor → drawArrow
画面变蓝
```

**一条单向链**。InputController 不知道 fire()，ArrowView 不知道触摸，GameController 作为控制器串起一切。

### 文件 5：`BoardView.ts` 删掉 `setData` 里的首次 render

现在刷新由 `refreshAllArrows` 主动调用。**删掉 ArrowView 原先 `setData` 末尾的 `this.redraw()`**，改成 `setData` 只存数据，不画；`refresh(rt)` 才画。

已经在上面"文件 2"里体现了。

## 易错点

### 易错 1：节点没 `UITransform` 或 `contentSize` 是 0，触摸收不到

这是本章最常见的坑。**`TOUCH_*` 事件只在节点实际占据的像素区域内触发**。BoardView 节点初始 contentSize 是 `(0,0)`。不主动设就永远收不到触摸。

**规则**：需要接收触摸的节点，**必须有 UITransform + 非零 contentSize**。

### 易错 2：`getUILocation` vs `getLocation`

```typescript
event.getLocation()     // 屏幕物理像素坐标
event.getUILocation()   // UI 坐标（考虑设计分辨率、DPR）
```

用 `getLocation` 在高 DPR 手机（iPhone）上会**偏移成 2~3 倍**。

**规则**：Cocos 3.8 触摸处理永远用 `getUILocation`。

### 易错 3：忘了 `convertToNodeSpaceAR` 的 AR

```typescript
ui.convertToNodeSpace(world)   // ❌ 不考虑 anchor
ui.convertToNodeSpaceAR(world) // ✅ Anchor-Relative，考虑锚点
```

Cocos 3.8 节点 anchor 默认 `(0.5, 0.5)`（中心）。**AR 版本**会把坐标换算成"相对节点中心"，和我们 `gridToPixel` 输出的"中心为原点"坐标对得上。不带 AR 会以左下角为原点，**坐标全偏**。

### 易错 4：`ArrowRuntime[]` 和 `ArrowView[]` 下标不一致

如果你关卡过滤掉某些箭头、或者排序了 ArrowView，**下标就不对应了**。`refreshArrow(2)` 可能把第 2 根的状态刷到第 0 根的视图上。

**规则**：**用 index 关联的数据，必须顺序完全一致**。`runtimes[i]` 对应 `levelData.arrows[i]` 对应 `boardView.getArrowViews()[i]`，三条全程一致。哪天想改顺序，三处一起改。

### 易错 5：触摸后没 `refresh`，画面不变

```typescript
fire(rt, false);
// 忘了 this.refreshArrow(idx);
```

状态变了、日志打了，但画面还是灰色。因为 Cocos 没有"数据变了自动刷画面"的响应式机制（Vue/React 那种）。**手动调 refresh 是必须的**。

**对照 G3_FBase**：参考项目用 `defineRender` 的 `[arrowComponent, 'color']` 依赖数组实现了响应式，数据改 → 视图自动刷。**Cocos 3.8 原生没有，我们用手动调的土办法**。大多数场合够用。
