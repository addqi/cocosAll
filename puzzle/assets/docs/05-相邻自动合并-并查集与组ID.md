# 05 · 相邻自动合并 — 并查集与组 ID

> **目标**：相邻两块"相对位置正确"时**自动合并**到一组（同一 groupId）。**注意**——本节只更新 groupId，不改 slots 也不改 position。06 节才会让"组" 真正联动起来（一拖整组都动）。

---

## 起点回顾

04 节末态：

- 拖一块到另一块的位置 → 两块 swap。
- 每块 `groupId === pieceId`（每块自成一组）—— 01 节的初始值，后续没人改它。

本节做两件事：

1. 实现并查集（Union-Find）数据结构 + 算法。
2. 写 `mergeScan()`：扫描所有相邻槽位对，相对位置正确的合到同一组。

**不改的地方**：slots 不动、position 不动、UI 不变。本节是**纯逻辑层**升级——肉眼看不到任何变化，但 console.log 能验证 groupId 在动。06 节才把 groupId 用起来。

---

## 1. 关键概念：Rule B（相对位置正确）

最直觉的判合并："两块都在正确槽位 + 相邻 = 合并"——这是 **Rule A**。

**Rule A 的问题**：

```text
slots = [3, 4, ?, ?, ?, ?, ?, ?, ?]
       ↑   ↑
       槽 0    槽 1
```

3 号块在槽 0、4 号块在槽 1。它们**相邻**（槽 0/槽 1 横向挨着），但**都没在正确位置**（3 号该在槽 3、4 号该在槽 4）。

**它们应该合并吗？应该！**

因为 3 号块和 4 号块的"正确相对位置" 是"3 在 4 左边"——而它们现在的实际相对位置也是"3 在 4 左边"（slots[0]=3 在 slots[1]=4 的左边）。**相对关系对了**。

这就是 **Rule B**：判合并只看**相对位置**，不看**绝对位置**。

```text
Rule A: piece.row === piece.正确row && piece.col === piece.正确col
        AND 邻居 piece.row === 邻居.正确row && 邻居.col === 邻居.正确col

Rule B: piece.row + 1 === 邻居.正确row（左邻居）
        OR piece.col + 1 === 邻居.正确col（上邻居）
        ...类似
```

Rule B 比 Rule A 宽松得多，符合玩家心智："我把对的那两块凑一起就该合上"——不用先拼回正确位置。

**这是拼图玩法的微妙细节，决定游戏体验的好坏**。商业拼图游戏多数用 Rule B。

## 2. 关键概念：并查集（Union-Find）

并查集解决一类问题：**N 个元素，反复"合并两个" / "查询是否同组"**。

朴素做法：每个元素挂个 `groupId`，合并时把一组所有元素的 groupId 都改成另一组的——O(N) per merge。

**并查集做法**：每个元素挂个 `parent[i]`，初始 `parent[i] = i`（每元素自成树根）。

```text
合并：union(a, b) → 找 a 的根 ra、b 的根 rb，让 ra 的 parent 指向 rb（rb 成新根）
查根：find(x) → 沿 parent 链一路向上，直到 parent[x] === x（x 是根）
查组：find(a) === find(b) → 同组
```

加上**路径压缩**（find 时把链上所有节点直接挂到根下），均摊复杂度近乎 O(1)/op。

**模板（10 行）**：

```typescript
function find(parents: number[], x: number): number {
    while (parents[x] !== x) {
        parents[x] = parents[parents[x]]; // 路径压缩
        x = parents[x];
    }
    return x;
}

function union(parents: number[], a: number, b: number): boolean {
    const ra = find(parents, a);
    const rb = find(parents, b);
    if (ra === rb) return false; // 已经同组，无操作
    parents[ra] = rb;
    return true;
}
```

`union` 返回布尔：true = 这次操作真的合并了两个不同组、false = 它们已经同组。后面 mergeScan 会用这个返回值判"本轮有没有发生新合并"。

## 3. 关键概念：mergeScan 全量扫描

每次 swap 后，重新算所有 piece 的 groupId。

**为什么全量扫**：增量更新（"swap 时只看那两块"）容易出 bug——swap 后 4 号块到了新位置，它原来的同组邻居怎么办？新位置的相邻块要不要合？需要"拆组" 吗？写起来 case 一堆。

**全量扫的简单**：直接"重新算一遍" 整盘的 groupId。9 块/64 块的规模下，全量扫 O(N) 跟一帧渲染比是零成本。**笨办法 = 简单办法 = 不出 bug**。

mergeScan 的步骤：

```text
1. parents[i] = i for all i（每块自成一组的初始状态）
2. 双层循环：扫所有槽位 (sr, sc)
   for each 右邻居：检查 piece(sr, sc) 和 piece(sr, sc+1) 是否相对位置正确 → union
   for each 下邻居：检查 piece(sr, sc) 和 piece(sr+1, sc) 是否相对位置正确 → union
3. 写回 piece.groupId = find(parents, piece.pieceId) 给所有块
```

每次 swap 后调一次 mergeScan，groupId 就重算到位。

**关键设计：parents 在 mergeScan 内部"重置 + 重建"**

不要把 parents 当成持久状态——它**只在 mergeScan 这一次执行内有效**。这样的好处是：

- swap 把 A 块和 B 块拆开（不再相邻） → 下次 mergeScan 时 parents 从 0 开始，A 和 B 自然分到不同组。
- "组拆解" 这个动作不需要写代码——**全量重扫天然支持**。

如果 parents 是持久的字段（每次 union 后只加新关系），就要写"拆组" 逻辑——并查集**没有 disunion 操作**（路径压缩破坏了拆解能力），实际上不可能写。**全量重扫规避了这个根本难题**。

## 4. 改 `PuzzleBoard.ts`：加 `mergeScan` + 4 节 swap 后调用

打开 `PuzzleBoard.ts`，**新增 3 个方法**：

```typescript
/**
 * Union-Find 找根（带路径压缩）。
 * 走一遍 parent 链，途中把每个节点直接挂到根下，下次 O(1)。
 */
private find(parents: number[], x: number): number {
    while (parents[x] !== x) {
        parents[x] = parents[parents[x]];
        x = parents[x];
    }
    return x;
}

/**
 * Union-Find 合并。返回 true = 这次真的合了两组、false = 已同组。
 */
private union(parents: number[], a: number, b: number): boolean {
    const ra = this.find(parents, a);
    const rb = this.find(parents, b);
    if (ra === rb) return false;
    parents[ra] = rb;
    return true;
}

/**
 * 全量扫描合并。每次 swap 后调一次。
 *
 * 算法：
 *   1. parents[i] = i（每块自成组的初始状态）
 *   2. 扫所有槽位的右邻居 + 下邻居，相对位置正确就 union
 *   3. 把每块 groupId 写为 find(parents, pieceId)
 *
 * 返回 mergedAny：本轮有没有发生新合并（可用于"播合并音效"等）。
 */
private mergeScan(): boolean {
    const parents: number[] = [];
    for (let pid = 0; pid < this.pieceCount; pid++) parents.push(pid);

    let mergedAny = false;
    for (let sr = 0; sr < this.pieceGrid; sr++) {
        for (let sc = 0; sc < this.pieceGrid; sc++) {
            const idx = sr * this.pieceGrid + sc;
            const pid = this.slots[idx];
            const piece = this.pieceNodes[pid].getComponent(PuzzlePiece)!;

            // 右邻居：当前块的"正确右邻居" 应该是 row=piece.row, col=piece.col+1
            if (sc + 1 < this.pieceGrid) {
                const rPid = this.slots[idx + 1];
                const rPiece = this.pieceNodes[rPid].getComponent(PuzzlePiece)!;
                if (rPiece.row === piece.row && rPiece.col === piece.col + 1) {
                    if (this.union(parents, pid, rPid)) mergedAny = true;
                }
            }

            // 下邻居：当前块的"正确下邻居" 应该是 row=piece.row+1, col=piece.col
            if (sr + 1 < this.pieceGrid) {
                const dPid = this.slots[idx + this.pieceGrid];
                const dPiece = this.pieceNodes[dPid].getComponent(PuzzlePiece)!;
                if (dPiece.row === piece.row + 1 && dPiece.col === piece.col) {
                    if (this.union(parents, pid, dPid)) mergedAny = true;
                }
            }
        }
    }

    // 写回 groupId
    for (let pid = 0; pid < this.pieceCount; pid++) {
        const piece = this.pieceNodes[pid].getComponent(PuzzlePiece)!;
        piece.groupId = this.find(parents, pid);
    }

    return mergedAny;
}
```

**改 `onPieceTouchEnd`** 在 swap 末尾加一行 mergeScan + log：

```typescript
private onPieceTouchEnd(event: EventTouch): void {
    // ... 原有逻辑（弹回 / swap 写 slots + 改 position）...

    // ===== 05 节：swap 后扫一次合并 =====
    const merged = this.mergeScan();
    if (merged) {
        const groupCounts = this.countGroups();
        console.log(
            `[PuzzleBoard] merge happened — 当前 ${groupCounts.length} 组，` +
            `最大组 ${Math.max(...groupCounts)} 块`,
        );
    }
}

/** 统计当前每个组的大小，返回数组（按 groupId 排序）。 */
private countGroups(): number[] {
    const counts: { [key: number]: number } = {};
    for (let pid = 0; pid < this.pieceCount; pid++) {
        const piece = this.pieceNodes[pid].getComponent(PuzzlePiece)!;
        counts[piece.groupId] = (counts[piece.groupId] || 0) + 1;
    }
    return Object.values(counts);
}
```

**还要在 `onLoad` 末尾调一次 mergeScan**——开局洗牌后可能恰好"两块就在相对正确位置"，要立刻合上：

```typescript
onLoad() {
    // ...
    this.layoutAllPieces();
    this.mergeScan(); // ← 新增：开局扫一次
}
```

## 5. 跑起来

▶ 预览。

**测试动作**：

1. 找两块"出生 row/col 相邻" 的（比如 0 号块 = (0,0)、1 号块 = (0,1)）。
2. 把它们 swap 到挨着的位置（哪两个槽都行，**只要 0 号在 1 号左边即可**）。
3. 观察控制台：

```
[PuzzleBoard] swap pid=0@4 ↔ pid=3@1
[PuzzleBoard] merge happened — 当前 8 组，最大组 2 块
```

`8 组` = 9 块里 2 块合成 1 组、剩下 7 块各自 1 组 → 7+1 = 8 组。`最大组 2 块` = 那对合并了的对子。

4. 继续操作把更多块凑在一起。每次合并都看到 `merge happened` 和组数下降。

5. 把已经合并的对子又"拖开"（swap 走其中一块）→ 控制台**不打印 merge happened**（mergedAny=false），但下次再凑回去会重新合上。

**控制台预期**（一盘游戏完整流程）：

```
[PuzzleBoard] 洗牌完成 — pieceCount=9, misplaced=8, attempts=1
... (没有初始 merge，因为 9 块都没相对正确)
[PuzzleBoard] swap pid=0@4 ↔ pid=3@1
[PuzzleBoard] merge happened — 当前 8 组，最大组 2 块
[PuzzleBoard] swap pid=4@7 ↔ pid=2@3
[PuzzleBoard] merge happened — 当前 7 组，最大组 3 块
...
```

**不正常的信号**：

- **永远不 merge**：Rule B 写错——把 `rPiece.col === piece.col + 1` 写成 `rPiece.col === piece.col`（不加 1）→ 这是 Rule A 的"必须在自己正确槽" 而不是"右邻居关系"。看条件式。
- **乱合一通（不该合的也合）**：union 调用条件没过滤——比如忘加 `if (... rPiece.col === piece.col + 1)` 直接 union。
- **`pieceNodes[rPid]` 拿到 null**：这又是"slots 数据被破坏" 的征兆。检查 swap 写 slots 的顺序。
- **mergeScan 在 onLoad 里跑出错（pieceNodes 没建满）**：检查 onLoad 顺序——`createPieces` → `layoutAllPieces` → `mergeScan`。第三步前两步必须先完成。
- **groupId 老不变（每次 swap 都打 9 组）**：mergeScan 的最后一段"写回 groupId" 没跑或跑了但没生效。**Cocos 的 Component 字段直接赋值是同步的**，不需要 set 调用。检查 piece.groupId 那一行。
- **同样的 swap 两次，第二次没打 merge log**：正常——第二次 swap 后 parents 重置，但合的还是同一对，第一对的 union 因为 ra === rb 返回 false，mergedAny 也是 false。**`merge happened` 的意思是"本轮新建了组"，不是"现在有合并状态"**。

## 6. 复盘：这节教了什么

**1. Rule B = 相对位置 = 真的好品味**

Rule A 和 Rule B 看起来都对——但**用户体验天差地别**。

- Rule A："必须把每块放回正确槽才合"——玩家拼一块就要回头看"我以前拼好的那两块还在原位吗"，**焦虑爆棚**。
- Rule B："只要相对正确就合"——玩家可以"先拼一个 2×2 小块，再整体挪到正确位置"，**自然轻松**。

商业拼图游戏几乎都是 Rule B。Rule A 是"看起来逻辑严谨" 但**违背玩家心智** 的反例。

写算法时永远要问："我这条规则**符合用户的心智模型**吗？"

**2. 并查集是"合并集合" 的正解**

**N 个集合反复合并** + **查询同集合** 这类问题，并查集近乎 O(1)。背下来这 10 行模板，一辈子受用。

不光拼图——以下场景全靠它：

- **网络连通性**：Kruskal 最小生成树
- **图像分割**：连通域 4-邻域 / 8-邻域聚合
- **社交网络**：推荐"你可能认识的人"
- **物理模拟**：碰撞分组（同一个刚体组件树）

记住关键技巧：

- **路径压缩** (`parents[x] = parents[parents[x]]`)：让链变扁平。
- **按秩合并**（更老练）：合并时让小树挂大树下。教学版不用——3×3 规模无所谓。

**3. 全量扫描 = 笨办法 = 简单办法**

每次 swap 后**重新扫整盘**——9 次 piece 比较、9 次 union 调用。CPU 不痛不痒。

如果改成"增量更新"：

- swap 影响哪两块？被拖块 + 被替换块。
- 这两块原来的同组邻居怎么办？要把组拆开吗？
- 新位置的邻居要合吗？
- ...几十种 case 写不完，bug 一抓一大把。

**全量扫规避一切边界问题**。Linus："**Bad programmers worry about the code. Good programmers worry about data structures.**" —— 全量扫 + 重置 parents 这个数据结构选择，省掉了 90% 的边界判断。

什么时候不能用全量扫？**N 大到全量扫一帧跑不完**。3×3 = 9 是 36 次 union 调用，1ms 不到。8×8 = 64 是 224 次，仍然不到 1ms。**几千块以下都没问题**。

**4. 本节"不改视觉" 是有意为之**

很多教程会把"合并自动调整位置"（让两块紧贴）也放在 05 节，但我们故意拆开：

- **05 节** = 算 groupId（数据层）
- **06 节** = 用 groupId 让"拖一块带一组"（视觉/输入层）

**先把数据搞对，再让数据驱动视觉**——分两节做，每节只改一类东西，**调试容易、教学清晰**。

如果你 05 节调出"groupId 算错"，能立刻定位是 mergeScan 算法 bug。如果一锅炖，"组没合上" 可能是算法 bug、可能是位置算错、可能是事件没触发——**多变量一起改 = 调试地狱**。

## 7. 常见坑

- **parents 数组初始化遗漏**：mergeScan 每次都要从头建 parents——`for (let pid = 0; pid < pieceCount; pid++) parents.push(pid);`。如果改成 `parents = []` 后忘了 push，所有 union 都对空数组操作，访问 undefined 报错。
- **`find` 路径压缩写错**：`parents[x] = parents[parents[x]]` 是**让 x 跳过父节点直连爷节点**，循环里 x 跟着上跳——压缩力度温和但永不破坏正确性。常见错写：
  - `parents[x] = x`（直接指向自己，把 x 变成新根，破坏整个并查集）
  - 漏掉这一行（不压缩，能跑但慢，3×3 规模看不出）
- **Rule B 的"上下左右"写错方向**：右邻居关系是 `nb.col === self.col + 1`、下邻居关系是 `nb.row === self.row + 1`（注意 row+1 是"下方"，因为 row 从上到下增长）。写反了 = 永远合不上。**强行 console.log 一下 piece.row / piece.col / 邻居.row / 邻居.col 看一下数据流**就清楚了。
- **`pieceNodes[pid].getComponent(PuzzlePiece)!` 返回 null**：理论不可能，01 节代码里 createPieces 给每块都 addComponent 了。如果真返回 null，要么 createPieces 漏了（验证：`console.log(this.pieceNodes[0].components.map(c => c.constructor.name))`），要么 PuzzlePiece import 路径不对。
- **mergeScan 内部反复 getComponent 性能炸了**：64 块时一次 mergeScan 调 ~150 次 `getComponent`——Cocos 的 getComponent 内部是字符串匹配 + 数组扫描。如果你担心性能，可以缓存：
  ```typescript
  private pieceComps: PuzzlePiece[] = [];
  // createPieces 末尾：this.pieceComps.push(piece);
  // mergeScan 内部用 this.pieceComps[pid] 替代 getComponent
  ```
  3×3 不需要做。8×8 也不需要做。100×100 时再说。
- **forgot 在 onLoad 里调 mergeScan**：开局洗牌后偶尔会"两块刚好相对正确"——需要立刻合。如果只在 swap 后 mergeScan，开局这种情况会拖着"虚假的不合并状态" 直到玩家第一次 swap。
- **mergedAny 用法误会**：`merged === true` 不是"现在有合并状态"，是"本轮 mergeScan 中产生了**新的** union 操作"。组没变 → false；组从 9 个降到 8 个 → true；下次 swap 后组还是 8 个（同一对维持）→ false。

## 8. 下一节预告

groupId 算对了，但**玩家完全感觉不到** —— 拖一块只动那一块，跟 04 节没区别。

**06 节**：让 groupId 真正起作用。

- **拖一块 = 拖整组**：DragMove 把同 groupId 的所有块一起按 UIDelta 挪。

- **被挤块回填**：被组员目标位置占据的非组员，反向填到组原占位的空槽。
