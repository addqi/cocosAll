# 04 · 松手即换 — 命中槽位的两两交换

> **目标**：拖一块到另一块的位置 → 松手 → 两块**交换位置**。拖到 3×3 外面 → 弹回原位不交换。
>
> 完成后玩家**真的能玩**——一步步把每块挪到正确位置就能赢（虽然胜利反馈在 07 节才接）。

---

## 起点回顾

03 节末态：

- `onPieceTouchMove` 让块跟手指走（累加 UIDelta）。
- `onPieceTouchEnd` 一律弹回原槽（`slots.indexOf(pid)` → `slotToPosition`）。
- 9 块开局乱序、能拖、松手回原位——**没法拼**。

本节 `onPieceTouchEnd` 改成 3 分支：

1. 落点越界（手指松开时块的中心在 3×3 外）→ 弹回。
2. 落点 = 原槽 → 弹回（等于没动）。
3. 落点 = 别的槽 → swap：被拖块去落点槽，原住民来源槽。

---

## 1. 关键概念：指针 → 槽位反算

我们已经有了 `slotToPosition(slotIdx) → {x, y}`（01 节）。本节需要它的**反函数**：`pointerToSlot(x, y) → slotIdx`。

数学很简单。`slotToPosition` 是：

```text
x = (sc - center) * pieceDisplay
y = (center - sr) * pieceDisplay
```

反过来：

```text
sc = round(x / pieceDisplay + center)
sr = round(center - y / pieceDisplay)
slotIdx = sr * pieceGrid + sc
```

`round` 而不是 `floor`：因为 anchor 在中心，块中心稍微偏左（比如 x=-90）应该归到中间格 (1)，而不是左边格 (0)。

**越界检查**：`sr / sc` 不在 `[0, pieceGrid)` 范围 → 落点在网格外，返回 -1。

## 2. 改 `PuzzleBoard.ts`：加 `pointerToSlot` + 改 `onPieceTouchEnd`

加新私有方法：

```typescript
/**
 * 把屏幕坐标反算成槽位下标。
 *
 * 输入：被拖块当前的 Board-local 坐标（即 node.position）。
 *      因为 PuzzleRoot 在 Canvas 中心 = (0, 0)，所以 node.position 已经是 board-local。
 * 输出：0..pieceCount-1 = 命中某槽；-1 = 落点在网格外。
 *
 * 用 round 而不是 floor —— 块的 anchor 在中心，
 * 块中心落在格子中央时应归到该格，而不是左下角偏移。
 */
private pointerToSlot(boardLocalX: number, boardLocalY: number): number {
    const center = (this.pieceGrid - 1) / 2;
    const sc = Math.round(boardLocalX / this.pieceDisplay + center);
    const sr = Math.round(center - boardLocalY / this.pieceDisplay);
    if (sr < 0 || sr >= this.pieceGrid || sc < 0 || sc >= this.pieceGrid) return -1;
    return sr * this.pieceGrid + sc;
}
```

改 `onPieceTouchEnd`：

```typescript
private onPieceTouchEnd(event: EventTouch): void {
    const node = event.target as Node;
    const piece = node.getComponent(PuzzlePiece)!;
    const srcIdx = this.slots.indexOf(piece.pieceId);
    const dstIdx = this.pointerToSlot(node.position.x, node.position.y);

    // 分支 1 + 2 合并：越界或回原槽 → 弹回
    if (dstIdx < 0 || dstIdx === srcIdx) {
        const { x, y } = this.slotToPosition(srcIdx);
        node.setPosition(x, y, 0);
        console.log(`[PuzzleBoard] piece=${piece.pieceId} 弹回 slot=${srcIdx} (dst=${dstIdx})`);
        return;
    }

    // 分支 3: swap
    const dstPid = this.slots[dstIdx];
    const dstNode = this.pieceNodes[dstPid];

    this.slots[srcIdx] = dstPid;
    this.slots[dstIdx] = piece.pieceId;

    const srcPos = this.slotToPosition(srcIdx);
    const dstPos = this.slotToPosition(dstIdx);
    node.setPosition(dstPos.x, dstPos.y, 0);
    dstNode.setPosition(srcPos.x, srcPos.y, 0);

    console.log(
        `[PuzzleBoard] swap pid=${piece.pieceId}@${srcIdx} ↔ pid=${dstPid}@${dstIdx}`,
    );
}
```

## 3. 跑起来

▶ 预览。

**测试动作**：

1. 拖 0 号块到 4 号槽（中央）的位置 → 松手 → 两块交换，**0 号块进入中央，原中央那块去 0 号位置**。
2. 拖块到屏幕外松手 → 弹回原位。
3. 拖块到自己原位置（拖动幅度小、松手时还在原槽中央）→ 等同弹回，无交换。
4. 反复操作把 9 块**手动拼回正确位置**。每块拼对了视觉上没特殊反馈（07 节才有胜利判定），但你能感到"已经拼好了"。

**控制台预期**：

```
[PuzzleBoard] swap pid=0@3 ↔ pid=7@4
[PuzzleBoard] piece=2 弹回 slot=5 (dst=-1)         ← 拖出去
[PuzzleBoard] piece=4 弹回 slot=2 (dst=2)          ← 拖回原位
```

**不正常的信号**：

- **拖到别处也弹回原位（永远不 swap）**：`pointerToSlot` 返回 -1。多半是 round 写成 floor 了——块中心稍微偏一点就被划归到下一格，常见于"block 拖到目标槽中央但还是弹回"。
- **swap 后两块位置错了**：`slots[srcIdx]` 和 `slots[dstIdx]` 写入顺序错——必须**先存** `dstPid = this.slots[dstIdx]`，再写 `slots[srcIdx] = dstPid`，否则后写的会覆盖。看代码顺序：先取 dstPid → 写 srcIdx → 写 dstIdx。
- **swap 后控制台 log 跟视觉对不上**：`slots.indexOf(piece.pieceId)` 在 swap 后返回的就是 dstIdx 了——log 在 swap **之前**算 srcIdx。看代码顺序。
- **块挪到目标位置但**目标块**没动（堆叠）**：`dstNode.setPosition(...)` 没执行。检查那一行。
- **AreaCheck 失误（拖到边缘外但 dstIdx 不是 -1）**：你拖动幅度小、块中心还没真正出网格，`pointerToSlot` 算出来还在 [0..2] 内——这不是 bug，是设计：游戏规则就是"块中心在哪格就归哪格"。要更严格的"边缘外即取消"，可以加 buffer：`if (sr < 0 || sr >= pieceGrid + 1.0)` 之类。教学版不做。

## 4. 复盘：这节教了什么

**1. 数据驱动的渲染**

swap 流程：

```text
1. 算 srcIdx（slots.indexOf）
2. 算 dstIdx（pointerToSlot）
3. 改 slots（数据层 swap）
4. 改 position（视觉层 swap）
```

**先改数据，再改视觉**——这是数据驱动开发的铁律。

为什么不能反过来？想象 step 4 失败（比如 setPosition 抛异常）：

- 先改数据再改视觉：数据已是新状态，视觉还旧 → 一帧后某 layout 调用会按数据重摆，自动同步。
- 先改视觉再改数据：视觉新、数据旧 → 后续任何"重新 layout" 操作会按旧数据把视觉**改回旧状态** → 玩家看到"swap 完又回来了"。

**永远先改真相源（slots），再算投影（position）**。这跟 React/Vue 那套"数据驱动 UI" 是一个道理。

**2. 单块 swap 是 06 节"整组 swap" 的退化**

本节代码：

```typescript
this.slots[srcIdx] = dstPid;
this.slots[dstIdx] = piece.pieceId;
```

06 节会把这扩成：N 个组员从 srcSlots 到 dstSlots，N+M 块同时挪。但**算法骨架一致**——都是"先算路径、再写 slots、再改视觉"。

**N=1 是 N=k 的特例**——这是 06 节算法的核心好处：**一份代码覆盖单块和整组**，不写两套。

记住这句话："**N=1 时整组循环退化成单块 swap**"。06 节会反复出现。

**3. round 还是 floor，是命中算法的性格**

```typescript
sc = Math.round(x / pieceDisplay + center);
```

用 round → "块中心落在格子的哪边就归哪边"——边界友好，玩家拖到差不多就行。

如果用 floor → "块中心落在格子的左下角才归这格"——拖到中心偏右下才命中，**用户感觉很难拖准**。

**round vs floor 的差别看似一行代码、其实是两种用户体验**。游戏开发里这种"细微但要命" 的选择无数：

- 长按多久算长按？250ms / 500ms / 1s 是不同体验。
- 双击间隔阈值多少？300ms / 500ms。
- 滑动多少像素算"开始拖动"？4px / 8px / 16px。

**没有正确答案，只有"你的用户喜欢哪个"**。教学版用 `round` 是因为它对玩家最宽容。

**4. 三种"无操作" 的统一处理**

```typescript
if (dstIdx < 0 || dstIdx === srcIdx) {
    // 越界 + 回原槽 都走"弹回" 分支
}
```

把"越界"和"原地不动" 用 `||` 合并到一个分支——它们的**结果一样**（弹回），写两个 if-else 是冗余。

```typescript
// 反例：把它们拆开
if (dstIdx < 0) {
    // 越界，弹回
} else if (dstIdx === srcIdx) {
    // 原地，弹回
} else {
    // swap
}
```

代码长 2 倍、概念多 1 个（两种"弹回"是不是一样的？）、可读性更差。

Linus："**好代码没有特殊情况**"——能合并的分支就合并。

## 5. 常见坑

- **`Math.round(0.5) === 1`，但 `Math.round(-0.5) === 0`（不是 -1）**：JavaScript 的 round 用"四舍五入到偶数" 是不严格的——它对 .5 的处理是"向 +∞ 取整"。本游戏的 boardLocal 坐标范围 `[-1, 1] * pieceDisplay`，round 几乎不会遇到 .5 边界，**没事**。但写算法时知道一下。
- **`this.slots[dstIdx] = piece.pieceId` 顺序写反了**：先写 srcIdx（被拖块新位置）再写 dstIdx（被替换块新位置）。如果你写反，dstPid 已经被 piece.pieceId 覆盖了，再 `slots[srcIdx] = dstPid` 就是用 piece.pieceId 自身覆盖——swap 失效。本节代码是先**取出** dstPid 再**写两个槽**——顺序对的。
- **`pieceNodes[dstPid]` 拿到 null**：理论不可能，因为 `pieceNodes` 是 createPieces 里建满的。但如果你把 createPieces 改成"按 row 顺序加" 而忘了 push，下标就错位了。检查 push 的顺序。
- **swap 后立刻能再拖动同一块**：可以——本节没加输入锁。06 节起会加 0.2s 缓动锁。
- **拖到一半被另一个手指/鼠标干扰**：本节没多指处理，理论上 Cocos 默认单指交互，没问题。如果你的手机不小心多指误触，可能产生多个并发拖动——多指处理超出本教学链。
- **`event.target` 实际是 `node` 还是 `currentTarget`**：注册回调时 `node.on(..., handler, this)`，回调里 `event.target` 是命中节点 = node 本身。如果你的层级里有"事件冒泡"（比如 PieceLayer 也注册了 TOUCH_END），target 仍然是命中那块——冒泡时 currentTarget 才是冒泡到的层。**单层注册时两者一致**。

## 6. 下一节预告

能 swap 了——但有个怪现象：

**你把 0 号块拖到 1 号块旁边（**正确相邻位置**）→ 它们应该"自动合并" 不再拆开**。但本节里它们仍然各管各的——下次你拖 0 号块 1 号块就会被你撞开。

**05 节**：自动合并！

- **并查集 (Union-Find)** 数据结构：10 行模板。
- **mergeScan 全量扫描**：每次 swap 后扫一遍，找到"相对位置正确"的相邻对，合到同一组（groupId）。
- **Rule B**：判合并不要求"块在正确槽位"，只要求"两块的相对位置正确"——这是拼图玩法的微妙细节。

完成后两块只要"挨在一起 + 相对方向对" 就**永远在一起**，下节起拖一块 = 拖整组。
