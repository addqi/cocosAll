# 拼图游戏从零到一 · Cocos Creator 3.8 + TypeScript

> 这是一份**通用版**教程：用最常见的引擎（Cocos Creator 3.8）和最常见的语言（TypeScript），从空场景写到一个可玩的 3×3 拼图。
>
> 跟我们仓库里 `puzzle-docs/` 那份是同一个游戏，只是那份用的是公司内部 ECS 框架 AtomEngine。这份去掉 AtomEngine，换成 Cocos 原生 Component + Node API，**任何懂一点 TS 的人都能跟着走完**。

---

## 适合谁读

- 用 Cocos Creator 3.8 写过 Hello World，会创建场景、贴一张图。
- 知道 TypeScript 的 `class` / `interface` / `Array<T>`。
- **不要求懂 ECS / 状态机 / 并查集**——会一边写一边讲，用最笨最直白的实现。

---

## 教学链规划

| 节 | 主题 | 核心知识点 | 大约代码量 |
|---|------|-----------|----------|
| **00** | 项目初始化与一张完整图 | 空场景 + 加载本地 PNG → 整图渲染到屏幕中央 | ~50 行 |
| **01** | 一张图切成 3×3 = 9 块 | `SpriteFrame.rect`、9 个 Node 定位、数据结构 `slots: number[]` | ~120 行 |
| **02** | Fisher–Yates 洗牌 | "位置是 slots 的投影"、最少错位重洗 | ~50 行 |
| **03** | 拖起来 — 按住跟手与松手弹回 | `touchstart/move/end`、节点局部坐标、弹回 | ~100 行 |
| **04** | 松手即换 — 命中槽位的两两交换 | "屏幕坐标 → 槽位下标"、双块 swap 缓动 | ~80 行 |
| **05** | 相邻自动合并 — 并查集 + Rule B | 相对位置正确、`groupId`、union-find | ~100 行 |
| **06** | 整组拖动 — 多换多 + 空位填充 | 按 `groupId` 整体搬运、`displaceFrom/fillTo` 集合差 | ~150 行 |
| **07** | 胜利判定 — 缓动 / 输入锁 / 状态机 | "胜利 = groupId 全相同"、计时、终止态锁 | ~80 行 |
| **08** | Shader 自绘边框与真合批 | 自定义 effect、共享 SpriteFrame/Material、顶点色编码身份、方向感知 SDF | ~150 行 + 1 个 .effect |

**完成后你有的东西**：

- 一个能拖、能合、能赢、**10×10 也能 1~2 DC 跑**的拼图，主控类 ~850 行 TS + 1 个 ~180 行 .effect。
- 9 个独立可读的概念：切图、洗牌、拖拽、命中、合并、整组、胜利、状态机、shader 真合批。
- 每个概念的"为什么这么写"——很多教程只教"怎么写"，这份花一半篇幅在"为什么不那样写"。

## 行文约定

每一节都是这个结构：

1. **目标**：一句话说本节做完应该看到什么。
2. **起点回顾**：上节结束的状态。
3. **正文**：拆成 3-5 个小步骤，每步给完整代码。
4. **跑起来**：开 Cocos Creator → 拖到场景 → 看现象。
5. **复盘**：本节教了什么核心概念。
6. **常见坑**：我（作者）写这版时踩过的坑，按出现频率排。
7. **下一节预告**：本节遗留什么、下节补什么。

代码风格：

- **严格 TypeScript**：所有字段都加类型，不写 `any`。
- **`const` 优先**：能 const 不写 let。
- **不抽 helper**：教学版禁止"private 方法"。如果同一段逻辑出现两次，就**老老实实复制粘贴**，因为：教学链每节都有"对比"——这一节的实现 vs 上一节的，复制粘贴比"改一下抽出来的 helper" 更容易看清楚演化。
- **注释只解释 why，不解释 what**：`x = 1; // 把 x 设为 1` 这种废话注释一律删除。

## 项目准备

> 详细步骤见 `00-项目初始化与一张完整图.md`，这里只列依赖。

- **Cocos Creator 3.8.x**（任何 3.8 patch 版都行，3.8.0 / 3.8.4 / 3.8.6 都测试过）
- **TypeScript 4.7+**（Creator 自带）
- **一张 PNG 图**：尺寸至少 300×300，**正方形最佳**（不是正方形也行，01 节会按 min 边切方）。把它丢进 `assets/textures/` 即可。

完整目录结构（跟着教程一节节加）：

```
assets/
├── scenes/
│   └── puzzle.scene             # 主场景（00 节建）
├── scripts/
│   └── puzzle/
│       ├── PuzzleBoard.ts       # 主控（00 节建，每节扩展）
│       ├── PuzzlePiece.ts       # 单块控制（01 节建）
│       └── UnionFind.ts         # 并查集工具（05 节建）
└── textures/
    └── kitten.png               # 你的拼图源图
```

## 与 AtomEngine 版的差异

如果你已经看过 `puzzle-docs/` 那版，会发现这两份的"算法部分"几乎逐行对应——洗牌、并查集、Rule B、整体搬运的 displaceFrom/fillTo，**核心逻辑一字不差**。差异只在"代码壳"：

| 维度 | AtomEngine 版 | Cocos 3.8 版 |
|------|---------------|--------------|
| 数据存哪 | EntityComponentAtom（按字段批量数组） | `class` 实例字段 |
| 触发 | SignalAtom + LogicAtom | `node.on('touch...')` |
| 拖拽逻辑 | DragBegin/Move/End 三个独立 LogicAtom | `onLoad` 里一处 `node.on('touchstart')` 等 |
| 合并扫描 | FunctionAtom 单独一个文件 | `class` 私有方法 |
| 上下文 | 一切原子皆 `@bindAtom` 注入 | 直接 `this.xxxx` |
| 文件数 | ~22 个 .ts | ~3 个 .ts |
| 代码行数 | ~1200 | ~700 |

**为什么 AtomEngine 版行数多**：每一个"职责"被拆成独立 Atom，各自带类型定义、绑定声明、独立类——是工程化的代价。3×3 拼图本身的算法只 ~700 行，剩下 500 行是框架样板。

**为什么我们公司选 AtomEngine**：百万行规模下，"职责清晰拆分" + "可测试" + "可热替换"的优势就显出来了。但**学拼图算法本身**，本份通用版反而更直接。

## 反馈

如果你写到某一节卡住了：

1. 先看那节的"常见坑"。
2. 再看 `puzzle-docs/` 同号节的"代码事实对照"——也许 AtomEngine 版踩过同一个坑、解法可以反向迁移到 Cocos 版。
3. 还是不行，把你卡住的步骤截图加问题告诉作者。

**祝你写代码愉快。** 拼图游戏写完，你也"通"了一半的小游戏开发——拖放、合并、网格、状态机，这些都是后续游戏（消除、塔防、roguelike）的通用底座。
