# 06 · 整组拖动 — 多换多与空位填充

> **目标**：拖一块 = 拖整组（同 groupId 全部一起动）。松手时按"组首块的目标槽" 整体搬运，**被挤掉的非组员反向填到组原占位的空槽**。
>
> 完成后玩家可以"先拼出 2×2 小块、再整体平移到正确位置"——拼图玩法**完整成型**。

---

## 起点回顾

05 节末态：

- mergeScan 让"相对正确" 的相邻块共享 groupId。
- groupId 已算对，但**没人用它** —— 拖动还是单块行为。

本节升级 3 个回调：

- `onPieceTouchStart`：把整组提到 z-order 最前（拖动期间不被别块遮挡）。
- `onPieceTouchMove`：所有同 groupId 的块**一起** 按 UIDelta 挪。
- `onPieceTouchEnd`：算"组目标槽位" → 用 `displaceFrom / fillTo` 集合差**多换多 + 空位填充**。

---

## 1. 关键概念：被拖块 vs 整组

每次 touchstart 时，"被点中的块" 是**领头块**。同 groupId 的其他块叫**组员**。

```typescript
const dragPid = piece.pieceId;
const dragGroupId = piece.groupId;
const groupPids: number[] = []; // 含 dragPid
for (let pid = 0; pid < this.pieceCount; pid++) {
    if (this.pieceNodes[pid].getComponent(PuzzlePiece)!.groupId === dragGroupId) {
        groupPids.push(pid);
    }
}
```

`groupPids` 永远 ≥ 1（至少包含 dragPid 自己）。N=1 是单块拖动的退化——本节代码统一处理 N=1 和 N>1。**这就是 04 节复盘说的"N=1 是 N=k 的特例"**。

**关键设计**：`groupPids` **不存为字段**。它是 touchstart 时根据当前 groupId 现扫的临时变量——但拖动期间保持不变（在 touchend 之前 mergeScan 不会重跑、groupId 不变）。

如果 mergeScan 在拖动期间跑了（比如本节没遵守约定）就 bug——**严格执行"mergeScan 只在 swap 提交后调一次"** 的纪律是关键。

---

## 2. 关键概念：组的目标槽位

单块 swap 时，dst 是"指针落点对应的槽"。整组时呢？

**最直觉做法**：领头块的指针落点 = 领头块的目标槽。其他组员按"和领头块的相对偏移" 摆。

```text
领头块 dragPid 当前在槽 srcDragIdx。
领头块松手时指针落在槽 dstDragIdx。
偏移量 = dstDragIdx - srcDragIdx（在 row/col 维度上）。
其他组员目标槽 = 它们当前槽 + 同样偏移。
```

例：组员当前 slots = [3, 7]（领头 pid=3 在槽 0、组员 pid=7 在槽 1），领头落在槽 5。偏移 = 5 - 0 = 5。组员目标槽 = 1 + 5 = 6。

但要小心：**槽下标是一维的，row/col 才能直接加偏移**。所以要拆 row/col 算：

```typescript
const dragSrcSlot = this.slots.indexOf(dragPid);
const dragDstSlot = this.pointerToSlot(dragNode.position.x, dragNode.position.y);

const dragSrcRow = Math.floor(dragSrcSlot / this.pieceGrid);
const dragSrcCol = dragSrcSlot % this.pieceGrid;
const dragDstRow = Math.floor(dragDstSlot / this.pieceGrid);
const dragDstCol = dragDstSlot % this.pieceGrid;

const dRow = dragDstRow - dragSrcRow;
const dCol = dragDstCol - dragSrcCol;

// 每个组员 newSlot = (oldRow + dRow) * pieceGrid + (oldCol + dCol)
```

如果有任何组员的 newRow / newCol 落到了 [0, pieceGrid) 之外 → 整组弹回，不搬运。

## 3. 关键概念：displaceFrom / fillTo（集合差消除特殊情况）

整组搬运时：

- 组员从原槽集合 `srcSlots` 搬到目标槽集合 `dstSlots`。
- `dstSlots` 上原本有非组员（被挤者）。这些"被挤者" 应该填到哪里去？

**最朴素做法**：把被挤者塞到 srcSlots 里"反向 swap"。

但**有个边界情况**：如果 srcSlots ∩ dstSlots ≠ ∅（搬运距离短，新旧位置重叠），怎么办？比如 2×1 横条从槽 [0, 1] 搬到槽 [1, 2]：

```text
srcSlots = {0, 1}
dstSlots = {1, 2}
重叠 = {1}（这个槽既是源又是目标）
```

朴素做法会把 dstSlots 的所有原住民填到 srcSlots：槽 1 的非组员？没非组员，槽 1 上是组员自己——没人要被挤。槽 2 的非组员要填到 srcSlots 的某个位置——填槽 0 还是槽 1？槽 1 现在是组员位置不能填——**case 一堆**。

**集合差解法**：

```typescript
displaceFrom = dstSlots \ srcSlots = {2}    // 真正"被挤" 的槽：dst 上不在 src 的部分
fillTo       = srcSlots \ dstSlots = {0}    // 真正"空出来" 的槽：src 上不在 dst 的部分
```

`displaceFrom.length === fillTo.length` —— 这是数学保证（两组对称差等长）。一一对应：把 displaceFrom 上的非组员搬到 fillTo 的对应位置。

```text
displaceFrom = [2]
fillTo       = [0]
槽 2 上的非组员搬到槽 0。完成。
```

**没有"重叠"特殊情况了**——重叠部分自动从两侧消掉。这就是"**消除特殊情况**" 的实战。

```typescript
function diff<T>(a: T[], b: T[]): T[] {
    return a.filter(x => !b.includes(x));
}

const displaceFrom = diff(dstSlots, srcSlots);
const fillTo = diff(srcSlots, dstSlots);
```

3×3 / 8×8 规模 includes 性能完全够用。N 大时换 Set。

## 4. 关键概念：z-order（拖动期间整组提到最前）

拖动一组时，组员要**视觉上覆盖** 网格里别的块——否则拖一组到中央会看到组员被中央的非组员挡住。

Cocos 的 z-order 机制：**同父节点下，sibling 列表后面的覆盖前面的**。提到最前 = `node.setSiblingIndex(99999)` 或循环把整组每块都 setSiblingIndex 到末尾。

```typescript
// 拖动开始：整组提到末尾
for (const pid of groupPids) {
    this.pieceNodes[pid].setSiblingIndex(this.pieceNodes[pid].parent!.children.length - 1);
}
```

注意 `setSiblingIndex` 是**绝对索引** —— 同一帧多次调用会让它们在末尾重新排，最后调的那个真在最末尾。如果想保持组员之间的相对顺序，要按从小到大依次 setSiblingIndex：循环跑完后顺序就是 groupPids 的入参顺序。**3×3 / 8×8 规模无所谓哪个组员在最末尾**——只要整组都在网格外其他块**之后**就行，被组员之间互相遮挡用户也看不出。

`onPieceTouchEnd` 不需要把 z-order 改回去——下次 layoutAllPieces 不动 sibling 顺序，只改 position。新拖动的组会自动覆盖前面留下的 z-order，**永远是"刚拖的最前"**——这跟用户习惯一致。

## 5. 改 `PuzzleBoard.ts`：升级 3 个回调

**改 `onPieceTouchStart`**（之前是空实现）：

```typescript
private onPieceTouchStart(event: EventTouch): void {
    const node = event.target as Node;
    const piece = node.getComponent(PuzzlePiece)!;
    const groupId = piece.groupId;

    // 整组提到 sibling 末尾，拖动期间不被别块遮挡
    for (let pid = 0; pid < this.pieceCount; pid++) {
        const pNode = this.pieceNodes[pid];
        if (pNode.getComponent(PuzzlePiece)!.groupId === groupId) {
            pNode.setSiblingIndex(pNode.parent!.children.length - 1);
        }
    }
}
```

**改 `onPieceTouchMove`**（按整组同步挪）：

```typescript
private onPieceTouchMove(event: EventTouch): void {
    const node = event.target as Node;
    const piece = node.getComponent(PuzzlePiece)!;
    const groupId = piece.groupId;
    const delta = event.getUIDelta();

    for (let pid = 0; pid < this.pieceCount; pid++) {
        const pNode = this.pieceNodes[pid];
        if (pNode.getComponent(PuzzlePiece)!.groupId === groupId) {
            const pos = pNode.position;
            pNode.setPosition(pos.x + delta.x, pos.y + delta.y, pos.z);
        }
    }
}
```

**改 `onPieceTouchEnd`**（整组搬运 + 集合差填充）：

```typescript
private onPieceTouchEnd(event: EventTouch): void {
    const node = event.target as Node;
    const piece = node.getComponent(PuzzlePiece)!;
    const dragPid = piece.pieceId;
    const dragGroupId = piece.groupId;

    // 收集组员
    const groupPids: number[] = [];
    for (let pid = 0; pid < this.pieceCount; pid++) {
        if (this.pieceNodes[pid].getComponent(PuzzlePiece)!.groupId === dragGroupId) {
            groupPids.push(pid);
        }
    }

    // 算偏移
    const dragSrcSlot = this.slots.indexOf(dragPid);
    const dragDstSlot = this.pointerToSlot(node.position.x, node.position.y);

    if (dragDstSlot < 0) {
        this.snapBackGroup(groupPids);
        return;
    }

    const dRow = Math.floor(dragDstSlot / this.pieceGrid) - Math.floor(dragSrcSlot / this.pieceGrid);
    const dCol = (dragDstSlot % this.pieceGrid) - (dragSrcSlot % this.pieceGrid);

    // 算所有组员的源/目标槽，越界检查
    const srcSlots: number[] = [];
    const dstSlots: number[] = [];
    for (const pid of groupPids) {
        const srcSlot = this.slots.indexOf(pid);
        const srcRow = Math.floor(srcSlot / this.pieceGrid);
        const srcCol = srcSlot % this.pieceGrid;
        const dstRow = srcRow + dRow;
        const dstCol = srcCol + dCol;
        if (dstRow < 0 || dstRow >= this.pieceGrid || dstCol < 0 || dstCol >= this.pieceGrid) {
            this.snapBackGroup(groupPids);
            return;
        }
        srcSlots.push(srcSlot);
        dstSlots.push(dstRow * this.pieceGrid + dstCol);
    }

    // 集合差：被挤的槽 / 空出来的槽
    const displaceFrom = dstSlots.filter(s => !srcSlots.includes(s));
    const fillTo = srcSlots.filter(s => !dstSlots.includes(s));

    // 写新 slots：先复制再修改（避免读 / 写交叉）
    const newSlots = this.slots.slice();
    for (let i = 0; i < groupPids.length; i++) {
        newSlots[dstSlots[i]] = groupPids[i];
    }
    for (let i = 0; i < displaceFrom.length; i++) {
        // 注意：被挤的非组员从 OLD slots 读、写到 newSlots 的空槽
        newSlots[fillTo[i]] = this.slots[displaceFrom[i]];
    }
    this.slots = newSlots;

    // 重新摆位：组员 + 被挤者
    for (const pid of groupPids) {
        const slot = this.slots.indexOf(pid);
        const { x, y } = this.slotToPosition(slot);
        this.pieceNodes[pid].setPosition(x, y, 0);
    }
    for (let i = 0; i < displaceFrom.length; i++) {
        const displacedPid = newSlots[fillTo[i]];
        const { x, y } = this.slotToPosition(fillTo[i]);
        this.pieceNodes[displacedPid].setPosition(x, y, 0);
    }

    console.log(
        `[PuzzleBoard] 整组搬运 — group=${dragGroupId}, ` +
        `groupPids=[${groupPids}], dRow=${dRow}, dCol=${dCol}, ` +
        `displaceFrom=[${displaceFrom}], fillTo=[${fillTo}]`,
    );

    // 合并扫描
    const merged = this.mergeScan();
    if (merged) {
        const groupCounts = this.countGroups();
        console.log(
            `[PuzzleBoard] merge happened — 当前 ${groupCounts.length} 组，` +
            `最大组 ${Math.max(...groupCounts)} 块`,
        );
    }
}

/** 整组弹回原槽 */
private snapBackGroup(groupPids: number[]): void {
    for (const pid of groupPids) {
        const slot = this.slots.indexOf(pid);
        const { x, y } = this.slotToPosition(slot);
        this.pieceNodes[pid].setPosition(x, y, 0);
    }
}
```

## 6. 跑起来

▶ 预览。

**测试动作**：

1. 把 0、1 号块凑成横向相邻 → mergeScan 合成一组。
2. 拖任意一块（0 或 1）→ **两块同步跟手**。
3. 拖到中央位置松手 → 两块整体到中央，**原中央那两块反向回填到组的原位置**。
4. 拖到边缘外松手 → 整组弹回原位（**snapBackGroup**）。
5. 多次操作直到拼好。

**控制台预期**：

```
[PuzzleBoard] 整组搬运 — group=0, groupPids=[0,1], dRow=1, dCol=0, displaceFrom=[3,4], fillTo=[0,1]
[PuzzleBoard] merge happened — 当前 6 组，最大组 4 块
```

`displaceFrom=[3,4]` 表示槽 3、4 上的非组员被挤；`fillTo=[0,1]` 表示它们填到槽 0、1。

**部分重叠示例**——把组 [0,1] 从槽 0,1 拖到槽 1,2（往右 1 格）：

```
[PuzzleBoard] 整组搬运 — group=0, groupPids=[0,1], dRow=0, dCol=1,
                          displaceFrom=[2], fillTo=[0]
```

注意 `displaceFrom` 只有 1 个元素（槽 2，因为槽 1 既在 src 又在 dst，被对称差消掉），`fillTo` 也只有 1 个元素。**集合差自动处理重叠**——不用写任何"src/dst 是否相交"的 if 判断。

**不正常的信号**：

- **整组拖一块 + 别块没动**：onPieceTouchMove 的循环写错——只对 `pid === dragPid` 改了 position，其他组员没改。检查循环条件 `groupId === dragGroupId`。
- **整组挪到目标但被挤者回填错位**：两个常见错因：
  - `newSlots[fillTo[i]] = newSlots[displaceFrom[i]]`（用 newSlots 读，但写完组员后 newSlots 上 displaceFrom 槽已经是组员了，读到的是组员 pid）→ 整盘错乱。**必须从 OLD slots 读** = `this.slots[displaceFrom[i]]`。
  - `displaceFrom` 和 `fillTo` 的对应关系搞混（顺序错位）→ 表现为"被挤者乱填"。**关键**：displaceFrom[i] 和 fillTo[i] 是按数组顺序对应的，不需要"建立对应关系"——`filter` 顺序稳定，**前提是 srcSlots 和 dstSlots 数组顺序也稳定**。本节代码里两个数组都是按 groupPids 顺序 push 的，OK。
- **拖到边缘整组卡半空**：snapBackGroup 没调用。检查越界判断分支是否 return。
- **z-order 不对（拖到中央时被中央挡住）**：onPieceTouchStart 漏写了。
- **拖完两次后组员"分家"**：mergeScan 调了但没生效，或 mergeScan 内部 union 条件错——回 05 节复核 Rule B。
- **ghost piece（视觉上有第三块跟着动）**：组员判断错——把 `groupId === dragGroupId` 写成 `groupId !== dragGroupId`。

## 7. 复盘：这节教了什么

**1. 集合差是"消除特殊情况" 的教科书案例**

朴素做法处理"src/dst 重叠" 要写好几个 if-else。

集合差做法：**`displaceFrom = dstSlots \ srcSlots; fillTo = srcSlots \ dstSlots;`** ——两行代数表达式自动消掉重叠。

| 维度 | 朴素 | 集合差 |
|------|------|--------|
| 代码量 | ~40 行（含各种 if） | 2 行 |
| 概念 | 4 种特殊 case | 0 |
| 数学保证 | 没有 | `|displaceFrom| === |fillTo|` 永真 |
| 调试 | 每个 case 单独测 | 一次性对（被数学保证） |

Linus："**好代码消除特殊情况**"——这就是消除特殊情况的实战。

记住这句话：**当你要写 `if (A 和 B 是否相交)` 时，先想想能不能用集合代数表达**。多数时候能。

**2. 单一真相源 + 投影 在整组场景的扩展**

03 节"零状态拖拽" 的精神在本节扩展：

- snapBackGroup 不存"按下时位置" 快照，组员各自从 `slots.indexOf(pid)` 现算。
- 整组搬运时**先全算 srcSlots/dstSlots**（基于 OLD slots），**再统一写 newSlots，再读 newSlots 摆位置**。

**关键纪律**：写 newSlots 期间**不**穿插读 newSlots 来推导后续结果——否则"先写组员、再读槽位决定被挤者填到哪" 就会读到组员而不是被挤者。

**先全部读 → 全部算 → 全部写**，这个三阶段流水线是数据驱动开发的稳态写法。

**3. 拖动体验的细节：z-order**

你调 z-order 的代码只有 4 行（onPieceTouchStart 里的循环），但**没这 4 行用户立刻发现**——拖一组到中央，组员被中央挡住一半，**眼睛痛**。

这是用户体验的"不易察觉但必要" 类细节：

- 如果你做对了，没人会赞美。
- 如果你做错了，每个用户都觉得"这游戏有 bug"。

游戏开发的细节战不是"加新功能"，是**砍掉这种隐形 bug**。

**4. 事务模型雏形**

本节实际上已经形成了"事务" 模式：

```text
1. BEGIN：onPieceTouchStart 锁定 dragGroupId、提 z-order
2. UPDATE：onPieceTouchMove 同步挪整组 position
3. COMMIT：onPieceTouchEnd 写 slots + 摆位 + mergeScan
```

3 个阶段对应"事务" 的 BEGIN / WORK / COMMIT。COMMIT 之前 slots 完全不变（拖动期间是"虚位移"），COMMIT 是原子的（slots 一次性更新到 newSlots）。

07 节会给这个事务加上"输入锁 + 缓动" —— 让 COMMIT 阶段的视觉变化有动画过程，期间锁定用户输入。

## 8. 常见坑

- **`newSlots = this.slots.slice()` 漏 .slice()**：直接 `newSlots = this.slots` 是引用赋值，`newSlots[x] = y` 改的是同一个数组——读 / 写交叉污染整盘。**`.slice()` 必加**。
- **被挤者用 `newSlots[displaceFrom[i]]` 读**：如前所述，这时 newSlots 对应槽已经是组员了——**必须用 OLD slots = `this.slots[displaceFrom[i]]`**。
- **filter 顺序假设错**：JS Array.prototype.filter 保证返回顺序与原数组一致——所以 `dstSlots.filter(...)` 和 `srcSlots.filter(...)` 的顺序保留。但如果你换成 `Set` 操作，**Set 是无序的**——`dstSet.difference(srcSet)` 返回顺序不定，对应关系崩溃。**坚持用数组 + filter**。
- **groupPids 遗漏 dragPid**：如果你的循环条件写成 `pid !== dragPid && groupId === dragGroupId`，groupPids 里就没有 dragPid——后续算偏移、写 slots 都会漏一块。**dragPid 自己也是 groupId 的成员**——条件只看 groupId 即可。
- **越界检查放错位置**：必须在"算 dstSlots" 阶段就检查每个组员的 dstRow/dstCol——而不是等到"写 newSlots" 时遇到下标越界报错。早检查早返回。
- **整组拖动时 mergeScan 跑了**：本节 onPieceTouchMove 不调 mergeScan、onPieceTouchStart 也不调——只在 onPieceTouchEnd 末尾跑一次。如果你画蛇添足往 onPieceTouchMove 里加了 mergeScan，每帧都会重算 groupId，**拖动中途组可能解散**——bug 难调。
- **同 groupId 的块物理上不连续**：比如组 [0, 1, 2]（横排）被某个 swap 弄散了——理论上 mergeScan 会把它们解散到不同 groupId。如果你看到"3 块同 groupId 但物理上分开"，证明 mergeScan 的 Rule B 判定有 bug 或某次没跑（漏调用）。
- **拖一组拖出网格但有部分组员仍在网格内**：本节代码是"任何一个组员 dst 越界 → 整组弹回"——这是保守做法，符合多数玩家直觉。如果你想做"贴边即可" 的策略需要更复杂的边界裁剪，超出本教学链。

## 9. 下一节预告

完整的拖拽 / 合并 / 群组拖动都有了——但是**胜利了系统不知道**。控制台还在打"merge happened"，但没有"恭喜！" 的反馈，没有计时，没有"再来一局" 按钮。

**07 节**：胜利判定 + 缓动 + 输入锁。

- **胜利条件**：所有块的 groupId 相同（== mergeScan 后只剩 1 组）。
- **计时**：onLoad 记 startTime，胜利时算 elapsed。
- **结算反馈**：胜利后控制台打 elapsed，可选弹个 UI。
- **缓动 (tween)**：把 swap 后的"瞬移摆位" 改成 0.2s 平滑动画。
- **输入锁 (`inputLocked`)**：缓动期间禁止再拖——避免半路 swap 出现 race condition。
- **状态机**：用一个 `victorious` 字段早退 onPieceTouchStart——胜利后玩家拖不动，等"再来一局"。
