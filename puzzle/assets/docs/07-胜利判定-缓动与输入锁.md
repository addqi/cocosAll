# 07 · 胜利判定 — 缓动 / 输入锁 / 状态机

> **目标**：合成 1 组 = 胜利。胜利后**禁止再拖**、**报告用时**、（可选）弹结算 UI。同时 swap 后的瞬移改成 **0.2s 平滑缓动**，缓动期间锁输入避免操作叠加。
>
> 完成后这是一个**真完整、可上线**的拼图游戏——只差美术和关卡系统。

---

## 起点回顾

06 节末态：

- 拖一块 = 拖整组、`displaceFrom / fillTo` 多换多。
- swap 后立即 `setPosition(...)` 瞬移到位 + 调 mergeScan。
- 胜利状态没人检测、操作期间可以反复拖。

本节加 4 件事：

1. `victorious` 字段（boolean，初值 false）。
2. mergeScan 返回新数据 `allInOneGroup`（组数 === 1）。
3. onPieceTouchEnd 末尾检查 `allInOneGroup` 并报告用时。
4. swap 后位置变化改成 tween 缓动 + 期间设 `inputLocked` 锁。

---

## 1. 关键概念：胜利判定属于"提交后"

胜利触发的时机：**slots 已经更新、mergeScan 已经跑完**——即 06 节里 onPieceTouchEnd 的**末尾**。

**为什么不是 onPieceTouchStart**？因为开始拖的瞬间 slots 还没变，mergeScan 也没跑——这时检测胜利会**漏一帧**：上一次拖动让组合成功，但 mergeScan 标记完之后没人检测，玩家这次按下时才发现"哦原来已经赢了"。**胜利信号晚一帧 = 用户感知"卡了一下"**。

**为什么不是 onPieceTouchMove**？拖动期间 slots 不变，groupId 不变——胜利状态不会动。把检测放进 move 是无意义的每帧浪费。

**只有 COMMIT 阶段（slots 提交 + mergeScan 跑完）有可能产生胜利状态变化**——那就只在那一刻检测。

```text
03 节: onPieceTouchEnd 弹回（无 swap）
04 节: onPieceTouchEnd swap + 写 slots
05 节: onPieceTouchEnd swap + 写 slots + mergeScan
06 节: onPieceTouchEnd 整组 swap + 写 slots + mergeScan
07 节: onPieceTouchEnd 整组 swap + 写 slots + mergeScan + 胜利判定 ← 本节加
```

胜利判定永远在最末尾——是事务的"最后一步"。

---

## 2. 关键概念：mergeScan 返回 allInOneGroup

05 节 mergeScan 已经返回 `mergedAny`。本节加返回 `allInOneGroup`：

```typescript
// 数组里有几个不同的 root（即组数）
const roots = new Set<number>();
for (let pid = 0; pid < this.pieceCount; pid++) {
    roots.add(this.find(parents, pid));
}
const allInOneGroup = roots.size === 1;

return { mergedAny, allInOneGroup };
```

`allInOneGroup === true` 表示所有块属于同一组——**胜利**。

**重要**：**胜利状态由 mergeScan 派生**，不是另存的字段。slots / groupId 是真相源——胜利只是"slots 形成全部相对正确" 的派生事实。再次"消除特殊情况"——不存冗余状态。

---

## 3. 关键概念：输入锁（inputLocked）

加缓动后，swap 提交期间有 200ms 的动画过程。**这期间禁止再拖**——否则：

```text
T = 0ms:   拖一组、松手、开始 200ms 缓动
T = 100ms: 玩家又按下另一块（缓动还没结束，slots 已更新但 position 在动画中）
           → onPieceTouchStart 算 dragGroupId、提 z-order
T = 200ms: 第 1 次缓动结束、mergeScan 跑、检测胜利
T = 300ms: 玩家在第二个块上松手 → 又一次 swap 触发
           → 第二次缓动开始
           → 但第一次的 mergeScan 刚跑完，groupId 跟 slots 不一致（因为第二次 swap 还没 mergeScan）
```

要么 race condition、要么数据不一致。**简单解法**：缓动期间 `inputLocked = true`，所有 onPieceTouchStart / Move / End **早退**。

```typescript
private inputLocked: boolean = false;

private onPieceTouchStart(event: EventTouch): void {
    if (this.inputLocked || this.victorious) return;
    // ...
}
```

`inputLocked` 在 swap 后立刻设 true，缓动结束后设 false 并跑 mergeScan + 胜利检测。Cocos 的 `tween` 有 `.call(callback)` 让缓动结束后回调——把"清锁 + mergeScan + 胜利检测" 都塞进 callback。

---

## 4. 关键概念：tween 缓动

Cocos 3.x 内置 tween API：

```typescript
import { tween, Vec3 } from 'cc';

tween(node)
    .to(0.2, { position: new Vec3(targetX, targetY, 0) })
    .call(() => { /* 完成后回调 */ })
    .start();
```

`to(duration, props)` 让节点在 `duration` 秒内插值到 `props` 描述的目标值。

**整组 swap 时有 N+M 个 tween（N 个组员 + M 个被挤者）**。M = displaceFrom.length。

**问题**：N+M 个 tween 都在跑，**只想最后一个跑完时清锁**——怎么知道哪个最后？

**答案**：所有 tween 时长相同（都是 0.2s）→ 它们**同时结束**。任选一个挂 callback 即可。教学版用"最后一个 tween 挂 callback"——具体看代码。

更老练的做法用 `Promise.all([...tweens.map(tweenToPromise)])` 等所有完成——那是工程化版本，本教学链用最简的"挂 callback 在最后一个"。

**改 mergeScan 返回类型**（注意返回值类型变了）：

```typescript
private mergeScan(): { mergedAny: boolean; allInOneGroup: boolean } {
    const parents: number[] = [];
    for (let pid = 0; pid < this.pieceCount; pid++) parents.push(pid);

    let mergedAny = false;
    for (let sr = 0; sr < this.pieceGrid; sr++) {
        for (let sc = 0; sc < this.pieceGrid; sc++) {
            const idx = sr * this.pieceGrid + sc;
            const pid = this.slots[idx];
            const piece = this.pieceNodes[pid].getComponent(PuzzlePiece)!;

            if (sc + 1 < this.pieceGrid) {
                const rPid = this.slots[idx + 1];
                const rPiece = this.pieceNodes[rPid].getComponent(PuzzlePiece)!;
                if (rPiece.row === piece.row && rPiece.col === piece.col + 1) {
                    if (this.union(parents, pid, rPid)) mergedAny = true;
                }
            }
            if (sr + 1 < this.pieceGrid) {
                const dPid = this.slots[idx + this.pieceGrid];
                const dPiece = this.pieceNodes[dPid].getComponent(PuzzlePiece)!;
                if (dPiece.row === piece.row + 1 && dPiece.col === piece.col) {
                    if (this.union(parents, pid, dPid)) mergedAny = true;
                }
            }
        }
    }

    for (let pid = 0; pid < this.pieceCount; pid++) {
        const piece = this.pieceNodes[pid].getComponent(PuzzlePiece)!;
        piece.groupId = this.find(parents, pid);
    }

    const roots = new Set<number>();
    for (let pid = 0; pid < this.pieceCount; pid++) {
        roots.add(this.find(parents, pid));
    }
    const allInOneGroup = roots.size === 1;

    return { mergedAny, allInOneGroup };
}
```

**重写 onPieceTouchEnd**（最复杂的一节）：

```typescript
private onPieceTouchEnd(event: EventTouch): void {
    if (this.victorious || this.inputLocked) return;

    const node = event.target as Node;
    const piece = node.getComponent(PuzzlePiece)!;
    const dragPid = piece.pieceId;
    const dragGroupId = piece.groupId;

    const groupPids: number[] = [];
    for (let pid = 0; pid < this.pieceCount; pid++) {
        if (this.pieceNodes[pid].getComponent(PuzzlePiece)!.groupId === dragGroupId) {
            groupPids.push(pid);
        }
    }

    const dragSrcSlot = this.slots.indexOf(dragPid);
    const dragDstSlot = this.pointerToSlot(node.position.x, node.position.y);

    if (dragDstSlot < 0) {
        this.snapBackGroupTween(groupPids);
        return;
    }

    const dRow = Math.floor(dragDstSlot / this.pieceGrid) - Math.floor(dragSrcSlot / this.pieceGrid);
    const dCol = (dragDstSlot % this.pieceGrid) - (dragSrcSlot % this.pieceGrid);

    const srcSlots: number[] = [];
    const dstSlots: number[] = [];
    for (const pid of groupPids) {
        const srcSlot = this.slots.indexOf(pid);
        const srcRow = Math.floor(srcSlot / this.pieceGrid);
        const srcCol = srcSlot % this.pieceGrid;
        const dstRow = srcRow + dRow;
        const dstCol = srcCol + dCol;
        if (dstRow < 0 || dstRow >= this.pieceGrid || dstCol < 0 || dstCol >= this.pieceGrid) {
            this.snapBackGroupTween(groupPids);
            return;
        }
        srcSlots.push(srcSlot);
        dstSlots.push(dstRow * this.pieceGrid + dstCol);
    }

    const displaceFrom = dstSlots.filter(s => !srcSlots.includes(s));
    const fillTo = srcSlots.filter(s => !dstSlots.includes(s));

    const newSlots = this.slots.slice();
    for (let i = 0; i < groupPids.length; i++) {
        newSlots[dstSlots[i]] = groupPids[i];
    }
    const displacedPids: number[] = [];
    for (let i = 0; i < displaceFrom.length; i++) {
        const dPid = this.slots[displaceFrom[i]];
        newSlots[fillTo[i]] = dPid;
        displacedPids.push(dPid);
    }
    this.slots = newSlots;

    // 锁输入 + 缓动
    this.inputLocked = true;
    const allMovingPids = [...groupPids, ...displacedPids];
    let lastTween: any = null;
    for (const pid of allMovingPids) {
        const slot = this.slots.indexOf(pid);
        const { x, y } = this.slotToPosition(slot);
        const t = tween(this.pieceNodes[pid])
            .to(0.2, { position: new Vec3(x, y, 0) });
        lastTween = t;
        t.start();
    }

    // 在最后一个 tween 末尾挂回调（所有时长 0.2s 同步结束，挂哪个都行）
    if (lastTween) {
        lastTween.call(() => this.commitAfterTween()).start();
        // 注意：上行其实多 .start() 一次会重新跑一次缓动——
        // 正确做法是构建链时就 .call() 再 .start()。下面"完整版"代码改正了。
    } else {
        // 没动任何块（理论不会到这里——dst 等于 src 的话 dragDstSlot === dragSrcSlot
        // 也会进入分支，dRow=dCol=0，srcSlots===dstSlots，displaceFrom 和 fillTo 为空，
        // groupPids 都不动）—— 但仍然安全：直接 commit。
        this.commitAfterTween();
    }
}

/** 缓动结束后的提交逻辑：清锁 + mergeScan + 胜利判定 */
private commitAfterTween(): void {
    this.inputLocked = false;
    const { mergedAny, allInOneGroup } = this.mergeScan();
    if (mergedAny) {
        const groupCounts = this.countGroups();
        console.log(
            `[PuzzleBoard] merge happened — 当前 ${groupCounts.length} 组，` +
            `最大组 ${Math.max(...groupCounts)} 块`,
        );
    }
    if (allInOneGroup && !this.victorious) {
        this.victorious = true;
        const elapsed = Date.now() - this.startTime;
        console.log(`[PuzzleBoard] 胜利！用时 ${(elapsed / 1000).toFixed(1)}s`);
    }
}

/** 整组弹回（带缓动） */
private snapBackGroupTween(groupPids: number[]): void {
    this.inputLocked = true;
    let lastTween: any = null;
    for (const pid of groupPids) {
        const slot = this.slots.indexOf(pid);
        const { x, y } = this.slotToPosition(slot);
        lastTween = tween(this.pieceNodes[pid])
            .to(0.2, { position: new Vec3(x, y, 0) });
        lastTween.start();
    }
    if (lastTween) {
        // 单独再启一个回调 tween（更稳的做法，独立链）
        tween(this.pieceNodes[groupPids[0]])
            .delay(0.2)
            .call(() => { this.inputLocked = false; })
            .start();
    } else {
        this.inputLocked = false;
    }
}
```

**注意**：上面有个"挂 callback 不要重复 .start()" 的细节——正确的链式写法是**构建时就把 .call() 接在 .to() 后面，最后 .start()**。下面给出**修正版**（替换 swap 那段缓动）：

```typescript
// 替换 onPieceTouchEnd 的 "锁输入 + 缓动" 那段，使用单独"timer tween"：
this.inputLocked = true;
const allMovingPids = [...groupPids, ...displacedPids];
for (const pid of allMovingPids) {
    const slot = this.slots.indexOf(pid);
    const { x, y } = this.slotToPosition(slot);
    tween(this.pieceNodes[pid])
        .to(0.2, { position: new Vec3(x, y, 0) })
        .start();
}
// 单独一个 timer tween 挂回调——所有 piece 0.2s 同步结束
tween(this.node)
    .delay(0.2)
    .call(() => this.commitAfterTween())
    .start();
```

`tween(this.node).delay(0.2).call(...)` 是"timer tween" 模式——挂在 PuzzleBoard 自己 node 上，只用作计时器，不动 position。

## 6. 跑起来

▶ 预览。

**测试动作**：

1. 拖一块到目标位置 → 看到**0.2s 平滑过渡** 而不是瞬移。
2. 缓动期间立刻按住另一块拖 → **没反应**（inputLocked 早退）。等 200ms 缓动结束 → 又能拖了。
3. 反复操作直到 9 块全合一组 → 控制台打：

```
[PuzzleBoard] merge happened — 当前 1 组，最大组 9 块
[PuzzleBoard] 胜利！用时 23.4s
```

4. 胜利后再点任何一块 → 拖不动（victorious 早退）。

5. 重启场景 → 重新洗牌、重新计时。

**控制台预期**（一盘完整流程）：

```
[PuzzleBoard] 洗牌完成 — pieceCount=9, misplaced=8, attempts=1
[PuzzleBoard] 整组搬运 — group=0, groupPids=[0], dRow=1, dCol=0, displaceFrom=[3], fillTo=[0]
[PuzzleBoard] merge happened — 当前 8 组，最大组 2 块
... (重复多次)
[PuzzleBoard] 整组搬运 — group=0, groupPids=[0,1,2,3,4,5,6,7], dRow=0, dCol=0, displaceFrom=[8], fillTo=[?]
[PuzzleBoard] merge happened — 当前 1 组，最大组 9 块
[PuzzleBoard] 胜利！用时 47.2s
```

**不正常的信号**：

- **缓动结束后回调跑了 2 次（"用时" 打 2 次）**：tween 链式构建时多调了一次 `.start()`——`tween(node).to(...).call(...).start()` 是**一次链一次 start**，不要分两步 start。
- **`commitAfterTween` 永远不跑**：timer tween 没启或回调没挂。检查 `tween(this.node).delay(0.2).call(...).start();` 那行是否存在、是否被某条件分支跳过。
- **inputLocked 卡死永远 true**：commitAfterTween 没跑 → 锁不被清。看上一条。或者 commitAfterTween 内部抛异常（比如 mergeScan 出错）—— Cocos 的 tween callback 抛异常**默认不会 crash 但锁不被清**。把 commitAfterTween 用 try-catch 包一层。
- **胜利后还能拖**：`onPieceTouchStart / Move / End` 的早退检查写漏了。每个回调入口**都要查 `this.victorious` + `this.inputLocked`**。
- **胜利打多次 "用时"**：`if (allInOneGroup && !this.victorious)` 的 `!this.victorious` 漏写——胜利后任何 mergeScan 仍返回 allInOneGroup=true，会反复触发。**必须加这层"已胜利" 早退**。
- **缓动出现"先飞到旁边再回来" 的诡异路径**：tween 默认是 linear 插值，本节没问题。如果你看到曲线，可能是 Cocos 默认 easing 是 sine 之类——不影响功能。
- **缓动期间用户狂点几下都没反应、缓动结束突然 swap 一次**：Cocos 的事件队列在 inputLocked 期间被早退丢弃——本来就该如此，不是 bug。
- **胜利状态错过了开局合并**：你 onLoad 里 mergeScan 没保存 allInOneGroup。但开局 mergeScan 不应该 allInOneGroup=true（除非洗牌失败、9 块都在原位）—— scatterRatio=0.7 保证 6+ 块错位，不可能开局就赢。除非你 scatterRatio 设成 0。**别这么设**。

## 7. 复盘：这节教了什么

**1. 状态机：3 个状态，3 个早退**

```text
状态：    可玩       (默认)
         缓动中     (inputLocked = true，0.2s 后自清)
         已胜利     (victorious = true，永不清)
```

每个 touch 回调入口都查这两个布尔，任一为真就早退。这就是状态机的**最简实现**——不需要 enum、不需要"状态切换" 函数。

很多教程一上来就讲 "状态机模式 = 一个 currentState + 一堆状态类 + transition 表"。**3×3 拼图 2 个布尔就够了**——别为复杂而复杂。

什么时候需要 enum + 转换表？**状态多 + 转换路径复杂 + 转换有副作用**。比如游戏关卡有"未开始、加载中、游戏中、暂停、结算、回放" 6 状态，且彼此可切换——这种再上重型状态机。教学版越简越好。

**2. 缓动是"事务的可视化"**

之前 04-06 节的 swap 是瞬时的——slots 一变、setPosition 一秒一变。**用户感知**：好像有什么东西"刷"地跳了。

加缓动后：

```text
T = 0:    用户松手、slots 立刻更新到新状态
T = 0~200ms: 视觉上块在缓动到目标位置
T = 200ms: mergeScan 跑、检测胜利
```

slots 这个真相源**在 T=0 那一刻已经定型**——缓动只是把"投影变化" 拉长到 200ms 让用户看清楚。

**真相源是瞬时的、视觉是延时的**——这是动画跟数据驱动配合的标准范式。React 的 transition、Vue 的 transition、Flutter 的 implicit animation——全是同一个思想。

**3. 胜利状态不存（派生自 mergeScan）**

`victorious` 字段技术上可以删掉——每次都从 `slots / groupId` 算"是不是只有一组"。

为啥还存？**早退开销**——每次 touchstart 跑一次 mergeScan O(N²) 太重。存个布尔 O(1) 查询。

但要小心：`victorious = true` 之后再也不重置。这是有意为之——**胜利是终止状态**，不应该被任何后续操作"反向触发回未胜利"。如果要"再来一局"，应该重置整盘：

```typescript
public restart(): void {
    this.victorious = false;
    this.inputLocked = false;
    this.startTime = Date.now();
    this.initSlots();
    this.shuffleSlots();
    this.layoutAllPieces();
    this.mergeScan();
}
```

调用方在 UI 按钮点击时调 `puzzleBoard.restart()`。**没让 mergeScan 自动反转 victorious**——胜利就是终止，需要外部显式重启。

**4. timer tween：拿 tween 当事件总线**

`tween(this.node).delay(0.2).call(...)` 不动 node、只用作"200ms 后跑回调"。

为啥不用 `setTimeout(..., 200)`？

- setTimeout 不绑生命周期——节点销毁了 setTimeout 还在跑，回调里访问销毁节点 → crash。
- tween 绑节点——节点销毁时 tween 自动取消。**生命周期安全**。
- tween 跟其他 tween 走同一调度——节奏感更稳定。

游戏里"延时回调" 一律用 tween 当 timer，不用 setTimeout。这是 Cocos / Unity / 其他游戏引擎的通行规约。

**5. 一句话总结整个教学链**

```text
01: 数据是真相、视觉是投影      （slots → position）
02: 洗牌只动数据                （shuffle slots）
03: 拖动是临时偏移              （setPosition during touch，弹回从 slots 现算）
04: swap 改数据、视觉跟着        （slots 改 → setPosition 跟）
05: groupId 派生自 slots         （mergeScan 全量重算）
06: 整组搬运用集合差             （displaceFrom = dst \ src，fillTo = src \ dst）
07: 胜利派生自 mergeScan         （allInOneGroup → victorious）
```

**所有状态都有唯一真相源**：

- 槽位真相 = `slots`
- 块身份真相 = `pieceId / row / col`（创建时定，永不变）
- 组身份真相 = mergeScan 临时算
- 胜利真相 = mergeScan 临时算 + victorious 缓存

**没有任何字段是"另存的、需要跟其他字段同步" 的状态**。Linus："好程序员关心数据结构"——就这意思。

## 8. 常见坑

- **缓动期间用户点了"再来一局" 按钮**：restart 跑了，但缓动还在跑——那个 callback 200ms 后还会跑 commitAfterTween，可能把 inputLocked 清掉、覆盖 restart 的初始化。**解决**：restart 时先 `Tween.stopAllByTarget(node)` 取消所有未完缓动。3×3 教学版可以不管。
- **`victorious` 重置成 false 但 startTime 没重置**：下一局的 elapsed 算成上一局总时间——计时假。**重置一切的状态字段**，跟 onLoad 一致。
- **tween 的 `to` 不动 node（看起来缓动没跑）**：Cocos 3.x 的 tween 需要 `start()` 触发——构建链不调 start = 没动。
- **`tween(this.node)` 跟 `tween(piece.node)` 混用**：本节用 PuzzleBoard 自身 node 当 timer，跟 piece tween 不冲突——它们是不同 node 的 tween，互不影响。如果你为了"一致" 把 timer tween 也挂在某个 piece node 上，万一那 piece 在缓动结束前被 destroy，timer 也被取消、commitAfterTween 不跑。**timer 挂在最稳定的 node**（PuzzleBoard 自己）。
- **多次 swap 嵌套触发**：本节用 inputLocked 防住——但**一定要在 swap 启动时立刻设 true**，不能拖到 tween 第一帧才设。看 `this.inputLocked = true;` 那行的位置——在所有 tween.start() **之前**。
- **commitAfterTween 内部异常导致锁不释放**：包 try-catch 或在 finally 里清锁：
  ```typescript
  private commitAfterTween(): void {
      try {
          // ... mergeScan + 胜利检测 ...
      } finally {
          this.inputLocked = false;
      }
  }
  ```
  教学版可以省略。
- **胜利后能拖动一块进去再拉出来产生"又赢一遍"**：`!this.victorious` 早退保证了这种事——**仅在第一次** allInOneGroup 触发 victory 逻辑、之后所有 onPieceTouchStart 都被 `victorious` 早退挡住——拖不动。理论闭环。
- **`Date.now()` 跟 Cocos 的 `director.getDeltaTime()` 不一样**：`Date.now()` 是 wall-clock 毫秒（1970 起），Cocos 的 deltaTime 是帧间隔。计时用 Date.now() 没问题——除非用户**改了系统时间**（比如玩到一半改了时区），那计时会"瞬移"。**3×3 教学版**没人在乎这种事。生产版本可以改 `performance.now()` 或 director 累加 deltaTime。

## 9. 教学链总结

到这里你已经写完一个**真正可玩的拼图游戏**——

| 节 | 核心知识点 | 加了什么 | 字段数（累计） |
|---|---|---|---|
| 00 | Cocos 项目 / Sprite / UITransform | 一张完整图 | 0 |
| 01 | SpriteFrame.rect / slots 投影 | N×N 切片 | 1（slots） |
| 02 | Fisher-Yates / 散度 / 兜底重试 | 洗牌 | 1 |
| 03 | 触摸捕获 / UIDelta / 零状态拖拽 | 跟手 / 弹回 | 1 |
| 04 | pointerToSlot / round vs floor | 单块 swap | 1 |
| 05 | 并查集 / Rule B / 全量重扫 | 自动合并 groupId | 1（slots）+ groupId（PuzzlePiece 字段） |
| 06 | 集合差消除特殊情况 / z-order | 整组拖动 / 多换多 | 1 |
| 07 | 状态机 / tween / 派生胜利状态 | 缓动 / 输入锁 / 胜利 | 4（slots, victorious, inputLocked, startTime） |

**总代码量**：`PuzzleBoard.ts` 约 350 行、`PuzzlePiece.ts` 约 12 行。所有"复杂玩法" 用一个 component 一个 data class 搞定——**没必要建什么 manager / service / state / store 的层级**，更没必要请第三方状态管理库。**简单的事简单做**。

**核心设计原则回顾**：

1. **位置是 slots 的投影** —— 所有 setPosition 都从 slots 现算，不存"按下时位置"。
2. **groupId 是 slots 的投影** —— mergeScan 全量重算，不维护"组拆解" 逻辑。
3. **胜利是 mergeScan 的投影** —— allInOneGroup 派生自组数。
4. **集合差消除特殊情况** —— displaceFrom/fillTo 用代数，不写 if。
5. **状态机 = 几个布尔早退** —— inputLocked / victorious 简单粗暴。
6. **事务 = BEGIN-WORK-COMMIT** —— TouchStart/Move/End 三阶段对应。

记住这 6 条，去做下一个游戏（华容道、连连看、消消乐）你都能写得又快又稳。

## 10. 进阶方向（延伸阅读）

教学链结束。下面是真上线时会做的事，**不是本教学链的范围**，但你可以自己探索：

- **关卡系统**：多个图、可选难度、分数榜。需要 UI（按钮 / 列表）。
- **本地存档**：用 `localStorage` 或 Cocos 的 `sys.localStorage` 存最佳成绩、当前进度。
- **碎片形状**：把 SpriteFrame.rect 切方块改成"凹凸边缘" 的拼图形状——需要每块挂自定义 mesh 或 mask。
- **音效 / 粒子**：合并时播 "click"、胜利时播 "fanfare" + 烟花粒子。
- **联机对战**：两人比谁先拼完。需要 WebSocket + 服务端权威 slots。
- **撤销 / 重做**：维护一个 slots 的栈。每次 swap 前 push、撤销时 pop + 重新 layout。
- **AI 提示**：算最少需要几步还原，给玩家"高亮下一步该挪哪块" 提示。
- **响应式**：屏幕旋转时 boardSize 跟着变。@property `boardSize` 改成动态算（比如 `Math.min(screenWidth, screenHeight) * 0.8`）。
- **Shader 边框 / 渐变 / 拼图凹凸效果**：手写 effect 文件（cocos 3.x 用 `.effect`）。
- **性能压测**：试试 32×32 = 1024 块，看哪里先扛不住。多半是 mergeScan 的 getComponent 调用——缓存 PuzzlePiece 指针就够。
