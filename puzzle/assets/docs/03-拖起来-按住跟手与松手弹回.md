# 03 · 拖起来 — 按住跟手与松手弹回

> **目标**：按住任意一块 → 块跟着手指走 → 松手块**弹回原槽**。本节不做"命中判定"——拖到哪里都弹回，目的是先把"输入流水线" 跑通。

---

## 起点回顾

02 节末态：

- 9 块开局打乱（散度 ≥ 70%）。
- `layoutAllPieces` 把每块按 `slots.indexOf(pid)` 摆到对应槽位。
- 屏幕看到一盘乱的拼图，**但没法操作**。

本节加一件事：让每块响应触摸。

---

## 1. Cocos 触摸事件四件套

Cocos 3.8 的 Node 自带触摸事件：

| 事件 | 触发时机 |
|------|----------|
| `Node.EventType.TOUCH_START`  | 手指/鼠标按下，**且命中本节点**（即点击位置在 UITransform 范围内） |
| `Node.EventType.TOUCH_MOVE`   | 命中后，手指/鼠标移动 |
| `Node.EventType.TOUCH_END`    | 手指/鼠标松开**在节点范围内** |
| `Node.EventType.TOUCH_CANCEL` | 手指/鼠标松开**在节点范围外**（滑出去后松手） |

**关键概念：触摸捕获**

- `TOUCH_START` 只发给"被点击命中" 的那个节点。
- `TOUCH_MOVE` / `TOUCH_END` / `TOUCH_CANCEL` 之后**只发给同一个节点**——即使手指滑出去了。
- 这意味着：**按下哪块，后续整个拖动过程都只这块收事件**。**不存在"边滑边切换"的歧义**。

利用这个机制，我们的 piece 拖拽逻辑**天然不需要"当前活跃块"全局变量**——回调函数的 `event.target` 永远就是那一块。

## 2. 关键设计：零状态拖拽

最朴素的拖拽实现是这样的：

```typescript
// 反例：维护"按下瞬间的位置"
private dragStartX: number = 0;
private dragStartY: number = 0;

onTouchStart(event) {
    this.dragStartX = node.position.x;
    this.dragStartY = node.position.y;
}

onTouchEnd(event) {
    node.setPosition(this.dragStartX, this.dragStartY); // 弹回
}
```

**问题**：

- 多了 2 个字段（`dragStartX/Y`）。
- 这 2 个字段**只在拖动期间有效**——没拖时它们是过期值。
- **真 bug**：如果按下瞬间块被另一逻辑临时挪到错误位置（比如同时有动画跑），`dragStartX/Y` 记下的就是错误位置，松手弹回到错位置。

**零状态做法**：弹回目标 = `slotToPosition(slots.indexOf(pieceId))`。

- **不存任何拖动期状态**。
- 弹回从 slots 这个真相源现算——slots 在拖动期间不变，永远算得对。
- "块的正确位置" 不依赖"按下瞬间的视觉状态"，只依赖逻辑数据。

这是 01 节"位置是 slots 的投影" 在拖拽场景的延伸：**位置是投影 → 弹回也是投影 → 不需要快照**。

## 3. 屏幕坐标 → Board-local 坐标

`event.getDelta()` 返回手指本帧的移动量（屏幕像素）。我们要把它累加到块的 `position`。

但**要注意**：`event.getDelta()` 是**屏幕坐标系**（Y 向下为正），Cocos 节点的 `position` 是**节点局部坐标系**（Y 向上为正）。差异：

| 系统 | X | Y |
|------|---|---|
| 屏幕坐标（手指） | 右为正 | **下为正** |
| Cocos UI 局部坐标（节点） | 右为正 | **上为正** |

所以 dy 要取反。

```typescript
const delta = event.getUIDelta(); // ← 用 UIDelta 不用 Delta
node.setPosition(node.position.x + delta.x, node.position.y + delta.y, 0);
```

`getUIDelta()` 是 Cocos 3.x 提供的"已经转到 UI 坐标系的 delta"，**Y 已经倒过来过**——直接加即可。

如果用 `getDelta()`（原始屏幕坐标）就要自己 `dy = -dy`。**用 getUIDelta**——简单不容易错。

## 4. 改 `PuzzleBoard.ts`：注册 + 处理触摸

打开 `PuzzleBoard.ts`，**在 `createPieces` 末尾每个 node 创建后追加事件注册**：

```typescript
import { _decorator, Component, Node, Sprite, SpriteFrame, UITransform, Rect, EventTouch, Vec3 } from 'cc';
//                                                                       ^^^^^^^^^^  ^^^^ 新增 import
```

`createPieces` 末尾每块 piece 创建后追加：

```typescript
for (let pid = 0; pid < this.pieceCount; pid++) {
    // ...原有 node / transform / sprite / piece 创建...

    // ===== 03 节新增：注册触摸事件 =====
    node.on(Node.EventType.TOUCH_START, this.onPieceTouchStart, this);
    node.on(Node.EventType.TOUCH_MOVE, this.onPieceTouchMove, this);
    node.on(Node.EventType.TOUCH_END, this.onPieceTouchEnd, this);
    node.on(Node.EventType.TOUCH_CANCEL, this.onPieceTouchEnd, this);
    //                                          ^^^^^^^^^^^^^^^^^^^ TouchCancel 复用 TouchEnd 处理函数

    this.pieceNodes.push(node);
}
```

**新增方法**（贴在类的末尾）：

```typescript
/**
 * 按下：本节什么都不用做。
 *
 * 触摸捕获机制保证后续 move/end 只发给本块——不需要"激活"什么状态。
 * 03 节先留着空方法，05 节起需要在这里做 z-order 提到最前。
 */
private onPieceTouchStart(event: EventTouch): void {
    // 03 节空实现
}

/**
 * 移动：按 UIDelta 累加 position。
 *
 * event.target 就是按下时命中的那块——触摸捕获保证不会跨块。
 */
private onPieceTouchMove(event: EventTouch): void {
    const node = event.target as Node;
    const delta = event.getUIDelta();
    const pos = node.position;
    node.setPosition(pos.x + delta.x, pos.y + delta.y, pos.z);
}

/**
 * 松手：弹回原槽。
 *
 * 弹回目标完全从 slots 现算——零状态拖拽。
 * 04 节起这里要先判"是不是命中了某个槽"，再决定弹回 / swap。
 */
private onPieceTouchEnd(event: EventTouch): void {
    const node = event.target as Node;
    const piece = node.getComponent(PuzzlePiece)!;
    const slotIdx = this.slots.indexOf(piece.pieceId);
    const { x, y } = this.slotToPosition(slotIdx);
    node.setPosition(x, y, 0);

    console.log(`[PuzzleBoard] piece=${piece.pieceId} 松手弹回到 slot=${slotIdx}`);
}
```

## 5. 跑起来

▶ 预览。

**测试动作**：

1. 鼠标点住任意一块、拖动 → 块跟着鼠标走。
2. 松开鼠标 → 块**瞬间弹回**到原槽（这节没缓动，下节起加）。
3. 把块拖到屏幕外松手 → 仍弹回原位（TouchCancel 触发）。
4. 在网格内乱拖（比如 0 号块拖到 8 号槽位置上松手）→ **依然弹回 0 号块原槽**。本节不做命中判定。

**控制台预期**：

```
[PuzzleBoard] piece=0 松手弹回到 slot=3
[PuzzleBoard] piece=7 松手弹回到 slot=1
```

`slot=3` 表示这块当前的"正确槽位号"是 3（洗牌后的位置）。

**不正常的信号**：

- **点不动 / 拖不响应**：节点没注册触摸事件。检查 `node.on(Node.EventType.TOUCH_START, ...)` 那几行是否在 createPieces 里执行。
- **拖动方向反了（手指右移块左移）**：用了 `event.getDelta()` 而不是 `event.getUIDelta()`。换 UIDelta。
- **块跟着手但有"飘移"（不贴手指）**：getUIDelta 是"本帧手指移动量"，多帧累加期间如果有别的逻辑也在改 position（比如跑了一个 tween），就会偏。本节没别的逻辑改 position，应该不会有这问题。
- **拖到一半松手块卡在半路**：onPieceTouchEnd 里没调用 setPosition 弹回，或 `slots.indexOf(piece.pieceId)` 返回 -1（pieceId 不在 slots 里 = 数据 bug）。
- **松手弹回到屏幕中央**：`slotToPosition(-1)` 返回的位置（slot=-1 时 sr=floor(-1/3)=-1 之类怪值）。证明 `slots.indexOf(piece.pieceId)` 返回 -1——即 slots 里**没有** pid 这个值。检查 02 节 shuffle 是否破坏了 slots 的内容（比如把 number 变成了 string）。
- **TOUCH_MOVE 不触发**：① TouchStart 没绑住或返回 false 阻断；② 节点的父节点（PieceLayer 或 PuzzleRoot）拦截了事件——把它们的 UITransform 检查一下、确保没装了 BlockInputEvents 之类的拦截组件。

## 6. 复盘：这节教了什么

**1. 触摸捕获 = 天然的"当前操作目标"**

Cocos 触摸捕获机制保证：**touchstart 命中谁，整次拖动 move/end 都给谁**。

收益：

- 不需要全局变量记"当前在拖哪块"。
- 不需要在 move 回调里再判"是不是当前块"。
- 多指触摸时，每个手指独立捕获各自的节点——天然支持多指拖动多块（虽然本游戏不需要）。

很多框架做拖拽都自己实现"全局拖拽 manager"，记 activeNode、activePoint 等等——**Cocos / Unity / 等绝大多数游戏引擎都内建捕获**，直接用就行。**框架已经替你想过了**。

**2. 零状态拖拽 = 少 2 个字段、少 1 个 bug**

`onPieceTouchEnd` 里：

```typescript
const slotIdx = this.slots.indexOf(piece.pieceId);
const { x, y } = this.slotToPosition(slotIdx);
node.setPosition(x, y, 0);
```

**这 3 行就是完整的弹回**。不需要 dragStartX、dragStartY 字段。

对比"维护按下时位置" 的写法：

| 维度 | 维护快照 | 零状态 |
|------|----------|--------|
| 字段数 | 2（dragStartX/Y） | 0 |
| 概念数 | 2（"按下位置"是个新概念） | 0 |
| 真 bug | 按下瞬间视觉异常导致弹回错位 | 不存在 |

少 2 个字段、少 2 个概念、少 1 个 bug 路径。这就是"消除特殊情况"。

**3. Y 轴方向是初学者的天坑**

屏幕坐标 Y 向下、UI 坐标 Y 向上——**90% 的"拖动反了"** 都是这里。

记住：**用 `event.getUIDelta()` 不用 `event.getDelta()`**——前者已经替你转好坐标系。如果你看老教程用 `getDelta` 然后手动 `dy = -dy`，那是 Cocos 3.0 之前的写法。

**4. console.log 是教学版的最佳调试器**

每个关键事件打一行 log 是教学版的好习惯：

- 拖动响应正常 → 看 log 数量验证。
- 拖错块 → 看 log 里 piece= 是不是预期值。
- slot 算错 → 看 log 里 slot= 跟手算 `slots.indexOf(pid)` 对不对得上。

**生产版**会去掉这些 log（性能 + 减噪）。教学版**留着**。

## 7. 常见坑

- **节点没 UITransform 时 TOUCH_START 永远不触发**：触摸命中检测靠 UITransform 的尺寸。01 节我们给每块 addComponent(UITransform) + setContentSize 了，本节继承不变。如果你后来手贱删了，触摸响应会失效。
- **`event.target` 类型问题**：`event.target` TS 类型是 `BaseNode | null`，需要 `as Node` 强转。或者用 `event.currentTarget`（注册时绑定的那个 Node）—— 单层注册时两者一致。
- **重复注册触发多次**：如果你在 `onLoad` 里注册一次、又在某个生命周期里再注册一次，就会触发 2 次。Cocos 的 `node.on` 不去重——加事件**只调一次**。如果非要确保不重复，先 `node.off(...)` 一遍再 `on`。
- **拖动期间帧率掉下来**：每帧 setPosition 触发 dirty + 重渲染。9 块拖 1 块**完全无感**，6×6=36 块拖 1 块也无感，更大规模可能要做"批量 commit" 优化——本教程规模不需要管。
- **TOUCH_END 漏处理 TOUCH_CANCEL**：手指滑出节点范围松手 → 触发 TOUCH_CANCEL 而不是 TOUCH_END。如果你只注册了 END 没注册 CANCEL，这种情况块**永远不弹回**——卡在屏幕边缘。**两个都注册同一个回调**最稳。
- **`PuzzlePiece` import 漏了**：用 `node.getComponent(PuzzlePiece)` 时如果忘了 `import { PuzzlePiece } from './PuzzlePiece'`，TS 会编译失败。
- **`slots.indexOf(piece.pieceId)` 返回 -1**：理论 slots 永远是 [0..pieceCount-1] 的全排列，indexOf 总能找到。如果返回 -1，证明 02 节 shuffle 破坏了数据——典型 bug 是 shuffle 时往里塞了字符串、null、重复 ID。**slots 长度恒定 = pieceCount，内容恒定 = [0..pieceCount-1]**，是不变量，违反了就是 bug。

## 8. 下一节预告

能拖能弹——但是松手永远弹回原位，**根本没法拼图**。

**04 节**：松手时判"落点是哪个槽" → 把被拖块和那个槽的"原住民" **swap**。

- **指针 → 槽位**：屏幕坐标反算成 0..8 的槽下标。
- **swap 写 slots**：交换两个槽位的值。
- **layoutAllPieces 重摆**：slots 变了 → 调一次 layoutAllPieces 把两块都摆到位。
- **越界判断**：拖出 3×3 范围 → 弹回原位（不 swap）。

完成后你能"拖一块到另一块的位置 → 它们交换"——拼图玩法**雏形成型**。
