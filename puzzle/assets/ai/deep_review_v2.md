# 仔细检查（Deep Review v2 - 通用工业级）

> **触发词**：当用户说「仔细检查」时，必须执行本流程  
> **适用范围**：代码 / 配置 / 架构设计 / 数据流程 / 系统模块

---

## 🎯 核心目标

> **不是“看起来没问题”，而是“证明没有问题”**

- 逻辑零缺陷
- 数据流闭环
- 可维护性达标（≥ A，目标 S+）
- 无隐性破坏

---

## ⚠️ 强制前置步骤（必须执行）

### 阶段 0：锁定审查范围（禁止跳过）

```
1. 明确审查目标：
   - 用户已指定 → 精确到文件 / 函数 / 模块
   - 未指定 → 必须先询问，不允许猜

2. 构建上下文：
   - 读取完整实现（禁止片段脑补）
   - 读取依赖（函数 / 类型 / 常量 / 配置）
   - 识别调用链（谁调用它，它调用谁）

3. 建立假设边界：
   - 输入来源（外部 / 内部 / 用户 / 网络）
   - 执行环境（前端 / 后端 / 引擎 / Node / 浏览器）
```

---

## 🔍 审查流程（四阶段）

---

### 第一遍：逻辑正确性（最高优先级）

> **逐函数 / 逐路径验证（必须覆盖所有分支）**

```
对每个函数 / 模块：

1. 前置条件：
   - 参数所有可能取值（包括非法输入）

2. 后置条件：
   - 所有 return 路径是否符合预期？

3. 控制流：
   - if / switch 是否覆盖所有情况？
   - 是否存在遗漏分支？

4. 边界条件：
   - null / undefined / NaN
   - 空数组 / 空对象
   - 0 / 负数 / 极值 / 溢出

5. 异步与并发：
   - await 是否遗漏？
   - 是否存在竞态条件？
   - Promise 错误是否被吞？

6. 状态与副作用：
   - 是否修改外部状态？
   - 修改是否可预期？
   - 是否存在隐式共享状态？

7. 终止性：
   - 循环 / 递归是否保证终止？
```

#### 输出格式（强制）：

```
【逻辑审计】

✅ functionName
   - 正确原因：[基于代码的证明]

❌ functionName
   - 问题：[精确描述]
   - 触发条件：[输入或路径]
   - 位置：[行号或代码片段]
   - 修复方案：[具体改法]

⚠️ functionName
   - 风险点：[不确定问题]
   - 建议验证：[如何复现或测试]
```

---

### 第二遍：数据流完整性

```
1. 数据来源：
   - 是否可信？
   - 类型是否严格？

2. 数据变换：
   - 是否丢失信息？
   - 是否重复转换？

3. 数据去向：
   - 消费方是否正确使用？
   - 是否存在未使用数据？

4. 异常数据：
   - 非法输入是否被拦截？
   - 防御是否充分？
```

输出：

```
【数据流】

✅ 正常闭环 / ⚠️ 风险点

- [风险描述]
- [影响范围]
- [建议修复]
```

---

### 第三遍：代码质量（Linus 标准）

```
1. 特殊情况：
   - 是否可以通过数据结构消除分支？

2. 复杂度：
   - 是否存在 >3 层嵌套？

3. 函数职责：
   - 是否单一职责？

4. 命名：
   - 是否表达真实意图？

5. 重复代码：
   - 是否可抽象？

6. 死代码：
   - 是否存在不可达路径？

7. 可读性：
   - 是否需要额外认知成本？
```

---

### 第四遍：兼容性与破坏性

```
1. 接口变化：
   - 参数 / 返回值是否改变？

2. 行为变化：
   - 默认值是否改变？

3. 影响范围：
   - 是否影响已有调用方？

4. 隐式破坏：
   - 是否改变执行顺序 / 状态时机？
```

## 🔥 铁律（升级版）

```
1. 不允许跳步骤（必须四阶段）
2. 不允许“感觉没问题”
3. 每个问题必须：
   - 有触发条件
   - 有代码位置
   - 有修复方案
4. 不确定必须标 ⚠️（禁止忽略）
5. 优先找“会出 bug 的点”，不是风格问题
```

---

## 🚀 大型项目增强建议

```
如果代码量较大：
→ 先输出【审查计划】
→ 分批审查（模块级 / 文件级）
→ 不允许一次性草率输出
```
